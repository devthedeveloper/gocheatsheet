Watch It Breathe

you can't fix what you can't see at 3 a.m. — give the system a pulse

In this chapter you'll learn ✍️

  • a metrics endpoint, dependency-free, in one middleware
  • the four golden signals every service should expose
  • what to actually alert on (hint: not CPU)
  • where Prometheus & Grafana fit

The pager goes off at 3 a.m.

The site is slow. Is it the database? A thundering herd? One wedged instance? A login flood? Without observability, you're debugging blind — restarting things and praying. With it, you glance at a dashboard and see which signal spiked. Everything we tuned in this book, we tuned by measuring; production is no different, except the measuring has to run all the time, automatically.

Metrics in one middleware

You don't need a big dependency to start. Go's standard library has expvar, and a single middleware can count and time every request. Here's gopherit's:

cmd/api/metrics.go 
package main

import (
	"expvar"
	"net/http"
	"strconv"
	"time"
)

// Observability, chapter 10. We keep it dependency-free with the standard
// library's expvar, which publishes variables as JSON at /debug/vars. In
// production you'd usually export Prometheus metrics instead (the chapter shows
// how), but the shape is the same: count requests, time them, expose them.
var (
	metricRequests    = expvar.NewInt("requests_total")
	metricErrors      = expvar.NewInt("responses_5xx_total")
	metricInFlight    = expvar.NewInt("requests_in_flight")
	metricTotalMicros = expvar.NewInt("request_duration_micros_total")
)

// statusRecorder remembers the status code so the metrics layer can see it.
type statusRecorder struct {
	http.ResponseWriter
	status int
}

func (r *statusRecorder) WriteHeader(code int) {
	r.status = code
	r.ResponseWriter.WriteHeader(code)
}

// metrics records one measurement per request: count, in-flight gauge,
// duration, and 5xx errors.
func (app *application) metrics(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		metricRequests.Add(1)
		metricInFlight.Add(1)
		defer metricInFlight.Add(-1)

		rec := &statusRecorder{ResponseWriter: w, status: http.StatusOK}
		next.ServeHTTP(rec, r)

		metricTotalMicros.Add(time.Since(start).Microseconds())
		if rec.status >= 500 {
			metricErrors.Add(1)
		}
	})
}

// metricsHandler serves a compact plaintext summary at /metrics — the golden
// signals (traffic, errors, latency, saturation) at a glance.
func (app *application) metricsHandler(w http.ResponseWriter, r *http.Request) {
	total := metricRequests.Value()
	avg := int64(0)
	if total > 0 {
		avg = metricTotalMicros.Value() / total
	}
	w.Header().Set("Content-Type", "text/plain")
	w.Write([]byte(
		"requests_total " + strconv.FormatInt(total, 10) + "\n" +
			"responses_5xx_total " + strconv.FormatInt(metricErrors.Value(), 10) + "\n" +
			"requests_in_flight " + strconv.FormatInt(metricInFlight.Value(), 10) + "\n" +
			"request_duration_micros_avg " + strconv.FormatInt(avg, 10) + "\n"))
}

It's one more layer on the middleware onion — the same pattern that's paid off all book. Now every request is counted and timed, and a plain-text /metrics endpoint reports the totals:

terminal
$ curl localhost:4000/metrics
requests_total 20531
responses_5xx_total 0
requests_in_flight 3
request_duration_micros_avg 1683

The four golden signals

Google's SRE book distills what to watch into four signals. Learn these and you can reason about almost any service:

SignalQuestion it answersgopherit metric
Traffichow much demand?requests_total (rate)
Errorshow much is failing?responses_5xx_total
Latencyhow slow, at the tail?request duration (p50/p95/p99)
Saturationhow full are the resources?requests_in_flight, pool usage
Alert on symptoms, not causes

The rookie mistake is alerting on CPU (“page me at 80%”). But high CPU isn't a problem if latency is fine — and low CPU during an outage tells you nothing. Alert on what your users feel: rising p99 latency and rising error rate. Those are symptoms of every problem, whatever the cause. CPU, memory, and pool metrics are for diagnosing once you're already paged — not for waking you up.

Where the big tools fit

Our expvar endpoint is real observability, but production usually standardises on Prometheus: your app exposes /metrics in its text format (the prometheus/client_golang library gives you histograms — real p99, not just an average like our toy), Prometheus scrapes it every few seconds, and Grafana draws the dashboards and fires the alerts. Same shape as what we built — count, time, expose — just with proper histograms and a scraper doing the remembering.

Logs, metrics, traces

Metrics tell you something is wrong (p99 spiked). Logs — gopherit's structured slog lines from the Reddit book — tell you what (which endpoint, which error). Traces (OpenTelemetry) tell you where in a request's journey the time went. Start with metrics + logs; add tracing when a request crosses enough services that “where” stops being obvious.

Pin to your brain 📌

  • Observability = measuring in production, always-on. A middleware + /metrics is enough to start.
  • Watch the four golden signals: traffic, errors, latency (p99!), saturation.
  • Alert on symptoms users feel (latency, errors), diagnose with resource metrics.
  • Graduate to Prometheus + Grafana for real histograms and dashboards; same shape as your toy.
