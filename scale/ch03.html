chapter3

Find the Bottleneck

profile before you fix — the answer is almost never where you'd guess

In this chapter you'll learn ✍️

  • wiring up pprof safely (and why “safely” matters)
  • reading EXPLAIN QUERY PLAN — the database showing its work
  • spotting the two words that mean “this query doesn't scale”
  • the actual culprit behind our 3 req/s (it's one line of SQL)

Resist the urge to guess

3 req/s. Every instinct screams a fix: “it's the database connection pool!”, “add caching!”, “more CPU!”. Every one of those instincts might be wrong, and acting on a wrong guess costs days. The rule: profile first. Let the machine tell you where the time actually goes.

pprof: Go's built-in profiler

Go ships a world-class profiler in the standard library. You expose it on a separate, localhost-only port — never the public one, because those endpoints leak internals and can be abused:

cmd/api/main.go — pprof on a private port 
	// Profiling endpoints (chapter 3), on a SEPARATE localhost-only port so
	// they're never exposed to the public internet. Visit
	// http://localhost:6060/debug/pprof/ while the server is under load.
	if cfg.env != "production" {
		go func() {
			mux := http.NewServeMux()
			mux.HandleFunc("/debug/pprof/", pprof.Index)
			mux.HandleFunc("/debug/pprof/profile", pprof.Profile)
			mux.HandleFunc("/debug/pprof/heap", pprof.Handler("heap").ServeHTTP)
			logger.Info("pprof on http://localhost:6060/debug/pprof/")
			http.ListenAndServe("localhost:6060", mux)
		}()
	}

Now, with hammer pounding the server, grab a 15-second CPU profile and open it:

terminal
# while the load test runs:
$ go tool pprof http://localhost:6060/debug/pprof/profile?seconds=15
(pprof) top
# ...time dominated by SQLite query execution — the feed query —
# not JSON encoding, not the HTTP layer, not GC.
(pprof) web   # opens a flame graph in your browser

The profile points squarely at the database: nearly all the time is spent inside one query, the feed. The HTTP layer, JSON encoding, the Go runtime — all noise. So we stop looking at Go and ask the database to explain itself.

Also run it under -race

Before you optimise anything concurrent, run your load test against a go build -race binary once. A data race that's invisible at low traffic will corrupt data under load. Correctness first, then speed — a fast wrong answer is still wrong.

EXPLAIN QUERY PLAN — the database shows its work

Every SQL database can tell you how it plans to run a query. In SQLite you prefix it with EXPLAIN QUERY PLAN. Here's the feed query as gopherit shipped it — the “hot” sort with its COUNT(*) OVER() window function for pagination totals:

sqlite — the feed query as shipped
EXPLAIN QUERY PLAN
  SELECT ..., COUNT(*) OVER() AS total
  FROM posts p JOIN users u ... JOIN subreddits sr ...
  ORDER BY p.hotness DESC LIMIT 20 OFFSET 0;

CO-ROUTINE (subquery-2)
SCAN p                          <-- reads ALL 50,000 posts
SEARCH u USING INTEGER PRIMARY KEY (rowid=?)
SEARCH sr USING INTEGER PRIMARY KEY (rowid=?)
SCAN (subquery-2)
USE TEMP B-TREE FOR ORDER BY    <-- sorts ALL 50,000 in a scratch tree

There they are — the two phrases that mean “this does not scale”:

  • SCAN p — a full table scan. Every request reads all 50,000 posts, even though we only return 20.
  • USE TEMP B-TREE FOR ORDER BY — the database builds a throwaway sorted tree of all 50,000 rows, every single request.

Why? The innocent-looking COUNT(*) OVER() — a window function we used to get the total row count for pagination in the same query. To count all matching rows, SQLite has to visit all of them. That one convenience defeats the LIMIT entirely: there's no point stopping at 20 rows if you must scan 50,000 to get the total.

Now watch what the same query looks like without the window function:

sqlite — page query, no window function
EXPLAIN QUERY PLAN
  SELECT ... FROM posts p JOIN users u ... JOIN subreddits sr ...
  ORDER BY p.hotness DESC LIMIT 20 OFFSET 0;

SCAN p USING INDEX idx_posts_hotness   <-- walks the index, stops at 20
SEARCH u USING INTEGER PRIMARY KEY (rowid=?)
SEARCH sr USING INTEGER PRIMARY KEY (rowid=?)

No full scan. No temp b-tree. With an index on the sort column, SQLite walks it in order and stops after 20 rows — touching 20 posts instead of 50,000. Same data, same result, a different plan — and that difference is the whole ballgame.

Why the profiler and EXPLAIN, together

pprof told us which code was slow (the feed query). EXPLAIN told us why (full scan + temp sort). You need both: the profiler narrows a whole program to one hot spot; EXPLAIN narrows one query to one bad line. Neither alone would have pointed at a window function — which is exactly why we didn't just start adding connection-pool size and hoping.

Pin to your brain 📌

  • Expose pprof on a private port; grab a CPU profile under load.
  • EXPLAIN QUERY PLAN shows how the DB runs a query — read it before optimising it.
  • SCAN (full table read) and USE TEMP B-TREE (throwaway sort) = doesn't scale.
  • Our villain: COUNT(*) OVER() forced a scan+sort of all 50k rows every request.
