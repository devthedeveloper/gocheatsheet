chapter9

Clones

run several copies behind a proxy — and pay the price of no longer being alone

In this chapter you'll learn ✍️

  • vertical vs horizontal scaling — bigger box vs more boxes
  • why gopherit is ready to clone (it's stateless — by design)
  • a load balancer in five lines
  • what breaks when you clone: per-instance state

Two ways to get bigger

When one machine isn't enough, you have two moves. Vertical: a bigger machine — simplest, but there's a ceiling and the price climbs fast. Horizontal: more machines running the same code, sharing the load — nearly unlimited, and what every large system eventually does. The catch is that horizontal scaling only works if your app is stateless: any instance can serve any request, because no instance holds anything private about a user's session.

gopherit was born ready

Remember the Reddit book choosing bearer tokens over server-side sessions? That decision pays off right now. gopherit keeps nothing in memory about who's logged in — the token in the request header is the whole story, and the database is the shared source of truth. So we can run ten copies and any of them can serve any request. Statelessness isn't a scaling feature you bolt on; it's a design choice you make early and cash in later.

Three gophers behind one door

Run three instances on different ports, and put a load balancer in front to spread requests across them. Any reverse proxy does this; here's Caddy in five lines:

Caddyfile — round-robin across three instances 
:8080 {
	reverse_proxy localhost:4001 localhost:4002 localhost:4003 {
		lb_policy round_robin
	}
}
terminal
$ ./gopherit -port 4001 -dsn "postgres://..." &
$ ./gopherit -port 4002 -dsn "postgres://..." &
$ ./gopherit -port 4003 -dsn "postgres://..." &
$ caddy run   # now http://localhost:8080 fans out across all three

Traffic now spreads three ways. (They must share one database — Postgres from the last chapter, not three separate SQLite files.) Add a fourth instance, a fifth; scale until the database becomes the bottleneck, which is the natural next wall.

What clones break: private state

Cloning exposes a subtle bug in two things we built. Both the feed cache (chapter 5) and the rate limiter (chapter 7) live inside one process. With three instances, you now have three separate caches and three separate limiters:

cache ×33 copies, 3× the DB misses,
up to 3× the staleness window
·
limiter ×3an IP can do 3× the requests —
one per instance

The in-process versions still work — they're just less effective when split three ways. The fix is to move that shared state out of the processes into a store they all share: Redis. One shared cache, one shared limiter, all instances agreeing:

the shared-cache idea (Redis) — ~20 lines 
// Instead of an in-process map, GET/SET the cached feed bytes in Redis,
// which every instance reads from and writes to. The handler barely changes —
// swap app.feed.fetch(...) for a Redis GET, rebuild-and-SET on a miss.
val, err := rdb.Get(ctx, "feed:"+key).Bytes()
if err == redis.Nil {          // miss: one instance rebuilds and shares it
	val = buildFeed()
	rdb.Set(ctx, "feed:"+key, val, time.Second)
}
// Redis also has atomic INCR + EXPIRE — a shared rate limiter in a few lines.
Don't reach for Redis too early

On a single instance, the in-process cache and limiter are better than Redis — no network hop, less to operate. You only need shared state once you're running multiple instances and the per-instance drift actually hurts. This is the whole book's discipline again: add the shared cache when the metrics say the split is costing you, not because “real apps use Redis.”

The session-affinity myth

People reach for “sticky sessions” (pin each user to one instance) to make in-memory session state work across clones. gopherit never needs it — there's no in-memory session to pin, because auth is a stateless token. Sticky sessions are a workaround for a problem you avoided by design. Prefer stateless-plus-shared-store over sticky-plus-local-state.

Pin to your brain 📌

  • Vertical = bigger box (ceiling); horizontal = more boxes (needs statelessness).
  • Stateless auth (bearer tokens) is what makes gopherit safe to clone.
  • A load balancer round-robins requests across instances in a few lines.
  • In-process cache/limiter become per-instance when cloned; move them to Redis when it hurts.
