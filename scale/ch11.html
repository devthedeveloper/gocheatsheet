The rematch

Chapter 2 pointed hammer at the feed and gopherit fell over at 3 req/s. Same command, same hardware, same seeded database — now against the finished system:

$ ./hammer -url http://localhost:4000 -scenario feed -workers 50 -duration 15s

3req/s (ch 2) ⟶ 21,506req/s (now) · 7,000×throughput

And latency: p50 from 8.3 seconds to 2 milliseconds, with zero errors. The hug of death became a shrug — on the same laptop, no new machines.

The whole journey, one table

Look at the shape of that table and rank what mattered:

  • Caching gave the single biggest jump (614 → 21,506).
  • The one deleted line of SQL was the unlock (6 → 614) — nothing else worked until it did.
  • The connection pool — the fix everyone reaches for first — was worth 3 → 6, and nothing more.

If you'd started by adding connections and buying a bigger box, you'd have spent money and a week to get from 3 req/s to maybe 12. The profiler pointed at a window function instead. That's the entire thesis of this book.

The map of what we didn't need

We hit tens of thousands of requests per second with an in-process cache, some indexes, a rate limiter, and (when writes demand it) Postgres. We never touched the buzzwords. Here's when you would — each with the trigger metric that tells you it's time:

Thing we skippedReach for it when…
CDNyou serve images/static assets, or users are geographically far — offload bytes to the edge
Read replicasread load outgrows one database even after caching (rare — caching usually wins first)
Shardingyour write volume outgrows the biggest single database you can buy
Message queuework can be done later (emails, thumbnails) — get it off the request path
Microservicesyour team is too big to coordinate on one codebase — it's an org solution, not a perf one
Kubernetesyou're operating enough services that manual deploys hurt more than k8s complexity does
Multi-regionyou need to survive a whole datacenter failing, or serve users on other continents with low latency
The trap of premature scale

Every item above adds real, permanent operational cost. Adopting them before the trigger metric fires doesn't make you web-scale — it makes you slow to ship and expensive to run, solving problems you don't have. The most senior move in this whole book is the restraint: measure first, and add the smallest thing that fixes the number in front of you.

The trilogy, closed

cheatsheet → linkboard → gopherit → gopherit at scale 🚀

You started not knowing how to write an API. You built one, then made it real, and now you can take it from falling over at three requests a second to serving tens of thousands — and, just as important, you know why each fix worked, and which ones to ignore. That last part — measuring instead of guessing, and adding the smallest thing that works — is the difference between someone who's read about scaling and someone who can actually do it.

Now go survive your own front page. 🐹🚀

The whole book on one sticky note 📌

  • Scaling is measurement: load → break → profile → fix one thing → measure again.
  • Profile before you touch anything; the bottleneck is never where you'd guess.
  • Biggest read-path wins: fix the query (indexes, no window functions), then cache it.
  • Protect expensive endpoints (rate limits), bound every request (timeouts), watch p99.
  • Add Postgres / clones / Redis / the rest only when a metric proves you need them.
