The Login Problem

bcrypt is slow on purpose — which makes login a denial-of-service button

In this chapter you'll learn ✍️

  • why your most secure endpoint is your most fragile one
  • measuring the bcrypt cliff: 50 logins pin every core
  • a per-IP token-bucket rate limiter (with a janitor goroutine)
  • turning a 3.4-second DoS into a 3-millisecond rejection

Slow on purpose is a double edge

The Reddit book hashed passwords with bcrypt at cost 12 — deliberately slow (~250ms) so attackers can't brute-force stolen hashes. Good security. But “deliberately expensive” plus “anyone can call it” equals a denial-of-service button. Every login burns a quarter-second of CPU before you even know the password is wrong. Point a crowd at /tokens and watch what happens. Let's measure it:

Fourteen requests a second saturates the entire machine. And here's the real damage: while bcrypt is chewing every core, the 21,000-req/s feed we built is starved too. One attacker with a laptop and a password list takes down the whole site — not by breaking in, just by knocking.

The fix: reject the flood before bcrypt

The insight: we can't make bcrypt fast (that would defeat its purpose), but we can refuse to run it for an abusive client. A rate limiter caps how many requests any one IP may make per second, and rejects the excess with a cheap 429 Too Many Requests — before the request ever reaches the expensive handler.

We use a token bucket (from golang.org/x/time/rate): each IP gets a bucket that refills at N tokens/second; each request spends one; an empty bucket means rejection. A per-IP map holds them, and a janitor goroutine evicts idle IPs so the map can't grow forever:

cmd/api/ratelimit.go 
package main

import (
	"net"
	"net/http"
	"sync"
	"time"

	"golang.org/x/time/rate"
)

// rateLimit is a per-IP token-bucket limiter. bcrypt makes POST /tokens
// expensive on purpose, which turns it into a denial-of-service target: a few
// dozen concurrent logins pin every CPU. This middleware caps how fast any one
// IP can hit us, rejecting the flood with a cheap 429 *before* it reaches
// bcrypt — so real traffic keeps flowing.
type ipLimiter struct {
	mu      sync.Mutex
	clients map[string]*client
	rps     rate.Limit
	burst   int
}

type client struct {
	limiter  *rate.Limiter
	lastSeen time.Time
}

func newIPLimiter(rps float64, burst int) *ipLimiter {
	l := &ipLimiter{clients: make(map[string]*client), rps: rate.Limit(rps), burst: burst}
	// A janitor goroutine evicts IPs we haven't seen in a while, so the map
	// can't grow without bound.
	go func() {
		for {
			time.Sleep(time.Minute)
			l.mu.Lock()
			for ip, c := range l.clients {
				if time.Since(c.lastSeen) > 3*time.Minute {
					delete(l.clients, ip)
				}
			}
			l.mu.Unlock()
		}
	}()
	return l
}

func (l *ipLimiter) allow(ip string) bool {
	l.mu.Lock()
	defer l.mu.Unlock()
	c, ok := l.clients[ip]
	if !ok {
		c = &client{limiter: rate.NewLimiter(l.rps, l.burst)}
		l.clients[ip] = c
	}
	c.lastSeen = time.Now()
	return c.limiter.Allow()
}

// rateLimit wraps a handler, rejecting over-limit IPs with 429. If no limiter
// is configured (the -rate flag is 0) it does nothing.
func (app *application) rateLimit(next http.Handler) http.Handler {
	if app.limiter == nil {
		return next
	}
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		ip, _, err := net.SplitHostPort(r.RemoteAddr)
		if err != nil {
			ip = r.RemoteAddr
		}
		if !app.limiter.allow(ip) {
			w.Header().Set("Retry-After", "1")
			app.errorResponse(w, http.StatusTooManyRequests, "rate limit exceeded — slow down")
			return
		}
		next.ServeHTTP(w, r)
	})
}

It slots into the middleware onion from the Reddit book — early, so rejections are cheap:

cmd/api/routes.go — one more onion layer 
// routes builds the router. Go 1.22+ ServeMux understands
// "METHOD /path/{param}" patterns, so we need zero external packages.
func (app *application) routes() http.Handler {
	mux := http.NewServeMux()

	// Health & metrics
	mux.HandleFunc("GET /api/v1/healthz", app.healthzHandler)
	mux.HandleFunc("GET /metrics", app.metricsHandler)

	// Users & auth
	mux.HandleFunc("POST /api/v1/users", app.registerUserHandler)
	mux.HandleFunc("POST /api/v1/tokens", app.loginHandler)

	// Subreddits
	mux.HandleFunc("GET /api/v1/subreddits", app.listSubredditsHandler)
	mux.HandleFunc("POST /api/v1/subreddits", app.requireAuth(app.createSubredditHandler))
	mux.HandleFunc("GET /api/v1/subreddits/{name}", app.getSubredditHandler)
	mux.HandleFunc("GET /api/v1/subreddits/{name}/posts", app.listSubredditPostsHandler)

	// Posts
	mux.HandleFunc("GET /api/v1/posts", app.listPostsHandler)
	mux.HandleFunc("POST /api/v1/posts", app.requireAuth(app.createPostHandler))
	mux.HandleFunc("GET /api/v1/posts/{id}", app.getPostHandler)

	// Comments & votes
	mux.HandleFunc("POST /api/v1/posts/{id}/comments", app.requireAuth(app.createCommentHandler))
	mux.HandleFunc("POST /api/v1/posts/{id}/vote", app.requireAuth(app.votePostHandler))
	mux.HandleFunc("POST /api/v1/comments/{id}/vote", app.requireAuth(app.voteCommentHandler))

	// The middleware onion, outermost first: recover from panics, count metrics,
	// log every request, rate-limit abusive IPs (cheap 429 before any real
	// work), add CORS headers, then identify the caller.
	return app.recoverPanic(app.metrics(app.logRequests(app.rateLimit(app.enableCORS(app.authenticate(mux))))))
}

Measure the flood, deflected

Same login flood, now with the limiter set to a strict few requests per second per IP:

login flood · 50 workersthroughputp50what happens
no rate limit14 req/s3.4severy request runs bcrypt; CPU pinned
+ per-IP rate limit11,554 req/s2–3msexcess rejected with cheap 429; CPU free

The abusive IP now gets its handful of real login attempts per second (those still run bcrypt), and everything beyond that is bounced in a couple of milliseconds without touching bcrypt at all. The CPU stays free, and legitimate users — logging in, reading the feed — sail through. A 3.4-second denial-of-service became a 3-millisecond shrug.

Reading that table honestly

The “11,554 req/s” is mostly 429 rejections — that's the point, not a throughput brag. The number that matters is what it protects: the server no longer melts, so real requests keep their millisecond latencies. When you rate-limit, watch your rejection rate and your real-traffic latency, not the raw request count.

Also cache the token lookup

Every authenticated request does a database SELECT to resolve its bearer token. Under heavy load that adds up. A tiny in-memory cache of token-hash → user (like chapter 5's, with a short TTL) skips the query for hot tokens — the same “don't recompute what hasn't changed” move, applied to auth. And behind a real proxy, read the client IP from X-Forwarded-For, not RemoteAddr (which would be the proxy).

Pin to your brain 📌

  • Deliberately-slow endpoints (bcrypt login) are DoS targets: 14 req/s pinned every core.
  • Rate-limit per IP with a token bucket; reject excess with a cheap 429 before the work.
  • Evict idle IPs with a janitor goroutine so the limiter map can't grow unbounded.
  • Judge a rate limiter by protected real-traffic latency, not raw request count. 3.4s → 3ms.
