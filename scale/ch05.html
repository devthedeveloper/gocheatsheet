Cache Rules Everything

the front page is the same for everyone — so build it once

In this chapter you'll learn ✍️

  • why the cheapest query is the one you never run
  • a ~60-line in-process TTL cache (map + RWMutex)
  • the thundering-herd problem, and singleflight
  • the biggest jump in the book: 614 → 21,506 req/s

The same bytes, over and over

614 req/s means we're running the feed query 614 times a second. But think about what it returns: the exact same front page for every visitor. Ada, Linus, and ten thousand strangers all get identical bytes. Rebuilding that from the database each time is pure waste. The fastest query is the one you never send — cache the answer.

A tiny in-process cache

No Redis yet — we don't need it. A map guarded by a read-write mutex, with a time-to-live on each entry, lives right inside the server process. First request builds and stores the page; everyone else within the TTL gets the stored copy in microseconds:

cmd/api/cache.go 
package main

import (
	"sync"
	"time"

	"golang.org/x/sync/singleflight"
)

// feedCache is a tiny in-process cache for the front page. The feed is the
// same bytes for everyone, so there is no reason to rebuild it from the
// database on every request — we build it once and hand the same bytes to
// everyone who asks within the TTL.
//
// singleflight solves the "thundering herd": when a cached entry expires under
// heavy traffic, only ONE goroutine rebuilds it; the rest wait for that single
// result instead of all hammering the database at once.
type feedCache struct {
	ttl   time.Duration
	group singleflight.Group

	mu      sync.RWMutex
	entries map[string]cacheEntry
}

type cacheEntry struct {
	data    []byte
	expires time.Time
}

func newFeedCache(ttl time.Duration) *feedCache {
	return &feedCache{ttl: ttl, entries: make(map[string]cacheEntry)}
}

// get returns cached bytes for key if they are still fresh.
func (c *feedCache) get(key string) ([]byte, bool) {
	c.mu.RLock()
	e, ok := c.entries[key]
	c.mu.RUnlock()
	if !ok || time.Now().After(e.expires) {
		return nil, false
	}
	return e.data, true
}

// fetch returns cached bytes, or builds them via build() if missing/stale.
// Concurrent callers for the same key share a single build.
func (c *feedCache) fetch(key string, build func() ([]byte, error)) ([]byte, error) {
	if data, ok := c.get(key); ok {
		return data, nil
	}
	data, err, _ := c.group.Do(key, func() (any, error) {
		// Re-check: another goroutine may have filled it while we queued.
		if data, ok := c.get(key); ok {
			return data, nil
		}
		fresh, err := build()
		if err != nil {
			return nil, err
		}
		c.mu.Lock()
		c.entries[key] = cacheEntry{data: fresh, expires: time.Now().Add(c.ttl)}
		c.mu.Unlock()
		return fresh, nil
	})
	if err != nil {
		return nil, err
	}
	return data.([]byte), nil
}

Two things make this more than a naive map:

  • An RWMutex — many readers can check the cache at once; only a writer refreshing an entry needs exclusive access. Reads are the common case, so this matters.
  • singleflight — the star of the show, explained next.

The thundering herd

Here's the trap a naive cache falls into. An entry expires. At that instant, 500 requests arrive, all see an empty cache, and all 500 stampede the database to rebuild the same page at once — a thundering herd. The cache made things worse: you get periodic 500-query spikes that can knock the database over.

golang.org/x/sync/singleflight fixes it in one call: when many goroutines ask for the same key at once, exactly one does the work and the rest wait for and share its result. 500 misses become 1 rebuild + 499 free-riders. Look back at fetch() in cache.go — that's the whole mechanism: c.group.Do(key, ...).

Wiring it into the feed handler: cache the response bytes, keyed by the query string.

cmd/api/posts.go — the handler, now cached 
// listPostsHandler — GET /api/v1/posts?sort=hot&page=1&page_size=20
// The site-wide front page.
func (app *application) listPostsHandler(w http.ResponseWriter, r *http.Request) {
	filters, err := app.readPostFilters(r)
	if err != nil {
		app.badRequest(w, err)
		return
	}

	// Serve the front page from the cache: everyone asking for the same
	// sort/page gets the same bytes for up to the TTL, and only one goroutine
	// ever rebuilds a given page (singleflight guards the thundering herd).
	body, err := app.feed.fetch(r.URL.RawQuery, func() ([]byte, error) {
		posts, meta, err := app.store.ListPosts(filters)
		if err != nil {
			return nil, err
		}
		return json.MarshalIndent(envelope{"posts": posts, "meta": meta}, "", "\t")
	})
	if err != nil {
		app.serverError(w, r, err)
		return
	}

	w.Header().Set("Content-Type", "application/json")
	w.Write(body)
}

Twenty-one thousand requests a second, p50 of two milliseconds, on the same four-core laptop that could manage three. Remember the napkin math from chapter 1 — a million daily users was ~3,000 req/s at peak. We're now comfortably past that, on one machine, serving the front page essentially for free.

The two hard things about caching

“There are only two hard problems in computer science: cache invalidation and naming things.” Our invalidation is the simplest that works: a 1-second TTL. Good enough — a post is at most 1s late to the feed. The alternative, explicitly clearing the cache on every new post/vote, is more code and more bugs. Start with a short TTL; reach for explicit invalidation only when a second of staleness genuinely matters.

Caches all the way out

We cached inside the process. The same idea repeats at every layer: HTTP Cache-Control and ETag headers let browsers and CDNs cache the response so it never even reaches your server; a shared cache (Redis) lets many server instances share one cache (chapter 9). Same principle every time: don't recompute what hasn't changed.

Pin to your brain 📌

  • The cheapest query is the one you never run. Cache responses that are the same for everyone.
  • An in-process cache is a map + RWMutex + a TTL. No infrastructure required.
  • singleflight collapses a thundering herd of misses into a single rebuild.
  • Start with a short TTL; it's the simplest invalidation that works. 614 → 21,506 req/s.
