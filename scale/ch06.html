chapter6

Contexts & Timeouts

a request that can't be cancelled is a leak — and under load, leaks are outages

In this chapter you'll learn ✍️

  • why every slow request is a resource held hostage
  • server-level timeouts (gopherit already has them — here's why)
  • context for deadlines and cancellation, end to end
  • the IOU from the Reddit book, finally paid

The slow-motion outage

Every in-flight request holds things: a goroutine, a connection, maybe a database handle. When requests finish in milliseconds, that's fine. But let one class of request hang for 30 seconds — a slow query, a wedged upstream, a client that vanished mid-download — and under load those held resources pile up faster than they're released. Connections exhaust, goroutines balloon, and the whole server tips over. A request with no deadline is a request that can take down your server.

Server timeouts: the outer wall

gopherit already carries the first line of defense, set back in the Reddit book — timeouts on the HTTP server itself:

cmd/api/main.go — the server's own timeouts 
	srv := &http.Server{
		Addr:         fmt.Sprintf(":%d", cfg.port),
		Handler:      app.routes(),
		IdleTimeout:  time.Minute,
		ReadTimeout:  5 * time.Second,
		WriteTimeout: 10 * time.Second,
	}
  • ReadTimeout — a client dribbling its request one byte at a time can't hold a connection forever (the classic “slowloris” attack).
  • WriteTimeout — a response that takes too long to send is cut off.
  • IdleTimeout — kept-alive connections that go quiet are reclaimed.

These bound the connection. But they don't stop the work: if a handler kicks off a 30-second database query, WriteTimeout closes the socket, yet the query keeps running, still holding its connection. To cancel the work itself, we need context.

context: cancellation that flows downstream

Every *http.Request carries a context.Context — r.Context() — that Go cancels automatically when the client disconnects. If you pass that context all the way down into your database calls, a query whose caller has already hung up gets cancelled instead of grinding on for nothing. You can also add a hard deadline:

the pattern: a per-request deadline 
func (app *application) listPostsHandler(w http.ResponseWriter, r *http.Request) {
	// Give this request at most 3 seconds of database time, and cancel the
	// query automatically if the client disconnects first.
	ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
	defer cancel()

	posts, meta, err := app.store.ListPosts(ctx, filters) // ctx flows in
	// ...
}

For that to work, the store's methods must accept a context and hand it to the driver — QueryContext / ExecContext instead of Query / Exec:

internal/store/posts.go — context-aware queries 
// Before: func (s *Store) ListPosts(f PostFilters) (...)
// After:  the context threads from the request down to the driver.
func (s *Store) ListPosts(ctx context.Context, f PostFilters) ([]Post, Meta, error) {
	rows, err := s.db.QueryContext(ctx, query, args...)
	// ...when ctx is cancelled, the driver abandons the query and returns.
}
The Reddit book's IOU

Back in the Reddit book, the store methods deliberately skipped context.Context to stay readable — with a promise to add it “when scale demands.” This is that moment. It's a mechanical refactor: thread ctx from each handler through each store method into each QueryContext/ExecContext. Tedious, but every method ends up cancellable — and under load, cancellable is the difference between shedding a slow request and drowning in it.

Context carries more than deadlines

The same context.Context is also how request-scoped values ride along — remember gopherit's auth middleware stashing the current user in r.Context()? Deadlines, cancellation, and request values all travel the same pipe, from the edge of your server to the deepest query. Learn it once; it's everywhere in Go.

Pin to your brain 📌

  • Unbounded requests pile up under load until resources run out. Bound everything.
  • Set ReadTimeout/WriteTimeout/IdleTimeout on the HTTP server.
  • r.Context() cancels when the client disconnects; WithTimeout adds a deadline.
  • Thread ctx into QueryContext/ExecContext so the work itself can be cancelled.
