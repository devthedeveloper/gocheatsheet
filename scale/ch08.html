chapter8

Graduate to Postgres

when one writer really is the wall — and how the store seam makes the move painless

In this chapter you'll learn ✍️

  • how to know it's actually time to leave SQLite (measure, don't guess)
  • the store interface as a seam: swap databases, keep handlers
  • the concrete diffs: driver, placeholders, pool sizing
  • why handlers change zero lines

First: is SQLite really the wall?

SQLite allows exactly one writer at a time. For a read-heavy site that's often fine for a surprisingly long way — we're serving 21,000 reads/s. But when your writes (votes, posts, comments) genuinely can't keep up — when a write-heavy benchmark shows requests queuing on the single writer and your p99 climbs no matter what else you tune — that's the signal. Not before. Postgres is more power and more operational cost; adopt it when the data says to, not because a blog post said “SQLite doesn't scale.”

Honesty box: what's measured here

Every other chapter's numbers come from live runs. This chapter is different: it's the migration recipe, shown as real, correct code, not a benchmark. The point isn't “Postgres is X× faster” (it depends entirely on your workload) — it's that a clean store layer makes the swap a contained change. Run the numbers on your write workload before and after; that's the only comparison that means anything.

The seam pays off

Back in the Reddit book we were strict about one thing: all SQL lives in the store package; handlers never touch the database directly. That discipline felt like bookkeeping then. Here's the payoff — the entire database swap happens behind that seam. Turn the concrete *store.Store into an interface the handlers depend on:

the store as an interface (the seam) 
// Handlers depend on this interface, not on SQLite specifically.
type Store interface {
	ListPosts(ctx context.Context, f PostFilters) ([]Post, Meta, error)
	CreatePost(ctx context.Context, ...) (*Post, error)
	Vote(ctx context.Context, ...) (int64, error)
	// ...the rest of the methods
}

// Two implementations satisfy it; a flag picks one at start-up.
func openStore(driver, dsn string) (Store, error) {
	switch driver {
	case "postgres":
		return openPostgres(dsn)
	default:
		return openSQLite(dsn)
	}
}

The three concrete changes

Moving a query from SQLite to Postgres is smaller than you'd fear — three mechanical differences:

ThingSQLitePostgres
driver importmodernc.org/sqlitegithub.com/jackc/pgx/v5/stdlib
placeholders?, ?, ?$1, $2, $3
pool sizingreaders share WAL; writes serializesize SetMaxOpenConns to your core count & PG's max_connections
a query, ported 
// SQLite:
s.db.QueryContext(ctx,
	`SELECT id, title FROM posts WHERE subreddit_id = ? ORDER BY hotness DESC LIMIT ?`,
	subID, limit)

// Postgres — same query, numbered placeholders:
s.db.QueryContext(ctx,
	`SELECT id, title FROM posts WHERE subreddit_id = $1 ORDER BY hotness DESC LIMIT $2`,
	subID, limit)

Spin up Postgres locally to develop against — one command:

terminal
$ docker run -d --name pg -e POSTGRES_PASSWORD=dev -p 5432:5432 postgres:16
$ ./gopherit -db postgres -dsn "postgres://postgres:dev@localhost:5432/postgres"

And the loudest sentence in this book: the handler files change zero lines. Every users.go, posts.go, votes.go handler still calls app.store.ListPosts(...). They never knew which database answered. That is what the store layer bought you — the freedom to change your mind about the biggest infrastructure decision in your app without touching your business logic.

Pool sizing rules of thumb

SetMaxOpenConns: start near your CPU core count for a CPU-bound workload, higher for IO-bound; never exceed Postgres's max_connections (minus headroom for other clients — or put PgBouncer in front). SetConnMaxLifetime (~5 min) recycles connections so a rolling database restart doesn't strand them. Measure p99 as you raise the pool: it improves, then plateaus, then gets worse as you oversubscribe the database. Stop at the plateau.

Migrations become real now

Our SQLite migration was “CREATE TABLE IF NOT EXISTS on start-up.” Once data lives in Postgres and matters, adopt numbered, reversible migration files with a tool like goose or golang-migrate — so schema changes are replayable and a bad one can be rolled back. The training wheels come off with the training database.

Pin to your brain 📌

  • Leave SQLite when a write benchmark proves the single writer is your wall — not before.
  • Depend on a store interface; a flag picks SQLite or Postgres at start-up.
  • The port is mechanical: driver, $1 placeholders, pool sizing. Handlers don't change.
  • Real data → real migrations (goose / golang-migrate), not IF NOT EXISTS.
