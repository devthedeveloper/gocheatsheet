Basics

variables, printing, converting — the stuff that slips first

A whole Go program start here

package main

import "fmt"

func main() {
	fmt.Println("Hello, gopher!")
}

Remember: executable programs are always package main with a func main(). Run with go run .

Declaring variables — 3 ways

var name string = "Ada" // full form
age := 30               // short form ✨
const MaxNotes = 100    // never changes

var a, b = 1, 2         // two at once
x := 5
x++                     // no ++x in Go

Use := 95% of the time. It only works inside functions — at package level you need var.

Zero values — nothing is uninitialized

var i int      // 0
var f float64  // 0.0
var s string   // ""
var ok bool    // false
var p *Note    // nil
// slices, maps, channels, funcs → nil too

Remember: declaring without assigning is safe — you always get the type's zero value, never garbage.

Printing — the verbs you forget

fmt.Println("plain", 42)  // + newline
fmt.Printf("%v\n", note)  // any value
fmt.Printf("%+v\n", note) // + field names
fmt.Printf("%d %s %t\n", 7, "hi", true)
fmt.Printf("%q %T\n", "quoted", note)
msg := fmt.Sprintf("id=%d", 42)

Debugging a struct? %+v is your best friend. %T tells you a value's type.

Converting between types

f := 3.9
i := int(f) // 3 — truncates!

n, err := strconv.Atoi("42") // str→int
s := strconv.Itoa(42)        // int→str

b := []byte("hello") // string → bytes
back := string(b)    // and back

Gotcha: string(42) is NOT "42" — it's the character '*'. Numbers ↔ strings always go through strconv or fmt.

Control flow

one loop, honest ifs, and the defer trick

if / else — with the init trick

if age >= 18 {
	// no parens needed, braces required
} else if age > 12 {
	// ...
}

// declare + check in one line:
if err := save(note); err != nil {
	return err
}

Use the init form when the variable is only needed for the check — it keeps err out of the surrounding scope.

for — Go's only loop

for i := 0; i < 5; i++ { } // classic

for count < 10 { }         // "while"

for {                      // forever
	break                  // ...until break
}

for i, v := range notes { } // range ↓

Missing while? It's just for with one condition. There is no do-while.

switch — no break needed

switch status {
case 200, 201: // several at once
	fmt.Println("ok")
case 404:
	fmt.Println("missing")
default:
	fmt.Println("uh oh")
}

switch { // no value = tidy if/else chain
case age < 13:
	// kid
case age < 20:
	// teenager
}

Unlike C/JS: cases don't fall through by default. You almost never need break here.

defer — cleanup that can't be forgotten

f, err := os.Open("notes.txt")
if err != nil {
	return err
}
defer f.Close() // runs at return,
                // on ANY exit path

// multiple defers run LIFO

Habit: the line after you acquire something (file, lock, rows) is where its defer goes. Pair them instantly.

Slices & maps

the two containers you'll use every single day

Slices — your everyday list

s := []int{1, 2, 3} // literal
s = append(s, 4, 5) // grow — reassign!
first := s[0]
last := s[len(s)-1]

t := make([]int, 0, 10) // len 0, cap 10
fmt.Println(len(t), cap(t)) // 0 10

Always s = append(s, …). Append returns a (possibly new) slice — dropping the result is a classic bug.

Slicing & copying — views vs clones

s := []int{10, 20, 30, 40}
mid := s[1:3] // [20 30] — a VIEW
mid[0] = 99   // s changed too!

clone := make([]int, len(s))
copy(clone, s) // independent now
// or: clone := slices.Clone(s)

Remember: s[a:b] shares memory with s. Need independence? copy it.

Maps — put, get, delete, check

ages := map[string]int{} // ready to use
ages["ada"] = 36         // put / update
delete(ages, "ada")      // remove

v, ok := ages["bob"] // comma-ok!
if !ok {
	fmt.Println("no bob") // v is 0 here
}

Gotcha: reading a missing key gives the zero value, silently. Use , ok when "missing" and "zero" mean different things.

range — all the shapes

for i, v := range slice { } // idx, val
for _, v := range slice { } // values
for i := range slice { }    // indexes
for k, v := range m { }     // map
for i := range 5 { }        // 0..4

Remember: map order is deliberately random — never rely on it. Need order? Sort the keys first.

Which container do I use?

A list of things (almost always)→slice []Notegrows with append, cheap to pass around
Look things up by a key→map map[string]NoteO(1) lookup, no order
Fixed size, known at compile time (rare)→array [3]intyou'll use these maybe twice a year
Need order and key lookup→slice + map togetherslice keeps order, map finds fast

Functions & methods

multiple returns, receivers, and the pointer question

Functions — returning two things

func divide(a, b float64) (float64, error) {
	if b == 0 {
		return 0, errors.New("b is zero")
	}
	return a / b, nil
}

result, err := divide(10, 2)

The Go signature: (value, error). The caller checks err first, then trusts the value.

Variadic & closures

func sum(nums ...int) int { // any #
	total := 0
	for _, n := range nums {
		total += n
	}
	return total
}
sum(1, 2, 3)

count := 0
inc := func() { count++ } // closure:
inc(); inc()              // count == 2

Closures capture variables, not copies — that's what made middleware and defer wg.Done() patterns work.

Methods — functions with a receiver

type Note struct {
	ID   int
	Text string
}

// (n Note) = the receiver ("this")
func (n Note) Loud() string {
	return strings.ToUpper(n.Text)
}

// pointer receiver: can modify
func (n *Note) Rename(t string) {
	n.Text = t
}

Call them the same way: note.Rename("hi") — Go takes the address for you.

Pointers — just two symbols

n := Note{ID: 1}
p := &n         // & = "address of"
p.Text = "hi"   // auto-dereferences 🎉
fmt.Println(*p) // * = "value there"

var q *Note   // nil until assigned
if q != nil { // check before use!
	fmt.Println(q.Text)
}

No pointer arithmetic, no ->. Just: & takes an address, * follows one.

Pointer receiver or value receiver?

The method changes the struct→pointer (n *Note)changes stick
The struct is big (copying is wasteful)→pointer (n *Note)passes 8 bytes, not the whole thing
Small and read-only→value (n Note)like time.Time does
Honestly unsure→pointer — and be consistentdon't mix receiver kinds on one type

Structs & interfaces

your own types, and Go's quiet superpower

Structs — define & create

type Note struct {
	ID   int    `json:"id"`
	Text string `json:"text"`
}

n := Note{ID: 1, Text: "hi"}
n.Text = "hello"

// the constructor convention:
func NewNote(text string) *Note {
	return &Note{Text: text}
}

Backtick tags tell encoders what to call fields: capital Text in Go, lowercase "text" in JSON.

Embedding — composition, not inheritance

type Timestamps struct {
	Created, Updated time.Time
}

type Note struct {
	Timestamps // ← embedded
	Text string
}

n := Note{}
n.Created = time.Now() // promoted!

Go has no inheritance. Embed a type to borrow its fields and methods; that covers most "extends" cravings.

Interfaces — satisfied automatically

type Storer interface {
	Save(n Note) error
}

// No "implements" keyword anywhere:
type FileStore struct{}

func (f FileStore) Save(n Note) error {
	return nil // right method ⇒ Storer ✓
}

Anything with the right method set satisfies the interface — the type never mentions it. That's why swapping a real store for a fake in tests is free.

Do I need an interface?

Only one implementation exists→no — use the concrete typeadd the interface the day a second one appears
Tests need a fake (DB, API client, clock)→yes — a small one, defined where it's used1–3 methods is a healthy size
Reading / writing / printing anything→stdlib already has itio.Reader, io.Writer, fmt.Stringer
Designing a package's API→accept interfaces, return structsflexible inputs, concrete outputs

Errors

no exceptions — errors are values you look at

The rhythm: check, return early

n, err := store.Get(ctx, id)
if err != nil {
	return err // handle or pass it up
}
use(n) // happy path stays un-indented

Yes, you type this a lot. That's by design — every failure path is visible right where it happens.

Wrap with %w — add breadcrumbs

if err != nil {
	return fmt.Errorf(
		"saving note %d: %w", id, err)
}

// errors read like a story:
// "saving note 7: connection refused"

%w not %v: %w keeps the original error inside so errors.Is still works on it.

Sentinels + errors.Is / errors.As

var ErrNotFound = errors.New("not found")

// deep in the store:
return Note{}, ErrNotFound

// in the handler — even if wrapped:
if errors.Is(err, ErrNotFound) {
	http.Error(w, "not found", 404)
}

var pathErr *os.PathError
if errors.As(err, &pathErr) { } // by TYPE

Is = "is it this value?" · As = "is it this type?" Both see through %w wrapping. Never compare with ==.

error or panic?

Bad input, missing row, network down→errorexpected failures are normal life
Programmer bug — "this can't happen"→panicindex out of range does this for you
Writing library / handler code→basically never panicreturn errors, let the caller decide
Unsure→errorpanics are for the impossible, not the inconvenient

Goroutines & channels

Go's superpower — used less often than you'd think

Wait — do I even need a goroutine?

HTTP handler doing one DB query→NOnet/http already runs every request in its own goroutine
Call 3 APIs and combine the answers→YES — fan outtotal time = slowest call, not the sum
Wait for a result OR a timeout/shutdown→YES — goroutine + selectone function can't block on two things
Do work after replying (email, cache warm)→YES — carefullyneeds its own context and error handling
"This loop feels slow" (no measurement)→NO — measure firstspeculative concurrency buys bugs, not speed

Starting a goroutine

go doWork()      // returns immediately

go func() {      // anonymous is common
	fmt.Println("running concurrently!")
}()

// ⚠ main() exiting kills ALL goroutines.
// You must wait for them ↓

A goroutine is cheap (a few KB) — but fire-and-forget without a way to wait or fail is a bug factory.

WaitGroup — wait for N to finish

var wg sync.WaitGroup

for _, url := range urls {
	wg.Add(1) // before starting!
	go func() {
		defer wg.Done() // always runs
		fetch(url)
	}()
}
wg.Wait() // waits for every Done

Pattern: Add(1) outside, defer Done() first line inside, Wait() after the loop.

Channels — typed pipes

ch := make(chan string) // unbuffered

go func() {
	ch <- "hi" // send: waits for reader
}()
msg := <-ch    // receive: waits for send

close(ch) // sender says "no more"
for v := range ch { } // until closed

Arrow points the way data flows: ch <- x puts in, x := <-ch takes out. Only the sender closes.

Buffered vs unbuffered

h := make(chan int) // handshake:
                    // send waits for recv

m := make(chan int, 3) // mailbox, 3 slots
m <- 1                 // no block...
m <- 2                 // ...until full

Unbuffered = synchronize two goroutines. Buffered = let a known number of results pile up (like one slot per worker).

select — whichever is ready first

select {
case msg := <-ch1:
	fmt.Println("ch1:", msg)
case msg := <-ch2:
	fmt.Println("ch2:", msg)
case <-time.After(2 * time.Second):
	fmt.Println("timed out ⏰")
}

This is how you wait on two things at once — results, timeouts, shutdown signals. It's the heart of our server's graceful shutdown.

Mutex — guard shared state

var (
	mu     sync.Mutex
	visits = map[string]int{}
)

func hit(page string) {
	mu.Lock()
	defer mu.Unlock()
	visits[page]++ // one at a time
}

Two goroutines + one map = crash without this. Run go run -race . to catch races in tests.

Real example — fan out, collect

urls := []string{"a.com","b.com","c.com"}
results := make(chan string, len(urls))

for _, u := range urls {
	go func() {
		results <- check(u)
	}()
}
for range urls {
	fmt.Println(<-results) // finish order
}

Payoff: 3 calls × 1s each = ~1 second total, not 3. The buffered channel means no goroutine waits to deliver.

Web & JSON

the four snippets behind our whole CRUD API

A server in 8 lines

mux := http.NewServeMux()

mux.HandleFunc("GET /notes/{id}", getNote)

func getNote(w http.ResponseWriter,
	r *http.Request) {
	id := r.PathValue("id") // from {id}
	fmt.Fprintf(w, "note %s", id)
}

http.ListenAndServe(":8080", mux)

Since Go 1.22 the pattern handles the method AND path params — no router library needed for most APIs.

JSON in, JSON out

type Note struct {
	ID   int    `json:"id"`
	Text string `json:"text"`
}

// request body → struct
var n Note
json.NewDecoder(r.Body).Decode(&n)

// struct → response body
w.Header().Set("Content-Type",
	"application/json")
json.NewEncoder(w).Encode(n)

Fields must be Capitalized or the encoder can't see them — the tag gives the lowercase JSON name.

Status codes — quick picks

w.WriteHeader(http.StatusCreated)  // 201

http.Error(w, "not found",
	http.StatusNotFound)           // 404 + text

// 200 got it    201 created
// 204 deleted   400 bad input
// 404 missing   500 we broke — log it

Order matters: set headers, then WriteHeader, then the body. Writing the body first locks in a 200.

context — the cancel signal

// in a handler: tied to THIS request
ctx := r.Context()
store.Get(ctx, id) // dies with client

// give anything a deadline:
ctx, cancel := context.WithTimeout(
	ctx, 5*time.Second)
defer cancel()

Rule: ctx is always the first parameter, and you pass it down — never store it in a struct field.

Gotcha wall

the ones that bite every new gopher exactly once

💥 Writing to a nil map panics

var m map[string]int
m["a"] = 1 // PANIC: nil map

m = make(map[string]int) // make it first
m["a"] = 1 // ✓

Reading a nil map is fine (zero values). Writing is the crash. make or use a literal {}.

nil slice vs empty slice in JSON

var a []int          // nil
b := []int{}         // empty

json: a → null   😬
json: b → []     ✓ what APIs want

Both work with append and len. Only JSON cares — return []Note{} from list endpoints, not a nil slice.

range hands you a copy

for _, n := range notes {
	n.Text = "edited" // edits the COPY 😬
}

for i := range notes {
	notes[i].Text = "edited" // sticks ✓
}

Mutating in a loop? Go through the index (or use a slice of pointers).

defer snapshots its arguments

x := 1
defer fmt.Println("x was", x) // grabs 1
x = 99
// prints: "x was 1"

defer func() { fmt.Println(x) }()
// closure reads x at run time → 99

Args evaluate at the defer line; the call runs at return. Want the final value? Wrap it in a closure.

Capital letter = public

type Note struct {
	Text string // exported (public)
	slug string // private to its package
}

// Same for funcs, types, methods:
// Save() is public, save() is private.

There's no public/private keyword — the first letter IS the access modifier. JSON encoding only sees Capitals too.

Unused = compile error

x := compute() // ✗ declared, not used

_ = compute()  // deliberately discard ✓
_ "github.com/go-sql-driver/mysql"
// blank import: run its init only

Not a warning — an error. Annoying for 2 days, then you'll love the always-clean codebase.

Strings are bytes, range gives runes

s := "héllo"
len(s)                    // 6 — bytes!
utf8.RuneCountInString(s) // 5 chars

for i, r := range s { }
// r is a rune; i can jump by >1

Slicing a string cuts bytes, which can split a character in half. For real text work, convert to []rune.

append sometimes shares memory

a := []int{1, 2, 3}
b := append(a[:2], 99)
// b = [1 2 99] — and a too! 😱
// (append reused a's backing array)

Appending to a sub-slice can overwrite the original. When branching off a slice, slices.Clone it first.
