Middleware: the Onion

panic recovery, request logs, CORS for your future frontend, and the auth gate

In this chapter you'll learn ✍️

  • the wrap-a-handler pattern behind every Go middleware ever written
  • four middlewares: recover, log, CORS, authenticate
  • smuggling the logged-in user through context
  • why the wrapping order matters

One pattern, infinite middlewares

Some jobs apply to every request: logging, panic safety, CORS headers, checking tokens. Copy-pasting that into 12 handlers would be misery. Instead we wrap the whole router in layers — each layer does its bit, then calls the next one. Requests travel inward through the onion, responses travel back out:

And here is the entire trick — a function that takes a handler and returns a wrapped one:

the shape of all middleware 
func (app *application) myMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// ... do things BEFORE the request is handled ...
		next.ServeHTTP(w, r) // pass the baton inward
		// ... do things AFTER (the response has been written) ...
	})
}

That's a closure captured over next. No interfaces to implement, no framework registry — just functions eating functions. Now the four real ones:

recoverPanic & logRequests

cmd/api/middleware.go 
// recoverPanic turns a panicking handler into a clean 500 response
// instead of a dropped connection.
func (app *application) recoverPanic(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		defer func() {
			if err := recover(); err != nil {
				w.Header().Set("Connection", "close")
				app.serverError(w, r, fmt.Errorf("panic: %v", err))
			}
		}()
		next.ServeHTTP(w, r)
	})
}

Each request runs in its own goroutine; an unrecovered panic in one would print a stack trace and kill that connection with an empty reply. This deferred recover() turns catastrophe into a tidy 500 — and because it's the outermost layer, it catches explosions in every layer beneath it.

cmd/api/middleware.go (continued) 
// logRequests writes one structured log line per request.
func (app *application) logRequests(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		start := time.Now()
		next.ServeHTTP(w, r)
		app.logger.Info("request",
			"method", r.Method,
			"path", r.URL.Path,
			"took", time.Since(start).String())
	})
}

Note the lines around next.ServeHTTP: start the clock before, log after. The middleware sandwich in action.

enableCORS — the “bring your own frontend” switch

Remember the promise: any frontend can plug into gopherit. There's a catch — browsers enforce the same-origin policy: JavaScript served from my-reddit-ui.com is not allowed to call api.gopherit.com… unless the API explicitly says it's fine. That's CORS (Cross-Origin Resource Sharing), and it's just response headers:

cmd/api/middleware.go (continued) 
// enableCORS lets browser frontends on other origins call this API —
// that's what makes gopherit "bring your own frontend".
func (app *application) enableCORS(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.Header().Set("Access-Control-Allow-Origin", "*")
		w.Header().Set("Access-Control-Allow-Headers", "Authorization, Content-Type")
		w.Header().Set("Access-Control-Allow-Methods", "GET, POST, PUT, PATCH, DELETE, OPTIONS")

		// Browsers send a "preflight" OPTIONS request before the real
		// one. Answer it directly with 204 No Content.
		if r.Method == http.MethodOptions {
			w.WriteHeader(http.StatusNoContent)
			return
		}
		next.ServeHTTP(w, r)
	})
}

The preflight wrinkle: before sending a “risky” request (anything with an Authorization header, say), browsers first send an OPTIONS request asking “may I?”. We answer 204 with the allow-headers — watch:

terminal
$ curl -s -i -X OPTIONS localhost:4000/api/v1/posts
HTTP/1.1 204 No Content
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
Access-Control-Allow-Origin: *
The * is a development setting

Allow-Origin: * means “any website may call me” — perfect while you and your friends build frontends. A production API with cookies or private data should echo back a specific allowed origin instead. One-line change in this one middleware — that's the payoff of the onion.

authenticate — reading the wristband

cmd/api/middleware.go (continued) 
// authenticate checks the Authorization header. No header = anonymous
// visitor, carry on. A valid "Bearer <token>" = load the user and stash
// them in the request context. A broken/expired token = 401 right here.
func (app *application) authenticate(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		w.Header().Add("Vary", "Authorization")

		authHeader := r.Header.Get("Authorization")
		if authHeader == "" {
			next.ServeHTTP(w, r) // anonymous is fine for public routes
			return
		}

		token, ok := strings.CutPrefix(authHeader, "Bearer ")
		if !ok || token == "" {
			app.invalidCredentials(w)
			return
		}

		user, err := app.store.UserForToken(token)
		if err != nil {
			switch {
			case errors.Is(err, store.ErrNotFound):
				app.invalidCredentials(w)
			default:
				app.serverError(w, r, err)
			}
			return
		}

		ctx := context.WithValue(r.Context(), userContextKey, user)
		next.ServeHTTP(w, r.WithContext(ctx))
	})
}

Three cases, three outcomes:

  • No header → anonymous browsing is legal (reading feeds needs no account) — pass through.
  • Valid token → load the user and stash them in the request's context, a per-request pocket that travels with r the rest of the way in.
  • Broken or expired token → 401 on the spot; nothing deeper ever runs.
Why the weird private contextKey type?

context.WithValue is a shared pocket — any package could write a value under the string key "user" and collide with ours. Keys of an unexported type (type contextKey string) can't be forged from outside the package. It's the documented idiom, promoted from folklore to convention.

Two small companions finish the story — a gate for routes that require login, and a getter handlers can call:

cmd/api/middleware.go (continued) 
// requireAuth guards a single route: no logged-in user, no entry.
func (app *application) requireAuth(next http.HandlerFunc) http.HandlerFunc {
	return func(w http.ResponseWriter, r *http.Request) {
		if app.currentUser(r) == nil {
			app.authRequired(w)
			return
		}
		next(w, r)
	}
}
cmd/api/middleware.go (continued) 
// currentUser fishes the logged-in user out of the request context.
// Returns nil for anonymous requests.
func (app *application) currentUser(r *http.Request) *store.User {
	user, _ := r.Context().Value(userContextKey).(*store.User)
	return user
}

Assemble the onion

The final routes.go — this is the file's finished form for the whole book, future endpoints included, so you can see the destination:

cmd/api/routes.go — final 
package main

import "net/http"

// routes builds the router. Go 1.22+ ServeMux understands
// "METHOD /path/{param}" patterns, so we need zero external packages.
func (app *application) routes() http.Handler {
	mux := http.NewServeMux()

	// Health
	mux.HandleFunc("GET /api/v1/healthz", app.healthzHandler)

	// Users & auth
	mux.HandleFunc("POST /api/v1/users", app.registerUserHandler)
	mux.HandleFunc("POST /api/v1/tokens", app.loginHandler)

	// Subreddits
	mux.HandleFunc("GET /api/v1/subreddits", app.listSubredditsHandler)
	mux.HandleFunc("POST /api/v1/subreddits", app.requireAuth(app.createSubredditHandler))
	mux.HandleFunc("GET /api/v1/subreddits/{name}", app.getSubredditHandler)
	mux.HandleFunc("GET /api/v1/subreddits/{name}/posts", app.listSubredditPostsHandler)

	// Posts
	mux.HandleFunc("GET /api/v1/posts", app.listPostsHandler)
	mux.HandleFunc("POST /api/v1/posts", app.requireAuth(app.createPostHandler))
	mux.HandleFunc("GET /api/v1/posts/{id}", app.getPostHandler)

	// Comments & votes
	mux.HandleFunc("POST /api/v1/posts/{id}/comments", app.requireAuth(app.createCommentHandler))
	mux.HandleFunc("POST /api/v1/posts/{id}/vote", app.requireAuth(app.votePostHandler))
	mux.HandleFunc("POST /api/v1/comments/{id}/vote", app.requireAuth(app.voteCommentHandler))

	// The middleware onion, outermost first: recover from panics, log
	// every request, add CORS headers, then identify the caller.
	return app.recoverPanic(app.logRequests(app.enableCORS(app.authenticate(mux))))
}
Order matters — read the last line inside-out

recoverPanic(logRequests(enableCORS(authenticate(mux)))): recover is outermost so it catches panics from everything; logging next so even CORS-rejected and 401'd requests get logged; authenticate is innermost because by then we know the request is real. Swap recover inside authenticate and a panicking token lookup crashes the connection unlogged. When in doubt: safety out, knowledge in.

Pin to your brain 📌

  • Middleware = a function that wraps a handler and returns a handler. Sandwich code around next.ServeHTTP.
  • recoverPanic turns crashes into 500s; logRequests times every hit.
  • CORS headers are the permission slip browsers demand before your frontend may call the API.
  • authenticate parses the Bearer token and pockets the user in the request context.
  • Wrap order: recover → log → CORS → auth → router.
