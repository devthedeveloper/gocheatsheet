A Real Project Skeleton

folders that scale, an app struct instead of globals, flags & structured logs

In this chapter you'll learn ✍️

  • the cmd/ + internal/ layout used by serious Go projects
  • dependency injection with a plain struct (no magic, no globals)
  • configuration via flags and logging via log/slog
  • why a bare ListenAndServe is never enough

From playground to project

Chapter 2's single files were fine for experiments, but gopherit will grow to ~15 files. Time to give it bones. This is the moment the real project starts — everything from here to the last page builds on this skeleton.

terminal
$ mkdir -p gopherit/cmd/api gopherit/internal/store
$ cd gopherit
$ go mod init gopherit
go: creating new go.mod: module gopherit
the folder plan 
gopherit/
├── go.mod                  # module name + dependency list
├── cmd/
│   └── api/                # package main — the HTTP layer
│       ├── main.go         #   startup: config, logger, server
│       ├── routes.go       #   the router, one page of URL patterns
│       └── ...handlers     #   users.go, posts.go, ... (coming soon)
└── internal/
    └── store/              # the data layer — all SQL lives here (ch 5)
  • cmd/api/ — “the executable lives here.” If gopherit ever grows a second binary (a CLI admin tool, say), it becomes cmd/admin/. Convention, not law — but every Go developer recognises it on sight.
  • internal/ — this one is law: the Go compiler forbids any other module from importing packages under internal/. It's your “private” keyword for whole packages.

The application struct — DI without a framework

Handlers need shared things: the logger, the config, soon a database. The rookie move is global variables; the enterprise move is a dependency-injection framework. Go's sweet spot is embarrassingly simple: put the shared stuff in one struct, and make your handlers methods on it.

cmd/api/main.go 
// Chapter 3 checkpoint: the project skeleton, before the database
// (chapter 5), JSON helpers (chapter 4) and graceful shutdown
// (chapter 12) get added.
package main

import (
	"flag"
	"fmt"
	"log/slog"
	"net/http"
	"os"
	"time"
)

const version = "0.1.0"

// config holds everything tweakable from the outside world.
type config struct {
	port int
	env  string
}

// application is the "toolbox" every handler can reach.
// Handlers hang off it as methods, so they can use the logger and
// config without any global variables.
type application struct {
	config config
	logger *slog.Logger
}

func main() {
	var cfg config
	flag.IntVar(&cfg.port, "port", 4000, "API server port")
	flag.StringVar(&cfg.env, "env", "development", "Environment (development|production)")
	flag.Parse()

	logger := slog.New(slog.NewTextHandler(os.Stdout, nil))

	app := &application{
		config: cfg,
		logger: logger,
	}

	srv := &http.Server{
		Addr:         fmt.Sprintf(":%d", cfg.port),
		Handler:      app.routes(),
		IdleTimeout:  time.Minute,
		ReadTimeout:  5 * time.Second,
		WriteTimeout: 10 * time.Second,
	}

	logger.Info("starting server", "addr", srv.Addr, "env", cfg.env, "version", version)
	err := srv.ListenAndServe()
	logger.Error("server stopped", "error", err)
	os.Exit(1)
}

Walk it slowly — this file is the template for every Go service you'll ever write:

  • config is filled from command-line flags: go run ./cmd/api -port 8080. Later you can read env vars into the same struct; handlers won't notice.
  • slog is the standard structured logger — instead of a sentence, it logs key=value pairs that log tools can search.
  • application holds both. Because healthzHandler is a method on it, the handler reaches the logger as app.logger. No globals, and in tests we can build an application with fakes.
  • The http.Server struct replaces bare ListenAndServe(addr, mux) so we can set timeouts. A server without timeouts lets one slow (or evil) client hold a connection open forever — set them once, forget them forever.
The v0.1 fine print

When this server stops, it stops hard — in-flight requests get cut off. Good enough while we build; in chapter 12 we'll teach it to shut down gracefully, finishing every request before the lights go out.

Routes and the first real endpoint

The router gets its own file. It'll grow by a line or two per chapter, and having the whole URL map on one page is a gift to future-you:

cmd/api/routes.go 
package main

import "net/http"

// routes builds the router. It grows a line or two every chapter.
func (app *application) routes() http.Handler {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /api/v1/healthz", app.healthzHandler)
	return mux
}

And the health check — the “is anyone home?” endpoint every service needs (load balancers ping it, uptime bots ping it, you ping it at 2 a.m.):

cmd/api/healthz.go 
package main

import (
	"encoding/json"
	"net/http"
)

// healthzHandler — GET /api/v1/healthz
func (app *application) healthzHandler(w http.ResponseWriter, r *http.Request) {
	data := map[string]string{
		"status":      "available",
		"environment": app.config.env,
		"version":     version,
	}

	js, err := json.Marshal(data)
	if err != nil {
		app.logger.Error("marshal failed", "error", err)
		http.Error(w, "internal error", http.StatusInternalServerError)
		return
	}

	w.Header().Set("Content-Type", "application/json")
	w.Write(js)
}
Every .go file begins with package main

See how routes.go and healthz.go above each open with package main and their own import block? That first line is not optional — a Go file missing it fails to compile with the cryptic expected 'package', found 'func'. Every file in one folder shares the same package, so all three files in cmd/api/ start with package main. When a later card is labelled new file but only shows a function to save space, keep its package line and imports at the top. Stuck? The complete, compiling file is always in code/gopherit/.

Why /api/v1/ in every path?

The v1 is an API version. The day you need to change a response shape in a breaking way, you mount the new behaviour at /api/v2/... and old frontends keep working against v1. Versioning is free to add on day one and miserable to retrofit on day 400.

Pin to your brain 📌

  • cmd/api = the binary; internal/ = compiler-enforced private packages.
  • Shared dependencies live in an application struct; handlers are its methods.
  • Config in via flags; logs out via slog with key=value pairs.
  • Always configure an http.Server with timeouts — never bare ListenAndServe.
  • Version your paths (/api/v1/) from day one.
