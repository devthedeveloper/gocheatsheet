Users & Passwords

bcrypt, fields that can never leak, and gopherit's first real endpoint

In this chapter you'll learn ✍️

  • why passwords are hashed, never encrypted, never stored
  • json:"-" — the tag that makes leaks impossible
  • INSERT … RETURNING, and catching UNIQUE violations cleanly
  • the full register endpoint: validate → hash → insert → 201

The one rule you never break

Repeat after me

We never store the password. Not encrypted. Not “just for debugging”. Never. We store a hash — a one-way fingerprint. Login re-hashes whatever the user typed and compares fingerprints. If our database leaks, attackers get fingerprints, not keys.

Not any hash, though. Fast hashes (SHA-256, MD5) can be guessed at billions per second on a GPU. Password hashing uses deliberately slow, salted algorithms. We use bcrypt — battle-tested since 1999 and one import away:

the whole bcrypt API we need 
import "golang.org/x/crypto/bcrypt"

// registering: plaintext -> hash (cost 12 ≈ a quarter second, on purpose)
hash, err := bcrypt.GenerateFromPassword([]byte(password), 12)

// logging in: does this attempt match the stored hash?
err = bcrypt.CompareHashAndPassword(storedHash, []byte(attempt))
// err == nil        -> correct password
// err != nil        -> wrong password (or corrupted hash)
  • Salted automatically: bcrypt mixes random bytes into every hash, so two users with the same password get different hashes, and rainbow tables are useless.
  • Slow on purpose: cost 12 means ~250ms per attempt. A user won't notice at login; someone guessing millions of passwords absolutely will.
  • 72-byte input limit: that's why our validation caps passwords at 72 chars.

The User model — with a tamper-proof pocket

internal/store/users.go 
package store

import (
	"database/sql"
	"errors"
)

// User is an account on gopherit. PasswordHash is tagged json:"-" so it
// can NEVER leak into an API response, even by accident.
type User struct {
	ID           int64  `json:"id"`
	Username     string `json:"username"`
	Email        string `json:"email"`
	CreatedAt    string `json:"created_at"`
	PasswordHash []byte `json:"-"`
}

// CreateUser inserts a new user and returns the stored row.
// Returns ErrDuplicate when the username or email is already taken.
func (s *Store) CreateUser(username, email string, passwordHash []byte) (*User, error) {
	var u User
	err := s.db.QueryRow(`
        INSERT INTO users (username, email, password_hash)
        VALUES (?, ?, ?)
        RETURNING id, username, email, created_at`,
		username, email, passwordHash,
	).Scan(&u.ID, &u.Username, &u.Email, &u.CreatedAt)

	if err != nil {
		if isUniqueViolation(err) {
			return nil, ErrDuplicate
		}
		return nil, err
	}
	return &u, nil
}

// UserByEmail fetches a user (including the password hash, for login).
// Returns ErrNotFound when no such account exists.
func (s *Store) UserByEmail(email string) (*User, error) {
	var u User
	err := s.db.QueryRow(`
        SELECT id, username, email, created_at, password_hash
        FROM users
        WHERE email = ?`,
		email,
	).Scan(&u.ID, &u.Username, &u.Email, &u.CreatedAt, &u.PasswordHash)

	if errors.Is(err, sql.ErrNoRows) {
		return nil, ErrNotFound
	}
	if err != nil {
		return nil, err
	}
	return &u, nil
}

Details worth savoring:

  • PasswordHash []byte `json:"-"` — the dash tells encoding/json: this field does not exist as far as you're concerned. Even if a future you writes writeJSON(w, 200, envelope{"user": user}) at 3 a.m., the hash cannot end up in a response. Guardrails beat vigilance.
  • RETURNING — one round trip inserts the row and hands back the DB-generated bits (id, created_at). No “insert then select” dance.
  • isUniqueViolation → ErrDuplicate — remember the sentinel-error vocabulary: the store reports “duplicate”, and the handler decides that means HTTP 409.

The register handler

cmd/api/users.go 
// registerUserHandler — POST /api/v1/users
// Body: {"username": "...", "email": "...", "password": "..."}
func (app *application) registerUserHandler(w http.ResponseWriter, r *http.Request) {
	var input struct {
		Username string `json:"username"`
		Email    string `json:"email"`
		Password string `json:"password"`
	}
	if err := app.readJSON(w, r, &input); err != nil {
		app.badRequest(w, err)
		return
	}

	input.Username = strings.TrimSpace(input.Username)
	input.Email = strings.TrimSpace(strings.ToLower(input.Email))

	p := make(problems)
	p.check(len(input.Username) >= 3, "username", "must be at least 3 characters")
	p.check(len(input.Username) <= 30, "username", "must be at most 30 characters")
	p.check(!strings.ContainsAny(input.Username, " \t\n"), "username", "must not contain spaces")
	p.check(strings.Count(input.Email, "@") == 1, "email", "must be a valid email address")
	p.check(len(input.Password) >= 8, "password", "must be at least 8 characters")
	p.check(len(input.Password) <= 72, "password", "must be at most 72 characters")
	if len(p) > 0 {
		app.failedValidation(w, p)
		return
	}

	// bcrypt: slow on purpose, salted automatically. Cost 12 ≈ 250ms.
	hash, err := bcrypt.GenerateFromPassword([]byte(input.Password), 12)
	if err != nil {
		app.serverError(w, r, err)
		return
	}

	user, err := app.store.CreateUser(input.Username, input.Email, hash)
	if err != nil {
		switch {
		case errors.Is(err, store.ErrDuplicate):
			app.conflict(w, "a user with this username or email already exists")
		default:
			app.serverError(w, r, err)
		}
		return
	}

	app.writeJSON(w, http.StatusCreated, envelope{"user": user})
}

Every mutating handler in gopherit follows this exact rhythm — learn it once, reuse it eleven times:

⟶

The anonymous input struct

Notice var input struct{...} declared inside the handler. The JSON a client sends (with a password!) and the struct we store are different shapes — separate types keep them from bleeding into each other. Inline structs are perfect for use-once request shapes.

Register the route (one new line in routes.go) and it's live:

cmd/api/routes.go — add 
mux.HandleFunc("POST /api/v1/users", app.registerUserHandler)

Why is email validation just “count the @”?

Full RFC-5322 email validation is a famous rabbit hole (the “correct” regex is two pages long, and still can't tell you the inbox exists). Real apps validate by sending a confirmation link. For gopherit, “exactly one @, trimmed, lower-cased” filters out the honest mistakes — good enough, honest about it.

Pin to your brain 📌

  • Store bcrypt hashes, never passwords. Cost 12. Compare with CompareHashAndPassword.
  • json:"-" makes leaking a field structurally impossible.
  • INSERT … RETURNING = create and read back in one query.
  • Handler rhythm: read → validate → store → respond. 400/422/409/201.
  • Request structs and storage structs are different types — keep them separate.
