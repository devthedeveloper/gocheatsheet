A Database That Fits in a File

SQLite without a C compiler, database/sql, and the schema behind every Reddit

In this chapter you'll learn ✍️

  • why SQLite is the perfect learning (and often production!) database
  • how database/sql + a driver actually fit together
  • gopherit's six-table schema — foreign keys, uniques, checks
  • the store: one struct that owns every query in the app

Why SQLite (and why this driver)

Postgres and MySQL are servers: install, configure, run, connect. SQLite is a library — your database is literally one file on disk (gopherit.db), and the engine runs inside your process. No installation, no ports, no passwords. It's not a toy: SQLite sits inside every phone, browser and airliner, and happily serves small-to-medium websites.

We'll use the modernc.org/sqlite driver, which is SQLite translated to pure Go — go build just works on Windows, Mac and Linux, no C toolchain (“cgo”) required. Two commands add our only real dependencies:

terminal
$ go get modernc.org/sqlite
$ go get golang.org/x/crypto        # bcrypt — we'll need it next chapter
$ go mod tidy
How database/sql + drivers fit together

Go's std lib ships database/sql — a generic API (Query, Exec, transactions, connection pooling) with no database inside. Drivers register themselves by name when imported. That's why the import looks weird: _ "modernc.org/sqlite" — the underscore means “run this package's init (which registers the "sqlite" driver), I won't call it directly.” Your code talks only to database/sql — swapping databases later barely changes it.

Every Reddit-shaped app is these six tables. Arrows mean “points at” (a foreign key):

Here's the whole store bootstrap — connection, pragmas, and the schema, annotated below:

internal/store/store.go 
// Package store is the data layer of gopherit. Everything that touches
// the database lives here, so the HTTP handlers never write SQL.
package store

import (
	"database/sql"
	"errors"
	"strings"

	_ "modernc.org/sqlite" // registers the "sqlite" driver with database/sql
)

// Sentinel errors. Handlers compare against these with errors.Is and
// translate them into the right HTTP status codes.
var (
	ErrNotFound      = errors.New("store: not found")
	ErrDuplicate     = errors.New("store: duplicate value")
	ErrInvalidParent = errors.New("store: parent comment does not belong to this post")
)

// Store wraps the database connection pool. Every model gets methods on
// this one struct: s.CreateUser(...), s.ListPosts(...), and so on.
type Store struct {
	db *sql.DB
}

// Open connects to the SQLite file at dsn, applies pragmas and runs the
// schema migration. Call Close when the application shuts down.
func Open(dsn string) (*Store, error) {
	db, err := sql.Open("sqlite", dsn+"?_pragma=foreign_keys(1)&_pragma=journal_mode(WAL)&_pragma=busy_timeout(5000)")
	if err != nil {
		return nil, err
	}

	// SQLite allows only one writer at a time. Limiting the pool to a
	// single connection sidesteps "database is locked" errors entirely.
	db.SetMaxOpenConns(1)

	if err := db.Ping(); err != nil {
		return nil, err
	}
	if _, err := db.Exec(schema); err != nil {
		return nil, err
	}
	return &Store{db: db}, nil
}

// Close releases the underlying connection pool.
func (s *Store) Close() error {
	return s.db.Close()
}

// isUniqueViolation reports whether err was caused by a UNIQUE constraint,
// e.g. registering a username that is already taken.
func isUniqueViolation(err error) bool {
	return err != nil && strings.Contains(err.Error(), "UNIQUE constraint failed")
}

// schema is executed on every start-up. Thanks to IF NOT EXISTS it is a
// no-op when the tables are already there — a poor man's migration that
// is perfect for a learning project.
const schema = `
CREATE TABLE IF NOT EXISTS users (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    username      TEXT    NOT NULL UNIQUE COLLATE NOCASE,
    email         TEXT    NOT NULL UNIQUE COLLATE NOCASE,
    password_hash BLOB    NOT NULL,
    created_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);

CREATE TABLE IF NOT EXISTS tokens (
    hash       BLOB PRIMARY KEY,
    user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    expires_at TEXT    NOT NULL
);

CREATE TABLE IF NOT EXISTS subreddits (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT    NOT NULL UNIQUE COLLATE NOCASE,
    title       TEXT    NOT NULL,
    description TEXT    NOT NULL DEFAULT '',
    creator_id  INTEGER NOT NULL REFERENCES users(id),
    created_at  TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);

CREATE TABLE IF NOT EXISTS posts (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    subreddit_id INTEGER NOT NULL REFERENCES subreddits(id) ON DELETE CASCADE,
    author_id    INTEGER NOT NULL REFERENCES users(id),
    title        TEXT    NOT NULL,
    body         TEXT    NOT NULL DEFAULT '',
    url          TEXT    NOT NULL DEFAULT '',
    score        INTEGER NOT NULL DEFAULT 0,
    created_at   TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);

CREATE TABLE IF NOT EXISTS comments (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    post_id    INTEGER NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
    author_id  INTEGER NOT NULL REFERENCES users(id),
    parent_id  INTEGER REFERENCES comments(id) ON DELETE CASCADE,
    body       TEXT    NOT NULL,
    score      INTEGER NOT NULL DEFAULT 0,
    created_at TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);

CREATE TABLE IF NOT EXISTS votes (
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    target_type TEXT    NOT NULL CHECK (target_type IN ('post','comment')),
    target_id   INTEGER NOT NULL,
    value       INTEGER NOT NULL CHECK (value IN (-1, 1)),
    PRIMARY KEY (user_id, target_type, target_id)
);

CREATE INDEX IF NOT EXISTS idx_posts_subreddit ON posts(subreddit_id);
CREATE INDEX IF NOT EXISTS idx_comments_post   ON comments(post_id);
`

The SQL vocabulary doing the heavy lifting:

PieceWhat it buys us
INTEGER PRIMARY KEY AUTOINCREMENTauto-numbered ids: 1, 2, 3…
UNIQUE COLLATE NOCASE“Ada” and “ada” are the same username — no lookalike accounts
REFERENCES users(id)a foreign key: the DB refuses a post whose author doesn't exist
ON DELETE CASCADEdelete a post → its comments & votes vanish too, automatically
CHECK (value IN (-1, 1))the DB itself rejects a vote of +5 — last line of defense
PRIMARY KEY (user_id, target_type, target_id)one vote per user per thing — enforced by the database, not by hope
DEFAULT (strftime(...))timestamps stamp themselves, in UTC, in RFC-3339 format
Why store timestamps as RFC-3339 strings?

SQLite has no real date type — you pick TEXT, INTEGER or REAL. We store 2026-07-07T15:45:08Z strings because they're human-readable in any DB browser, they're already exactly what JSON wants, and — neat property — UTC RFC-3339 strings sort alphabetically in time order, so expires_at > ? comparisons just work. We'll cash in on that in chapter 7.

Three pragmas you must not skip

foreign_keys(1) — SQLite ships with FK enforcement off for historical reasons. Turn it on or your REFERENCES lines are decoration.
journal_mode(WAL) — write-ahead logging: readers don't block the writer. Just better.
busy_timeout(5000) — when the file is briefly locked, wait up to 5s instead of instantly failing with “database is locked”.

And db.SetMaxOpenConns(1): SQLite allows one writer at a time; giving the pool a single connection makes lock errors impossible by construction. Plenty for a learning app.

“Poor man's migrations”

Real projects use numbered migration files (0001_create_users.sql, 0002_add_karma.sql…) with a tool like goose or golang-migrate so schema changes are replayable and reversible. Our CREATE TABLE IF NOT EXISTS restart-safe schema is the training-wheels version — perfect here, worth outgrowing later.

Sentinel errors — the store's vocabulary

Look again at the top of store.go: ErrNotFound, ErrDuplicate, ErrInvalidParent. These are sentinel errors — named values the store returns instead of raw SQL errors. The handler layer matches them with errors.Is and picks status codes:

store saysErrNotFound
⟶
handler says404
·
store saysErrDuplicate
⟶
handler says409

The store never mentions HTTP; handlers never mention SQL. That's the layer boundary from chapter 1, enforced by vocabulary.

Wire it into main

Three additions to main.go: a -dsn flag, opening the store, and handing it to the application struct:

cmd/api/main.go — the new bits 
type config struct {
	port int
	env  string
	dsn  string // NEW: where the SQLite file lives
}

type application struct {
	config config
	logger *slog.Logger
	store  *store.Store // NEW: the data layer, one field away from every handler
}

func main() {
	var cfg config
	flag.IntVar(&cfg.port, "port", 4000, "API server port")
	flag.StringVar(&cfg.env, "env", "development", "Environment (development|production)")
	flag.StringVar(&cfg.dsn, "dsn", "gopherit.db", "SQLite database file") // NEW
	flag.Parse()

	logger := slog.New(slog.NewTextHandler(os.Stdout, nil))

	st, err := store.Open(cfg.dsn) // NEW: connect + migrate
	if err != nil {
		logger.Error("cannot open database", "error", err)
		os.Exit(1)
	}
	defer st.Close()
	logger.Info("database ready", "dsn", cfg.dsn)

	app := &application{
		config: cfg,
		logger: logger,
		store:  st, // NEW
	}
	// ... http.Server setup unchanged ...
}

Peek inside your database anytime

Install any SQLite browser (DB Browser for SQLite is lovely) and open gopherit.db — you'll literally watch rows appear as you curl. Add *.db to your .gitignore; databases don't belong in git.

Pin to your brain 📌

  • SQLite = the whole database in one file, engine inside your process. modernc driver = no C compiler.
  • database/sql is the generic API; the blank-import driver plugs into it.
  • Let the schema defend itself: UNIQUE, CHECK, foreign keys, composite primary keys.
  • Pragmas: foreign_keys ON, WAL, busy_timeout. Pool size 1 for SQLite.
  • Store speaks sentinel errors; handlers translate them to status codes.
