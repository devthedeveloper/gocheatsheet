Login & Bearer Tokens

mint them, hash them, expire them — auth that a whole team can reason about

In this chapter you'll learn ✍️

  • how token auth works end-to-end (the wristband model 🎟️)
  • minting cryptographically random tokens — and why we store only their hash
  • the login endpoint, and defeating “user enumeration”
  • where JWTs fit in (and why we're not using them here)

The wristband model

HTTP is stateless — chapter 1's forgetful server. So how does “being logged in” work? The same way a music festival works:

show ID onceemail + password
POST /tokens
⟶
get a wristbanda random token,
expires in 7 days
⟶
flash it at every gateAuthorization: Bearer …
on each request

The client (browser, app, curl) keeps the token and attaches it to every request in a header:

the header, exactly 
Authorization: Bearer JMY673HHZGM3UEPTFBNCZ4SQGP

“Bearer” means whoever bears this token is treated as its owner — which is why tokens travel only over HTTPS in production, and why they expire.

Minting tokens — and hashing them too

internal/store/tokens.go 
package store

import (
	"crypto/rand"
	"crypto/sha256"
	"database/sql"
	"errors"
	"time"
)

// CreateToken mints a random bearer token for a user. The client gets the
// plaintext; we keep only a SHA-256 hash, so a stolen database does not
// hand out working logins.
func (s *Store) CreateToken(userID int64, ttl time.Duration) (plaintext, expiresAt string, err error) {
	plaintext = rand.Text() // 26 chars of crypto-random base32 (Go 1.24+)
	hash := sha256.Sum256([]byte(plaintext))
	expiresAt = time.Now().UTC().Add(ttl).Format(time.RFC3339)

	_, err = s.db.Exec(`
        INSERT INTO tokens (hash, user_id, expires_at)
        VALUES (?, ?, ?)`,
		hash[:], userID, expiresAt)
	if err != nil {
		return "", "", err
	}
	return plaintext, expiresAt, nil
}

// UserForToken exchanges a plaintext bearer token for the user it belongs
// to. Returns ErrNotFound for unknown or expired tokens.
func (s *Store) UserForToken(plaintext string) (*User, error) {
	hash := sha256.Sum256([]byte(plaintext))
	now := time.Now().UTC().Format(time.RFC3339)

	var u User
	err := s.db.QueryRow(`
        SELECT u.id, u.username, u.email, u.created_at
        FROM tokens t
        JOIN users  u ON u.id = t.user_id
        WHERE t.hash = ? AND t.expires_at > ?`,
		hash[:], now,
	).Scan(&u.ID, &u.Username, &u.Email, &u.CreatedAt)

	if errors.Is(err, sql.ErrNoRows) {
		return nil, ErrNotFound
	}
	if err != nil {
		return nil, err
	}
	return &u, nil
}
  • rand.Text() (Go 1.24+, from crypto/rand) returns 26 characters of cryptographically secure randomness — ~128 bits, unguessable. Never use math/rand for anything security-shaped; it's predictable by design.
  • We store SHA-256(token), not the token. Wait — chapter 6 said fast hashes are bad?! For passwords, yes: humans pick guessable ones, so hashing must be slow. A 128-bit random token is un-bruteforceable no matter how fast the hash — SHA-256 is safe here, and adds zero login latency. If the DB leaks, the thief holds hashes that open nothing.
  • The expiry check rides on chapter 5's trick: RFC-3339 UTC strings compare correctly as plain strings, so expires_at > ? needs no date math at all.

The login endpoint

cmd/api/users.go — add 
// loginHandler — POST /api/v1/tokens
// Body: {"email": "...", "password": "..."}
// Trades valid credentials for a bearer token that lasts 7 days.
func (app *application) loginHandler(w http.ResponseWriter, r *http.Request) {
	var input struct {
		Email    string `json:"email"`
		Password string `json:"password"`
	}
	if err := app.readJSON(w, r, &input); err != nil {
		app.badRequest(w, err)
		return
	}

	user, err := app.store.UserByEmail(strings.TrimSpace(strings.ToLower(input.Email)))
	if err != nil {
		switch {
		case errors.Is(err, store.ErrNotFound):
			// Same reply as a wrong password — don't reveal which
			// emails have accounts.
			app.invalidCredentials(w)
		default:
			app.serverError(w, r, err)
		}
		return
	}

	err = bcrypt.CompareHashAndPassword(user.PasswordHash, []byte(input.Password))
	if err != nil {
		app.invalidCredentials(w)
		return
	}

	token, expiresAt, err := app.store.CreateToken(user.ID, 7*24*time.Hour)
	if err != nil {
		app.serverError(w, r, err)
		return
	}

	app.writeJSON(w, http.StatusCreated, envelope{
		"token":      token,
		"expires_at": expiresAt,
		"user":       user,
	})
}
The identical-errors trick (user enumeration)

Unknown email and wrong password return the same 401 “invalid credentials”. If they differed, an attacker could probe POST /tokens with a list of emails and learn which ones have accounts here — that's called user enumeration, and it's how “haveibeenpwned” lists get cross-referenced. One boring shared message closes the door.

“Why not JWT?” — the interview question

JWTs are self-contained tokens: the server signs a blob of claims and verifies the signature later — no database lookup. Great for microservices where one auth service issues tokens that ten other services verify.

The cost: a JWT can't be revoked — until it expires, a stolen one works even after “logout” (unless you add a denylist… which is a database lookup, defeating the point). Add algorithm-confusion attacks, clock skew, and key rotation, and JWTs carry real footguns.

Our stateful tokens cost one indexed SELECT per request and give us instant revocation (DELETE FROM tokens = logged out everywhere), trivial code, and nothing to misconfigure. For a single-service API, that's the sweet spot. Learn JWTs when your architecture actually needs them — you'll recognize the moment.

Pin to your brain 📌

  • Login = trade credentials for a random bearer token; send it in Authorization: Bearer ….
  • Random tokens: crypto/rand always. Store the SHA-256, never the plaintext.
  • Fast hash for random tokens, slow hash for human passwords — different threats.
  • Unknown email and wrong password must be indistinguishable (401 “invalid credentials”).
  • Stateful tokens = revocable + simple. JWTs shine across service boundaries, not here.
