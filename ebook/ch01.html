what a REST API actually is, and the map of everything we're about to build

In this chapter you'll learn ✍️

  • how HTTP requests & responses really look on the wire
  • what makes an API “RESTful” (it's simpler than it sounds)
  • the architecture and full endpoint map of gopherit, our Reddit clone
  • how to get your machine ready

New to building APIs and feeling the jump? The one-evening Build Your First Go API warm-up builds a tiny link board first, and makes this book much smoother. Already comfortable? Read on.

So… what is an API?

Imagine Reddit with the skin peeled off. No orange buttons, no fonts, no dark mode. What's left is the part that actually does things: “create an account”, “fetch the hottest posts”, “add my comment to post 42”. An API (Application Programming Interface) is exactly that — a menu of actions a program offers to other programs.

A web API serves that menu over the internet using HTTP — the same protocol your browser uses to load pages. The conversation is always the same shape:

clientbrowser, app, curl…
— request ⟶
serverour Go program
⟵ response —
clientrenders the result

The client asks, the server answers, and then — this is important — the server forgets the conversation ever happened. Every request must carry everything the server needs. That's what “stateless” means, and it's what makes APIs easy to scale.

HTTP, dissected with a pocket knife

HTTP is plain text. Honestly, that's the whole secret. When a client asks Reddit for a post, the bytes that travel down the wire look like this:

an HTTP request, raw 
POST /api/v1/posts HTTP/1.1
Host: gopherit.example.com
Authorization: Bearer JMY673HHZGM3UEPTFBNCZ4SQGP
Content-Type: application/json

{"subreddit": "golang", "title": "Go is fun!"}

Read it top to bottom:

  • POST — the method: what kind of action this is
  • /api/v1/posts — the path: which thing the action targets
  • Headers — metadata, one per line: who you are, what format you're sending…
  • a blank line, then the body — the actual payload, JSON for us

And the server writes back the same kind of letter:

an HTTP response, raw 
HTTP/1.1 201 Created
Content-Type: application/json

{"post": {"id": 7, "title": "Go is fun!", "score": 0}}

That 201 is a status code — a three-digit verdict. You'll use maybe ten of them your whole career:

CodeMeansgopherit uses it for
200 OKhere's what you asked forfetching posts, feeds, votes
201 Createdmade a new thing for younew users, posts, comments
204 No Contentdone, nothing to sayCORS preflight replies
400 Bad RequestI can't even parse thatbroken JSON
401 Unauthorizedwho are you? log in firstmissing/expired token
404 Not Foundno such thing herepost 999999
405 Method Not Allowedpath exists, verb doesn'tDELETE on /posts
409 Conflictclashes with existing datausername already taken
422 Unprocessableparsed fine, values are wrongpassword too short
500 Server Errorwe broke, not youbugs, dead database
rule of thumb: 4xx = the client messed up, 5xx = WE messed up 🙈

REST in one paragraph (promise)

REST (REpresentational State Transfer) sounds academic, but in practice it means five habits:

  • Everything is a resource with a URL. A post lives at /api/v1/posts/42. A community lives at /api/v1/subreddits/golang.
  • URLs are nouns, methods are verbs. Not /getPost?id=42 — instead GET /posts/42. Not /deletePost — DELETE /posts/42.
  • Use HTTP's vocabulary. GET reads, POST creates, PUT/PATCH updates, DELETE deletes. Status codes carry the verdict.
  • Stateless. Every request stands alone, carrying its own auth token.
  • JSON in, JSON out. (REST doesn't require JSON, but the industry decided.)
Spot the smell

POST /api/createNewPostForUser?userId=7 — verbs in the URL, user in a query param, action instead of resource. The RESTful version: POST /api/v1/posts with the user identified by their auth token. Same feature, much cleaner contract.

Meet gopherit 🐹

Over the next eleven chapters we'll build gopherit — the complete backend of a Reddit clone. Users, communities, posts, nested comments, voting, hot ranking, auth, CORS, tests. Backend only, on purpose: when we're done, the API is a clean socket that any frontend can plug into — a React site, a Flutter app, a terminal client, or just curl. Your Reddit, your rules.

Here's the architecture. We keep it to two layers plus a database — small enough to hold in your head, big enough to teach you how real services are shaped:

handlerscmd/api — HTTP in/out,
validation, status codes
⟶
storeinternal/store — all SQL,
returns Go structs
⟶
SQLitegopherit.db,
one file on disk
Why layers?

Handlers know about HTTP but never touch SQL. The store knows SQL but has no idea HTTP exists. When you later swap SQLite for Postgres — or bolt on a CLI that skips HTTP entirely — you only touch one layer. This separation is 80% of “clean architecture” with 20% of the ceremony.

And the tech list is refreshingly short:

NeedWe useNot
HTTP server & routingnet/http (std lib, Go 1.22+ patterns)gin, echo, chi, fiber…
DatabaseSQLite via modernc.org/sqlite (pure Go)Postgres + docker + ORM
Passwordsgolang.org/x/crypto/bcryptrolling our own 💀
Logginglog/slog (std lib)zap, logrus
Testsnet/http/httptest (std lib)testing frameworks
Why no framework? Seriously?

In most languages you need a framework because the standard library is bare. Go is the opposite: net/http ships a production-grade server, and since Go 1.22 the built-in router understands methods and path parameters. Learning the std lib first means every Go framework you meet later (they all wrap net/http) will feel familiar — not magical.

Get your desk ready

Go a bit rusty? 🦫

This book assumes you can read structs, functions, and if err != nil. For anything that slips your mind, keep the gocheatsheet open next to this book — especially its structs, errors and JSON sections.

Pin to your brain 📌

  • HTTP is plain text: method + path + headers + body in, status + headers + body out.
  • REST = resources as URLs, HTTP methods as verbs, stateless requests, JSON bodies.
  • 4xx means the client goofed; 5xx means we goofed.
  • gopherit = handlers layer (HTTP) → store layer (SQL) → SQLite (a file).
  • No frameworks. The standard library carries us the whole way.
