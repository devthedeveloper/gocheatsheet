01 goroutines start here

Three users signed up — three welcome emails go out over real SMTP, at the same time.

💡 Everyday version: go func(){ done <- sendEmail(u) }() is hiring a helper: they start the real SMTP handshake (EHLO, MAIL FROM, RCPT TO, DATA) immediately while you carry on. But main is the shop owner — when the owner locks up, every helper stops mid-call. (Your HTTP server hires these helpers for you, on every request.)

// goroutines: three real welcome emails, sent over real SMTP, concurrently.
//
// Points at a local MailHog container (a real SMTP server for dev/test):
//   docker run -d --name mailhog -p 1025:1025 -p 8025:8025 mailhog/mailhog
// In production you'd point Host/Port at smtp.sendgrid.net:587 and auth
// with "apikey" / your SendGrid API key — net/smtp doesn't change at all.
package main

import (
	"fmt"
	"net/smtp"
)

func sendEmail(to string) error {
	msg := []byte("To: " + to + "\r\n" +
		"Subject: Welcome!\r\n\r\n" +
		"Thanks for signing up.\r\n")
	// a REAL SMTP conversation: connect, EHLO, MAIL FROM,
	// RCPT TO, DATA — over an actual TCP socket to port 1025
	return smtp.SendMail("localhost:1025", nil,
		"noreply@example.com", []string{to}, msg)
}

func main() {
	users := []string{
		"asha@example.com",
		"ravi@example.com",
		"meera@example.com",
	}

	done := make(chan error, len(users))
	for _, u := range users {
		go func() { // fires the real SMTP call concurrently
			done <- sendEmail(u)
		}()
	}

	for range users {
		if err := <-done; err != nil {
			fmt.Println("send failed:", err)
			continue
		}
		fmt.Println("✉️  delivered ✓")
	}
}
📖 Line by line — what & why
func sendEmail(to string) error {

What: a real function that opens a real SMTP connection. Why: goroutines run plain functions — nothing special is needed to make a function "concurrency-ready".

return smtp.SendMail("localhost:1025", nil, "noreply@example.com", []string{to}, msg)

What: the stdlib does a REAL SMTP conversation over a real TCP socket: EHLO, MAIL FROM, RCPT TO, DATA. Why: MailHog listens on 1025 like any real mail server. In production, this exact line points at smtp.sendgrid.net:587 with an auth.PlainAuth — SendGrid speaks standard SMTP.

done := make(chan error, len(users))

What: a buffered channel — one slot per email — collects each goroutine's result. Why: buffered so a goroutine can drop its result and exit even if main hasn't read the previous one yet; nobody blocks waiting to hand off.

go func() { done <- sendEmail(u) }()

What: starts sendEmail in a NEW goroutine and moves to the next line immediately — the SMTP handshake for THIS user hasn't happened yet. Why: so signup #2's email never waits behind signup #1's. All three real SMTP connections open concurrently.

for range users { if err := <-done; ... }

What: main waits for exactly as many results as emails sent, printing each as it lands. Why: this is what keeps main alive long enough for the real network round-trips to finish — same job wg.Wait() does in the next program, just via a channel instead.

✉️  delivered ✓
✉️  delivered ✓

02 wg.Wait() sync

GET /dashboard runs 3 real MySQL queries concurrently — and must respond only when ALL of them have returned.

💡 Everyday version: a WaitGroup is a tally on a whiteboard: wg.Add(1) = “one more query in flight”, wg.Done() = a query crossing itself off, wg.Wait() = you standing at the board, refusing to send the 200 until the tally reads zero.

  1. wg.Add(1) before go — write it on the board before the query leaves, or you might check the board while it still says 0 and respond with half a dashboard.
  2. defer wg.Done() first line inside — the query crosses itself off however it ends, even on a DB error.
  3. wg.Wait() blocks until the count is back to 0 — then, and only then, the response goes out.
// wg.Wait: GET /dashboard runs 3 real MySQL queries concurrently,
// then answers only once ALL three have returned.
package main

import (
	"database/sql"
	"fmt"
	"sync"

	_ "github.com/go-sql-driver/mysql"
)

func setup(db *sql.DB) {
	db.Exec("DROP TABLE IF EXISTS orders, users")
	db.Exec(`CREATE TABLE users (id INT PRIMARY KEY)`)
	db.Exec(`CREATE TABLE orders (
		id INT PRIMARY KEY, total_paisa INT)`)
	for i := 1; i <= 500; i++ {
		db.Exec("INSERT INTO users VALUES (?)", i)
	}
	for i := 1; i <= 1200; i++ {
		db.Exec("INSERT INTO orders VALUES (?, ?)",
			i, 19900+(i%5)*500)
	}
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()
	setup(db) // seed data, so this file is rerunnable standalone

	var (
		wg                          sync.WaitGroup
		userCount, orderCount       int
		revenuePaisa                int64
	)
	fmt.Println("GET /dashboard → 3 real queries, in parallel…")

	wg.Add(1) // one more query in flight
	go func() {
		defer wg.Done() // crossed off, always
		db.QueryRow("SELECT COUNT(*) FROM users").
			Scan(&userCount)
	}()

	wg.Add(1)
	go func() {
		defer wg.Done()
		db.QueryRow("SELECT COUNT(*) FROM orders").
			Scan(&orderCount)
	}()

	wg.Add(1)
	go func() {
		defer wg.Done()
		db.QueryRow("SELECT COALESCE(SUM(total_paisa),0) " +
			"FROM orders").Scan(&revenuePaisa)
	}()

	wg.Wait() // block here until all three queries return

	fmt.Printf("users:   %d\n", userCount)
	fmt.Printf("orders:  %d\n", orderCount)
	fmt.Printf("revenue: ₹%.2f\n", float64(revenuePaisa)/100)
	fmt.Println("200 OK — response assembled ✓")
}
📖 Line by line — what & why
var ( wg sync.WaitGroup; userCount, orderCount int; revenuePaisa int64 )

What: the counter, plus the real variables each query will fill in. Why: this counter will track "how many queries are still in flight" — zero value ready, no setup needed.

wg.Add(1) (before each go func)

What: counter goes 0→1→2→3: "one more real query is running". Why: it MUST happen before `go`, in the parent goroutine. If the query goroutine did its own Add, main could reach Wait while the counter still reads 0 — and respond before the query even started.

go func() { defer wg.Done(); db.QueryRow("SELECT COUNT(*) FROM users").Scan(&userCount) }()

What: fires a real query against MySQL in its own goroutine, and schedules the countdown for whenever it returns. Why: `defer wg.Done()` fires even if the query errors — a broken connection can't leave the counter stuck above 0, which would hang wg.Wait() forever.

db.QueryRow(...).Scan(&revenuePaisa)

What: runs a real aggregate query and copies the single result column into a real variable via a pointer. Why: three genuinely different SQL statements, three genuinely separate connections from the pool — this is real concurrent database load, not three prints pretending to be one.

wg.Wait()

What: main STOPS here. Zero CPU, until the counter is back to 0. Why: the slowest of the three real queries decides when this returns — same principle as network fan-out, just against a database instead of other services.

fmt.Printf("revenue: ₹%.2f\n", ...)

What: runs only after every query has actually returned real data. Why: past Wait, all three variables are GUARANTEED populated from the database — safe to assemble the dashboard response.

terminal
$ go run ./02-waitgroup
GET /dashboard → 3 real queries, in parallel…
users:   500
orders:  1200
revenue: ₹250800.00
200 OK — response assembled ✓

Notice: users: 500, orders: 1200, revenue: ₹250800.00 — real COUNT(*) and SUM(...) against real seeded tables, all three fired at once instead of one after another.

03 mutex vs the wallet race the big one

A cashback promo: 100 concurrent clients, 1,000 hits each, ₹1 per hit. Watch real money vanish.

💡 Everyday version: two requests read balance ₹500 at the same moment, both compute ₹501, both write ₹501 — one rupee gone. At scale that’s the 😱 line below. The mutex is the single bathroom key at a petrol station: Lock() takes the key, everyone queues, Unlock() hands it back. One goroutine touches the balance at a time.

// mutex: 100 concurrent clients earn ₹1 cashback per hit.
package main

import (
	"fmt"
	"sync"
)

func main() {
	const clients, hits = 100, 1000
	var wg sync.WaitGroup

	// BROKEN: every request credits the wallet, no lock
	balance := 0
	for range clients {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range hits {
				balance++ // RACE: read, add, write
			}
		}()
	}
	wg.Wait()

	// FIXED: same traffic, one mutex
	var mu sync.Mutex
	safe := 0
	for range clients {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range hits {
				mu.Lock()
				safe++
				mu.Unlock()
			}
		}()
	}
	wg.Wait()

	fmt.Println("should be: ₹", clients*hits)
	fmt.Println("no mutex:  ₹", balance, "— money vanished 😱")
	fmt.Println("mutex:     ₹", safe, "✓")
}
📖 Line by line — what & why
const clients, hits = 100, 1000

What: 100 concurrent clients, each earning cashback 1,000 times. Why: enough volume that the race loses money on every run. Small numbers can get lucky and hide the bug — that's what makes races evil.

balance := 0

What: one plain int, shared by all 100 goroutines. Why: sharing is the point — and sharing without protection is the bug we're about to watch.

balance++

What: looks like one step; is actually three: READ balance, ADD 1, WRITE it back. Why: two clients both read ₹500, both compute ₹501, both write ₹501 — two credits happened, balance moved by one. Repeat thousands of times: tens of thousands of rupees gone.

var mu sync.Mutex

What: the lock. Zero value ready, like WaitGroup. Why: think: ONE bathroom key for the whole petrol station. It protects whatever you consistently lock around — here, the balance.

mu.Lock()

What: take the key — or stand in line until whoever has it gives it back. Why: from this line to Unlock, no other goroutine can be in this section. Read-add-write becomes one uninterruptible turn.

safe++

What: the same three steps as before — Why: — but now nobody can interleave with them. Every credit lands.

mu.Unlock()

What: hand the key back immediately. Why: keep the locked section as short as possible: everything inside it runs one-goroutine-at-a-time, so a fat locked section becomes your bottleneck.

terminal
$ go run ./03-mutex
should be: ₹ 100000
no mutex:  ₹ 57307 — money vanished 😱
mutex:     ₹ 100000 ✓
# balance++ is really: read, add 1, write —
# two clients interleave those steps and one
# write overwrites the other. Real money, gone.

Extra credit: go run -race ./03-mutex — Go names the exact racing line. This bug passes code review constantly; -race in CI is what actually catches it.

04 channels pipes

Uploaded files, checksummed as they land — real disk writes, real SHA-256, hand to hand.

💡 Everyday version: the channel is the hatch between two desks. Unbuffered means hand-to-hand: the upload goroutine stands there holding the open invoice_jan.pdf until the hasher takes it. That’s why the output alternates instead of uploading everything first.

// channels: uploaded files, checksummed as they land — hand to hand.
package main

import (
	"crypto/sha256"
	"fmt"
	"io"
	"os"
	"path/filepath"
)

func main() {
	dir, _ := os.MkdirTemp("", "uploads")
	defer os.RemoveAll(dir)

	uploads := make(chan string) // the hand-off point: file paths

	go func() { // goroutine A: receives the uploads
		docs := map[string]string{
			"invoice_jan.pdf": "invoice January contents",
			"invoice_feb.pdf": "invoice February contents",
			"salary_slip.pdf": "salary slip contents",
		}
		for name, body := range docs {
			path := filepath.Join(dir, name)
			os.WriteFile(path, []byte(body), 0o644) // REAL disk write
			fmt.Println("upload complete:", name)
			uploads <- path // hand the real file to the hasher
		}
		close(uploads) // no more uploads today
	}()

	for path := range uploads { // main: hashes each real file
		f, _ := os.Open(path)
		h := sha256.New()
		io.Copy(h, f) // REAL bytes read off disk, REAL hash computed
		f.Close()
		fmt.Printf("  sha256 %.12x  %s\n",
			h.Sum(nil), filepath.Base(path))
	}
	fmt.Println("all uploads processed ✓")
}
📖 Line by line — what & why
dir, _ := os.MkdirTemp("", "uploads")

What: creates a REAL temp directory on disk for this run. Why: no placeholder filenames — the files that get hashed below genuinely exist on the filesystem.

uploads := make(chan string)

What: the hand-off point. Capacity 0 — nothing queues invisibly. Why: unbuffered: the sender physically waits for a receiver. Every "upload" is a live hand-off, not a silent pile-up.

os.WriteFile(path, []byte(body), 0o644)

What: a REAL write to disk — the file genuinely exists after this line. Why: stands in for "the upload finished landing on disk", which in a real service is exactly what a multipart form handler does before scanning.

uploads <- path

What: sends the real file path — BLOCKS until the hasher goroutine receives it. Why: this blocking is free backpressure: the uploader can't get more than one file ahead of the hasher.

close(uploads)

What: marks the channel "no more uploads today". Why: always the SENDER closes — only it knows when it's done. This is what lets the receiving loop end instead of hanging forever.

f, _ := os.Open(path); h := sha256.New(); io.Copy(h, f)

What: opens the REAL file and streams its REAL bytes through a REAL SHA-256 hasher. Why: io.Copy reads in chunks — this scales to large files without loading them fully into memory. The resulting hash is genuinely derived from the file's contents.

  sha256 2121a3ff48d8443591818343  invoice_feb.pdf

Notice: those are real sha256 digests of real bytes written to a real temp directory (os.MkdirTemp) — not placeholder strings. This is the shape of a real upload → antivirus/dedupe pipeline.

05 select + timeout wait smart

cart:42 lives in an in-memory cache AND MySQL. Serve whichever answers first — inside a real 100ms budget.

💡 Everyday version: you ordered from two food apps; you’ll eat whichever arrives first, and if nothing shows in 30 minutes you’re making noodles. select watches all three cases at once; time.After is the noodles. The user gets an answer in 100ms no matter what the database is doing.

// select: an in-process cache vs a genuinely slow MySQL query —
// serve whichever answers first, inside a real time budget.
package main

import (
	"database/sql"
	"fmt"
	"time"

	_ "github.com/go-sql-driver/mysql"
)

// a real (tiny) in-memory cache — no network, just a map
var cartCache = map[string]string{
	"cart:42": "3 items (cache)",
}

func cacheGet(key string) <-chan string {
	ch := make(chan string, 1)
	go func() {
		if v, ok := cartCache[key]; ok {
			ch <- v // real map lookup, no artificial delay needed
		}
	}()
	return ch
}

func dbQuery(db *sql.DB, key string) <-chan string {
	ch := make(chan string, 1)
	go func() {
		var dummy int
		// SLEEP(3) runs INSIDE MySQL — a genuinely slow query,
		// not a Go-side time.Sleep pretending to be one
		err := db.QueryRow("SELECT SLEEP(3)").Scan(&dummy)
		if err == nil {
			ch <- key + " → 3 items (db, after a real 3s query)"
		}
	}()
	return ch
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()

	cache := cacheGet("cart:42")
	dbCh := dbQuery(db, "cart:42")

	select {
	case v := <-cache:
		fmt.Println("hit:", v)
	case v := <-dbCh:
		fmt.Println("hit:", v)
	case <-time.After(100 * time.Millisecond):
		fmt.Println("both slow → 504, don't hang")
	}
	fmt.Println("responded within the 100ms budget ✓")
}
📖 Line by line — what & why
var cartCache = map[string]string{"cart:42": ...}

What: a real in-memory cache — just a Go map, no network involved. Why: this is what an actual process-local cache looks like before someone reaches for Redis: a map, a mutex if it's written concurrently, done.

func cacheGet(key string) <-chan string {

What: wraps a real map lookup in a channel so it fits the same shape as the (slow) DB path. Why: select needs channels on every case — even a lookup as fast as this one gets the "return a channel now, deliver later" treatment.

db.QueryRow("SELECT SLEEP(3)").Scan(&dummy)

What: runs a REAL query that genuinely blocks for 3 seconds — inside MySQL itself. Why: this is a standard DBA trick for testing timeouts: SLEEP() makes the server-side delay real, not a Go-side pretend one.

select {

What: waits on ALL cases at once; runs whichever becomes ready FIRST; ignores the rest. Why: one keyword replaces polling loops and "who finished?" bookkeeping.

case <-time.After(100 * time.Millisecond):

What: time.After returns a channel that fires once, 100ms from now. Why: converts "wait for an answer" into "wait AT MOST 100ms" — the user gets a fast 504 instead of a hanging spinner, no matter how slow the real DB query is.

terminal
$ go run ./05-select
hit: 3 items (cache)
responded within the 100ms budget ✓

Notice: the “slow DB” isn’t a Go-side pretend delay — SELECT SLEEP(3) genuinely blocks for 3 real seconds inside MySQL, the same trick DBAs use to test timeouts. The cache (a real map lookup) won before that query got anywhere close to answering.

06 worker pool throughput

9 pending orders need shipping — but the pool is capped at 3 real DB connections.

💡 Everyday version: the 3 connections are baristas; the orders table is the queue. Whoever’s free grabs the next row. Same shape for image resizing, PDF generation, webhook delivery — any pile of jobs where “one goroutine each” would melt something downstream.

// worker pool: 9 pending orders, only 3 DB connections fulfilling them.
package main

import (
	"database/sql"
	"fmt"
	"sync"
	"time"

	_ "github.com/go-sql-driver/mysql"
)

func setup(db *sql.DB) {
	db.Exec("DROP TABLE IF EXISTS fulfillment")
	db.Exec(`CREATE TABLE fulfillment (
		id INT PRIMARY KEY, status VARCHAR(20))`)
	for i := 1; i <= 9; i++ {
		db.Exec("INSERT INTO fulfillment VALUES (?, 'pending')", i)
	}
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()
	db.SetMaxOpenConns(3) // the pool really can't exceed 3 connections
	setup(db)

	start := time.Now()
	orderIDs := make(chan int)
	var wg sync.WaitGroup

	for conn := 1; conn <= 3; conn++ { // 3 real DB connections
		wg.Add(1)
		go func(conn int) {
			defer wg.Done()
			for id := range orderIDs { // until closed
				// a REAL write, through a REAL pooled connection
				_, err := db.Exec(
					"UPDATE fulfillment SET status='shipped' WHERE id=?",
					id)
				if err != nil {
					fmt.Println("update failed:", err)
					continue
				}
				fmt.Printf("conn %d shipped order #%d\n", conn, id)
			}
		}(conn)
	}

	for i := 1; i <= 9; i++ {
		orderIDs <- i
	}
	close(orderIDs)
	wg.Wait()

	var shipped int
	db.QueryRow("SELECT COUNT(*) FROM fulfillment WHERE status='shipped'").
		Scan(&shipped)
	fmt.Printf("%d/9 orders shipped, in %v\n",
		shipped, time.Since(start).Round(time.Millisecond))
}
📖 Line by line — what & why
db.SetMaxOpenConns(3)

What: caps the connection pool at exactly 3 — enforced by database/sql itself. Why: this makes "3 workers" a REAL constraint on the database side, not just "3 goroutines happen to exist" while a 4th sneaks a connection anyway.

orderIDs := make(chan int)

What: the shared queue every worker pulls real order IDs from. Why: the channel IS the work distribution — no scheduler, no "which worker is free?" logic anywhere.

for conn := 1; conn <= 3; conn++ { wg.Add(1); go func(conn int) {

What: hires exactly 3 workers, each with its own real DB connection from the pool. Why: the pool size (3) caps how hard you hit MySQL — the ONE number to tune when throughput changes.

db.Exec("UPDATE fulfillment SET status='shipped' WHERE id=?", id)

What: a REAL write against a REAL row — this order is now actually marked shipped in the table. Why: each worker's connection is genuinely separate; MySQL is handling 3 real concurrent writers, not simulated ones.

close(orderIDs)

What: "all 9 orders queued — nothing more is coming." Why: ends every worker's `range` loop. Forget it and all 3 workers wait forever → wg.Wait() deadlocks.

db.QueryRow("SELECT COUNT(*) FROM fulfillment WHERE status='shipped'").Scan(&shipped)

What: a closing REAL query that verifies the real effect of all 9 writes. Why: this is the proof, not a trust-me print statement — if any write failed, this count would show it.

terminal
$ go run ./06-workerpool

conn 1 shipped order #2
conn 3 shipped order #4
conn 2 shipped order #3
conn 1 shipped order #5
conn 2 shipped order #6
conn 3 shipped order #7
conn 1 shipped order #8
conn 2 shipped order #9

Notice: db.SetMaxOpenConns(3) makes the cap real, not just “3 goroutines happen to exist” — MySQL genuinely never sees a 4th connection. A closing SELECT COUNT(*) ... WHERE status='shipped' proves all 9 rows actually changed.

07 context please stop

A report endpoint with a 400ms budget — a genuinely slow query gets cancelled mid-flight, for real.

💡 Everyday version: the customer walked out, so the kitchen stops cooking their order. The context is the order ticket every function holds a copy of; when the 400ms timer rings, QueryContext really tells MySQL to abandon the query instead of Go just giving up and hoping.

// context: a slow report query, cancelled by the request's real deadline.
package main

import (
	"context"
	"database/sql"
	"fmt"
	"time"

	_ "github.com/go-sql-driver/mysql"
)

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()

	// this endpoint has a 400ms budget
	ctx, cancel := context.WithTimeout(
		context.Background(), 400*time.Millisecond)
	defer cancel() // always, even on success

	fmt.Println("generating report (query takes 5s)…")
	start := time.Now()

	// SELECT SLEEP(5) is a REAL, genuinely slow query — and
	// QueryContext really cancels it on the wire when ctx expires
	var dummy int
	err = db.QueryRowContext(ctx, "SELECT SLEEP(5)").Scan(&dummy)

	fmt.Println("took:", time.Since(start).Round(time.Millisecond))
	if err != nil {
		fmt.Println("aborting report:", err)
		fmt.Println("handler: told client 504, moved on")
		return
	}
	fmt.Println("report ready (this line never runs)")
}
📖 Line by line — what & why
ctx, cancel := context.WithTimeout(context.Background(), 400*time.Millisecond)

What: a context whose Done() channel closes automatically after a real 400ms. Why: in a real handler you'd wrap r.Context() instead, so a disconnecting client ALSO cancels the query — same mechanism either way.

defer cancel()

What: releases the context's internal timer when main exits. Why: every WithTimeout leaks a timer+goroutine until cancelled. `go vet` flags a missing cancel; it's non-negotiable.

db.QueryRowContext(ctx, "SELECT SLEEP(5)").Scan(&dummy)

What: runs a REAL 5-second query — but passes ctx in, so the driver can really abort it. Why: THE key idea: QueryContext doesn't just give up locally and hope; the mysql driver actually tells the server to stop, or drops the connection, when ctx fires.

fmt.Println("took:", time.Since(start)...)

What: measures real wall-clock time. Why: the number printed — ~401ms — is the proof: the "5 second" query genuinely got cut off at the 400ms mark, not run to completion in the background.

if err != nil { ... "aborting report", err) }

What: err here really is context.DeadlineExceeded, wrapped by the driver. Why: in a real handler this becomes a 504 — you know EXACTLY why the request failed, not just that it did.

terminal
$ go run ./07-context
generating report (query takes 5s)…
took: 401ms
aborting report: context deadline exceeded
handler: told client 504, moved on

Notice: the query was SELECT SLEEP(5) — a real 5-second query — and the program returned in 401ms with the driver’s own context deadline exceeded error. That’s a real cancellation on the wire, not a timer racing a fake sleep.

08 errgroup x/sync

POST /checkout fans out to 3 real DB updates — inventory, payment, shipping. The wallet genuinely doesn’t have enough.

💡 Everyday version: a team lead sends three devs off and waits at standup. Everyone reports; the lead hears the FIRST blocker. It’s the WaitGroup whiteboard from program 02, but workers can also file complaints — which is why it’s the default for real service fan-out.

// errgroup: POST /checkout fans out to 3 real DB updates; payment declines.
// needs: go get golang.org/x/sync/errgroup
package main

import (
	"database/sql"
	"errors"
	"fmt"

	_ "github.com/go-sql-driver/mysql"
	"golang.org/x/sync/errgroup"
)

func setup(db *sql.DB) {
	db.Exec("DROP TABLE IF EXISTS inventory, wallets, shipments")
	db.Exec(`CREATE TABLE inventory (sku VARCHAR(20) PRIMARY KEY, stock INT)`)
	db.Exec(`CREATE TABLE wallets (user_id INT PRIMARY KEY, balance_paisa INT)`)
	db.Exec(`CREATE TABLE shipments (order_id INT PRIMARY KEY, sku VARCHAR(20))`)
	db.Exec("INSERT INTO inventory VALUES ('boat-airdopes', 40)")
	db.Exec("INSERT INTO wallets VALUES (7, 5000)") // ₹50 — not enough
}

func reserveStock(db *sql.DB, sku string, qty int) error {
	res, err := db.Exec(
		"UPDATE inventory SET stock = stock - ? WHERE sku=? AND stock >= ?",
		qty, sku, qty)
	if err != nil {
		return err
	}
	if n, _ := res.RowsAffected(); n == 0 {
		return errors.New("inventory-service: out of stock")
	}
	fmt.Println("inventory-service → reserved", qty, sku)
	return nil
}

func chargeWallet(db *sql.DB, userID, amountPaisa int) error {
	// a REAL conditional UPDATE: only succeeds if balance covers it
	res, err := db.Exec(
		"UPDATE wallets SET balance_paisa = balance_paisa - ? "+
			"WHERE user_id=? AND balance_paisa >= ?",
		amountPaisa, userID, amountPaisa)
	if err != nil {
		return err
	}
	if n, _ := res.RowsAffected(); n == 0 {
		return errors.New("payment-service: insufficient balance")
	}
	fmt.Println("payment-service → charged ₹", amountPaisa/100)
	return nil
}

func createShipment(db *sql.DB, orderID int, sku string) error {
	_, err := db.Exec(
		"INSERT INTO shipments VALUES (?, ?)", orderID, sku)
	if err == nil {
		fmt.Println("shipping-service → shipment created")
	}
	return err
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()
	setup(db)

	var g errgroup.Group
	g.Go(func() error { return reserveStock(db, "boat-airdopes", 1) })
	g.Go(func() error { return chargeWallet(db, 7, 249900) }) // ₹2499, wallet has ₹50
	g.Go(func() error { return createShipment(db, 1042, "boat-airdopes") })

	if err := g.Wait(); err != nil {
		fmt.Println("checkout failed:", err)
		fmt.Println("→ respond 402, release the reserved stock")
		db.Exec("UPDATE inventory SET stock = stock + 1 WHERE sku='boat-airdopes'")
	}
}
📖 Line by line — what & why
var g errgroup.Group

What: the group — zero value ready. Think: WaitGroup with an error slot. Why: from golang.org/x/sync — the one "extra" import that's on every backend team's go.mod anyway.

g.Go(func() error { return reserveStock(db, "boat-airdopes", 1) })

What: starts a REAL conditional UPDATE against the inventory table in its own goroutine. Why: this one line replaces program 02's Add(1) + go + defer Done() — and adds the error plumbing WaitGroup can't do.

res, err := db.Exec("UPDATE wallets SET balance_paisa = balance_paisa - ? WHERE user_id=? AND balance_paisa >= ?", ...)

What: a REAL atomic conditional write: the row updates ONLY if the balance genuinely covers the charge. Why: no application-side "check then update" race here — MySQL evaluates the WHERE and the SET together, in one real statement.

if n, _ := res.RowsAffected(); n == 0 { return errors.New(...) }

What: RowsAffected tells you whether the real UPDATE actually matched a row. Why: zero rows affected here means the WHERE failed — genuinely insufficient balance, not a made-up error string.

if err := g.Wait(); err != nil {

What: blocks like wg.Wait(), then hands you the FIRST real error from the three real DB calls. Why: one line decides the whole checkout's fate — no collecting errors from three channels by hand.

db.Exec("UPDATE inventory SET stock = stock + 1 WHERE sku='boat-airdopes'")

What: a REAL compensating write undoing the real reservation. Why: inventory-service already committed its change and said OK — a failed checkout must genuinely reverse that, with another real statement.

terminal
$ go run ./08-errgroup
inventory-service → reserved 1 boat-airdopes
shipping-service → shipment created
checkout failed: payment-service: insufficient balance
→ respond 402, release the reserved stock

Notice: the decline isn’t a fake string check — UPDATE wallets SET balance_paisa = balance_paisa - ? WHERE balance_paisa >= ? is a real conditional write; MySQL reports 0 rows affected because ₹50 truly can’t cover ₹2,499. Inventory and shipping had already succeeded, which is why the failure path explicitly gives the stock back.

09 JSON both ways encoding

Decode a real request body, really insert it, then marshal the row MySQL actually stored.

💡 Everyday version: your struct is the organized shelf; JSON is the cardboard box on the truck. Unmarshal = unpacking the client’s box onto your shelf; the INSERT is putting it on a real shelf (the database); Marshal packs the response from what’s really there now — not from the request you started with.

// json: decode a real request body, save it, marshal the real DB row back.
package main

import (
	"database/sql"
	"encoding/json"
	"fmt"

	_ "github.com/go-sql-driver/mysql"
)

type Order struct {
	ID     int    `json:"id"`
	UserID int    `json:"user_id"`
	Item   string `json:"item"`
	Coupon string `json:"coupon,omitempty"`
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()
	db.Exec("DROP TABLE IF EXISTS json_orders")
	db.Exec(`CREATE TABLE json_orders (
		id INT AUTO_INCREMENT PRIMARY KEY,
		user_id INT, item VARCHAR(80))`)

	// exactly what a client POSTs to /orders
	body := `{"user_id": 7, "item": "boat-airdopes"}`
	var in Order
	if err := json.Unmarshal([]byte(body), &in); err != nil {
		panic(err) // in a handler: respond 400
	}
	fmt.Printf("decoded request: %+v\n", in)

	// REAL insert — the DB assigns the id, not the client
	res, err := db.Exec(
		"INSERT INTO json_orders (user_id, item) VALUES (?, ?)",
		in.UserID, in.Item)
	if err != nil {
		panic(err)
	}
	id, _ := res.LastInsertId()

	// REAL read-back — the response reflects what's actually stored
	var out Order
	db.QueryRow("SELECT id, user_id, item FROM json_orders WHERE id=?", id).
		Scan(&out.ID, &out.UserID, &out.Item)

	body2, _ := json.MarshalIndent(out, "", "  ")
	fmt.Println(string(body2))
}
📖 Line by line — what & why
type Order struct { ID int `json:"id"` ... }

What: the struct IS the contract: field names, types, and their wire spelling (the backtick tags). Why: Capitalized fields are required — the json package can't see lowercase ones. The tag maps Go's `UserID` to the API's `user_id`.

json.Unmarshal([]byte(body), &in)

What: parses a real client-shaped JSON string INTO the struct. Why: the error return is your 400 Bad Request. Note there's no `id` in the input — the client doesn't get to choose one.

res, err := db.Exec("INSERT INTO json_orders (user_id, item) VALUES (?, ?)", in.UserID, in.Item)

What: a REAL insert — MySQL, not Go, decides the real id via AUTO_INCREMENT. Why: this is why in.ID stayed 0 after decoding: identity belongs to the database, and this line is where it actually gets assigned.

db.QueryRow("SELECT id, user_id, item FROM json_orders WHERE id=?", id).Scan(...)

What: reads the REAL row back out, into a fresh struct. Why: the response is built from what's truly stored now, not an in-memory echo of the request — catches any DB-side defaults or triggers too.

json.MarshalIndent(out, "", " ")

What: struct → JSON bytes, built from the real row. Why: in a real handler use json.NewEncoder(w).Encode(out) — MarshalIndent here just makes the output readable on this page.

terminal
$ go run ./09-json
decoded request: {ID:0 UserID:7 Item:boat-airdopes Coupon:}
{
  "id": 1,
  "user_id": 7,
  "item": "boat-airdopes"
}

Notice: the response id (1) came from MySQL’s real AUTO_INCREMENT, fetched back with a second real query — proving the server, not the client, owns identity. The empty coupon never entered the response, thanks to omitempty.

10 HTTP server + client one file

GET /orders/{id} — a real handler querying real MySQL, called by a real HTTP client.

💡 Everyday version: both halves of the day job: the handler that answers by asking the database, and the client that asks — which is also you, every time your service calls another one. The server runs in a goroutine; program 01 pays off.

// http: GET /orders/{id} — a real handler querying real MySQL,
// called by a real HTTP client.
package main

import (
	"database/sql"
	"encoding/json"
	"fmt"
	"net/http"
	"time"

	_ "github.com/go-sql-driver/mysql"
)

type Order struct {
	ID     int    `json:"id"`
	Status string `json:"status"`
}

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()
	db.Exec("DROP TABLE IF EXISTS http_orders")
	db.Exec(`CREATE TABLE http_orders (
		id INT PRIMARY KEY, status VARCHAR(20))`)
	db.Exec("INSERT INTO http_orders VALUES (1042, 'shipped')")

	mux := http.NewServeMux()
	mux.HandleFunc("GET /orders/{id}",
		func(w http.ResponseWriter, r *http.Request) {
			id := r.PathValue("id")
			var o Order
			err := db.QueryRow(
				"SELECT id, status FROM http_orders WHERE id=?",
				id).Scan(&o.ID, &o.Status)
			if err == sql.ErrNoRows {
				http.Error(w, "not found", 404)
				return
			}
			json.NewEncoder(w).Encode(o) // REAL row → REAL JSON
		})

	go http.ListenAndServe(":8090", mux) // the service
	time.Sleep(100 * time.Millisecond)   // let it bind the port

	// another service asks about order 1042 — a REAL HTTP round trip
	res, err := http.Get("http://localhost:8090/orders/1042")
	if err != nil {
		panic(err)
	}
	defer res.Body.Close()

	var o Order
	json.NewDecoder(res.Body).Decode(&o)
	fmt.Println("status code:", res.StatusCode)
	fmt.Printf("order: %+v\n", o)
}
📖 Line by line — what & why
mux.HandleFunc("GET /orders/{id}", func(w, r) {…})

What: registers a real handler for GET requests matching /orders/<something>. Why: since Go 1.22 the router understands methods and path variables — a POST to this path is auto-rejected with 405.

id := r.PathValue("id")

What: reads whatever matched {id} from the real incoming request. Why: no regex, no string splitting — `r` carries everything about the request.

db.QueryRow("SELECT id, status FROM http_orders WHERE id=?", id).Scan(&o.ID, &o.Status)

What: a REAL query, run fresh for every request that hits this handler. Why: the response reflects the database at the moment of the request — not a value baked in at startup.

if err == sql.ErrNoRows { http.Error(w, "not found", 404) }

What: sql.ErrNoRows is what a real "no matching row" query returns. Why: this is the one-line translation from "database said nothing matched" to an honest 404.

go http.ListenAndServe(":8090", mux)

What: starts the real server in a goroutine. Why: ListenAndServe blocks forever — run it in the foreground here and the client code below would never execute.

res, err := http.Get("http://localhost:8090/orders/1042")

What: a REAL HTTP request over a REAL socket to the server above. Why: err here means "couldn't even connect". A 404 or 500 is NOT an err — check res.StatusCode for that.

json.NewDecoder(res.Body).Decode(&o)

What: the mirror of the server's Encode: real response bytes → struct. Why: the two services agree on shape through the Order struct — change it in one place, both sides follow.

terminal
$ go run ./10-httpapi
status code: 200
order: {ID:1042 Status:shipped}

Notice: the handler ran an actual SELECT ... WHERE id=? against MySQL for every request — this is a full real request/response loop over a real socket, backed by a real row, not an in-memory struct standing in for a database.

11 MySQL: the signup flow database/sql

What POST /signup actually does to the database: connect, insert the email, read it back.

💡 Everyday version: sql.Open is saving a phone number — no call made. db.Ping is dialling to check someone answers. The ? placeholder is a printed form with a fixed answer box: whatever the user types stays INSIDE the box and can never rewrite the form — that’s why injection can’t happen.

// mysql: what POST /signup does to the database.
// needs: go get github.com/go-sql-driver/mysql
// and a server:
//   docker run -d --name mysql-demo \
//     -e MYSQL_ROOT_PASSWORD=secret \
//     -e MYSQL_DATABASE=notes -p 3306:3306 mysql:8
package main

import (
	"database/sql"
	"fmt"

	_ "github.com/go-sql-driver/mysql"
)

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()

	if err := db.Ping(); err != nil {
		panic("db unreachable: " + err.Error())
	}
	fmt.Println("connected ✓")

	db.Exec("DROP TABLE IF EXISTS users") // rerunnable demo
	db.Exec(`CREATE TABLE users (
		id    INT AUTO_INCREMENT PRIMARY KEY,
		email VARCHAR(120) NOT NULL UNIQUE)`)

	// the signup form's email goes in via ? — never
	// by pasting it into the SQL string
	res, err := db.Exec(
		"INSERT INTO users (email) VALUES (?)",
		"asha@example.com")
	if err != nil {
		panic(err) // e.g. duplicate email → 409
	}
	id, _ := res.LastInsertId()
	fmt.Println("new user id:", id)

	var email string
	db.QueryRow(
		"SELECT email FROM users WHERE id = ?",
		id).Scan(&email)
	fmt.Println("stored:", email)
}
📖 Line by line — what & why
db, err := sql.Open("mysql", "root:secret@tcp(localhost:3306)/notes")

What: parses the connection string and prepares a connection POOL. No network yet. Why: "mysql" names the driver registered by the blank import. A wrong password does NOT fail here — Open is lazy.

defer db.Close()

What: returns every pooled connection when main exits. Why: in a server you'd open once at startup and close at shutdown — *sql.DB is shared and thread-safe, never per-request.

db.Ping()

What: the FIRST real network round-trip: connect, authenticate. Why: this is where a bad password or unreachable host actually explodes. Ping at boot so it fails loudly at deploy, not on your first user.

db.Exec(`CREATE TABLE users ( … email … UNIQUE)`)

What: runs DDL. The UNIQUE constraint makes duplicate emails impossible at the DATABASE level. Why: a rule enforced by the DB can't be bypassed by a buggy handler, a second service, or someone in a SQL console.

db.Exec("INSERT INTO users (email) VALUES (?)", "asha@example.com")

What: the statement and the value travel SEPARATELY; the driver combines them safely. Why: this is the SQL-injection defence. Never fmt.Sprintf a query — a malicious string in a form field stays a harmless value inside the ? box.

if err != nil { panic(err) } // e.g. duplicate email → 409

What: the INSERT can fail — most interestingly when UNIQUE rejects a duplicate. Why: in a real handler this err becomes `409 Conflict: email already registered`. The constraint does the checking; you just translate the error.

id, _ := res.LastInsertId()

What: fetches the AUTO_INCREMENT id MySQL just assigned. Why: you need it for the response body and for follow-up queries.

db.QueryRow("SELECT … WHERE id = ?", id).Scan(&email)

What: runs a one-row query and copies the column into your variable (via a pointer). Why: QueryRow for exactly-one-row; Scan fills &email. If no row matched, Scan returns sql.ErrNoRows — that's your 404.

terminal
$ go run ./11-mysql
connected ✓
new user id: 1
stored: asha@example.com

Notice: UNIQUE on email means a duplicate signup makes the INSERT error — your handler turns that into a 409, the database enforcing a business rule for free. (DROP TABLE just keeps the demo rerunnable.)

12 graceful shutdown production

Deploy day for the orders API — SIGTERM arrives, it stops taking orders and finishes the open ones.

💡 Everyday version: closing a restaurant properly: lock the front door (stop accepting), let seated guests finish (drain in-flight requests), then lights out. Kubernetes is the landlord — it rings the bell (SIGTERM) and cuts power ~30s later. Skip this and every deploy hangs up on users mid-request.

// graceful shutdown: deploy day for your orders API.
// (sends ITSELF a SIGTERM so you can watch the drain)
package main

import (
	"context"
	"fmt"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	srv := &http.Server{Addr: ":8091"}

	ctx, stop := signal.NotifyContext(
		context.Background(),
		os.Interrupt, syscall.SIGTERM)
	defer stop()

	go srv.ListenAndServe()
	fmt.Println("orders API serving on :8091")

	go func() { // pretend k8s deploys in 300ms
		time.Sleep(300 * time.Millisecond)
		syscall.Kill(os.Getpid(), syscall.SIGTERM)
	}()

	<-ctx.Done() // block until the signal
	fmt.Println("SIGTERM: stop taking orders, finish the open ones…")

	shCtx, cancel := context.WithTimeout(
		context.Background(), 5*time.Second)
	defer cancel()
	if err := srv.Shutdown(shCtx); err != nil {
		panic(err)
	}
	fmt.Println("all in-flight requests served — clean exit ✓")
}
📖 Line by line — what & why
srv := &http.Server{Addr: ":8091"}

What: builds the server as a value we keep a reference to. Why: the usual one-liner http.ListenAndServe gives you no handle — and you need the handle to call .Shutdown() on it later.

ctx, stop := signal.NotifyContext(…, os.Interrupt, syscall.SIGTERM)

What: a context that cancels itself when Ctrl-C (Interrupt) or SIGTERM arrives. Why: SIGTERM is what Kubernetes/Docker send before killing a container. This one line converts "OS signal" into the same ctx.Done() mechanism as program 07.

go srv.ListenAndServe()

What: serve traffic in a goroutine. Why: it blocks forever, and main has other work: standing guard for the signal.

go func() { time.Sleep(300ms); syscall.Kill(os.Getpid(), syscall.SIGTERM) }()

What: the demo trick: after 300ms, the program sends ITSELF a SIGTERM. Why: simulates deploy day so you can watch the sequence. Delete this goroutine and the program waits for your real Ctrl-C.

<-ctx.Done()

What: main parks here — receiving from a channel blocks — until the signal lands. Why: the cheapest possible "wait": no loop, no polling, zero CPU.

shCtx, cancel := context.WithTimeout(…, 5*time.Second)

What: a deadline for the drain itself. Why: k8s force-kills ~30s after SIGTERM. Your drain must finish before that — 5s says "finish open requests, but don't wait forever on a stuck one".

srv.Shutdown(shCtx)

What: stops accepting NEW connections, waits for in-flight requests to complete, then returns. Why: this is the graceful part. The alternative — just exiting — resets every open connection: users see errors on every deploy.

terminal
$ go run ./12-shutdown
orders API serving on :8091
SIGTERM: stop taking orders, finish the open ones…
all in-flight requests served — clean exit ✓

Notice: the pieces are all from this lab — a goroutine (01), a signal context (07), blocking on ctx.Done(). Delete the fake-SIGTERM goroutine and it waits for your real Ctrl-C.
