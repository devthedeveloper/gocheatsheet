01 goroutines start here

One keyword makes it concurrent. The catch: when main returns, everyone dies.

💡 Everyday version: your API gets 50 requests at once and handles them all simultaneously — that's goroutines, and the HTTP server spawns them for you. go say("A") is you hiring a helper: they start working while you carry on. But main is the shop owner — when the owner locks up and goes home, every helper stops mid-task.

package main

import (
	"fmt"
	"time"
)

func say(name string) {
	for i := 1; i <= 3; i++ {
		fmt.Println(name, i)
		time.Sleep(10 * time.Millisecond)
	}
}

func main() {
	go say("A") // fires and returns instantly
	go say("B")

	say("main") // runs in the main goroutine,
	//             keeping the program alive

	// when main returns, ALL goroutines die.
	// delete the say("main") line and watch
	// A and B never get a chance to print.
}
terminal
$ go run ./01-goroutines
main 1
A 1
B 1
B 2
A 2
main 2
main 3
A 3
B 3

Notice: the interleaving is different every run — the scheduler decides. Never depend on goroutine ordering.

02 wg.Wait() sync

The fix for "main dies too early": count work in, count work out.

💡 Everyday version: your endpoint must call the user service, the orders service and the notifications service, then respond once ALL three answer. A WaitGroup is a tally on a whiteboard: wg.Add(1) = "one more job is out there", wg.Done() = a worker crossing their job off, wg.Wait() = you standing at the board, refusing to move until the tally reads zero.

  1. wg.Add(1) before go — write it on the board before the worker leaves, or you might check the board while it still says 0 and walk off early.
  2. defer wg.Done() as the worker's first line — the job gets crossed off no matter how the worker exits, even on an error.
  3. wg.Wait() blocks until the count is back to 0 — then, and only then, main continues.

    package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var wg sync.WaitGroup

	for id := 1; id <= 3; id++ {
		wg.Add(1) // +1 BEFORE starting
		go func() {
			defer wg.Done() // -1, always
			wait := time.Duration(id) *
				100 * time.Millisecond
			time.Sleep(wait) // fake work
			fmt.Printf("worker %d done (%v)\n",
				id, wait)
		}()
	}

	fmt.Println("waiting for 3 workers…")
	wg.Wait() // blocks until counter hits 0
	fmt.Println("all done ✓")
}
    terminal
    $ go run ./02-waitgroup
waiting for 3 workers…
worker 1 done (100ms)
worker 2 done (200ms)
worker 3 done (300ms)
all done ✓

    Notice: total time is 300ms (the slowest), not 600ms (the sum) — the workers ran simultaneously.

03 mutex vs the race the big one

100 goroutines bump a counter 1,000 times each. Watch two-thirds of the increments vanish.

💡 Everyday version: two requests hit "add ₹100 to wallet" at the same moment. Both read balance 500, both compute 600, both write 600 — one deposit just vanished. That's the race. A mutex is the single bathroom key at a petrol station: Lock() takes the key, everyone else queues, Unlock() hands it back. Only one goroutine touches the money at a time.

package main

import (
	"fmt"
	"sync"
)

func main() {
	const workers, bumps = 100, 1000
	var wg sync.WaitGroup

	// BROKEN: a plain int, no lock
	unsafe := 0
	for range workers {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range bumps {
				unsafe++ // RACE: read+write
			}
		}()
	}
	wg.Wait()

	// FIXED: same thing behind a mutex
	var mu sync.Mutex
	safe := 0
	for range workers {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range bumps {
				mu.Lock()
				safe++
				mu.Unlock()
			}
		}()
	}
	wg.Wait()

	fmt.Println("expected:      ", workers*bumps)
	fmt.Println("without mutex: ", unsafe, "😱")
	fmt.Println("with mutex:    ", safe, "✓")
}
terminal
$ go run ./03-mutex
expected:       100000
without mutex:  33861 😱
with mutex:     100000 ✓
# x++ is really: read x, add 1, write x —
# two goroutines interleave those steps
# and one of the writes gets overwritten

Extra credit: go run -race ./03-mutex — Go pinpoints the racing line for you. Run -race in CI, always.

04 channels pipes

A typed pipe between goroutines. Sends block until someone receives — watch the lockstep.

💡 Everyday version: one teammate unzips uploaded files, another processes them. The channel is the hatch between their desks. Unbuffered means hand-to-hand: the sender stands there holding the file until the receiver takes it — that's why the output alternates send, receive, send, receive.

package main

import "fmt"

func main() {
	ch := make(chan string)

	go func() { // the sender
		menu := []string{"🍎", "🍌", "🍇"}
		for _, fruit := range menu {
			fmt.Println("sending", fruit)
			ch <- fruit // blocks until received
		}
		close(ch) // sender says: no more
	}()

	for f := range ch { // receive until closed
		fmt.Println("got    ", f)
	}
	fmt.Println("channel drained, bye")
}
terminal
$ go run ./04-channels
sending 🍎
sending 🍌
got     🍎
sending 🍇
got     🍌
got     🍇
channel drained, bye

Notice: sends and receives alternate in lockstep — that's the unbuffered handshake. Try make(chan string, 3) and watch all three sends happen first.

05 select + timeout wait smart

Wait on a fast worker, a slow worker, and a deadline — whichever speaks first wins.

💡 Everyday version: you ordered from two food apps and you'll eat whichever arrives first — and if nothing shows up in 30 minutes, you're making noodles. select watches all cases at once and takes the first one ready; time.After is the noodle deadline. Backend translation: try the cache and the DB together, take the fastest, never wait forever.

package main

import (
	"fmt"
	"time"
)

func worker(name string, d time.Duration) <-chan string {
	ch := make(chan string, 1)
	go func() {
		time.Sleep(d)
		ch <- name + " finished"
	}()
	return ch
}

func main() {
	fast := worker("fast", 50*time.Millisecond)
	slow := worker("slow", 5*time.Second)

	for {
		select {
		case msg := <-fast:
			fmt.Println(msg)
		case msg := <-slow:
			fmt.Println(msg) // never happens
		case <-time.After(200 * time.Millisecond):
			fmt.Println("⏰ timeout — not waiting 5s for slow")
			return
		}
	}
}
terminal
$ go run ./05-select
fast finished
⏰ timeout — not waiting 5s for slow

Notice: the program exits in ~250ms even though a 5-second worker is still out there. This is how you put deadlines on anything.

06 worker pool throughput

Nine jobs, three workers, one queue. The math shows up in the wall clock.

💡 Everyday version: a café with one order queue and three baristas — whoever's free grabs the next order. Backend translation: 10,000 images to thumbnail, but you only want 3 hitting the disk at once. The queue feeds a fixed crew instead of hiring 10,000 workers and melting the machine.

package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	start := time.Now()
	jobs := make(chan int)
	results := make(chan string)
	var wg sync.WaitGroup

	for w := 1; w <= 3; w++ { // exactly 3 workers
		wg.Add(1)
		go func() {
			defer wg.Done()
			for j := range jobs { // until closed
				time.Sleep(100 * time.Millisecond)
				results <- fmt.Sprintf(
					"worker %d finished job %d", w, j)
			}
		}()
	}

	go func() { // feed 9 jobs, then close
		for j := 1; j <= 9; j++ {
			jobs <- j
		}
		close(jobs)
	}()

	go func() { // close results when workers exit
		wg.Wait()
		close(results)
	}()

	for r := range results {
		fmt.Println(r)
	}
	fmt.Printf("9 jobs × 100ms on 3 workers = %v\n",
		time.Since(start).Round(10*time.Millisecond))
}
terminal
$ go run ./06-workerpool
worker 1 finished job 3
worker 2 finished job 2
worker 3 finished job 1
worker 3 finished job 6
worker 2 finished job 5
worker 1 finished job 4
worker 2 finished job 9
worker 1 finished job 8
worker 3 finished job 7
9 jobs × 100ms on 3 workers = 300ms

Notice: 900ms of work done in 300ms. Change it to 1 worker or 9 workers and re-check the clock.

07 context please stop

A worker that ticks forever — until the context's 350ms deadline pulls the plug.

💡 Everyday version: a customer walks out of the restaurant — the kitchen should stop cooking their order, not finish it for nobody. The context is the order ticket every function holds a copy of; when it's cancelled (client disconnected, or the timer rang), ctx.Done() fires and everyone holding that ticket stops. This is how a request timeout flows through your whole backend.

package main

import (
	"context"
	"fmt"
	"time"
)

func tick(ctx context.Context) {
	for i := 1; ; i++ {
		select {
		case <-ctx.Done(): // cancelled or timed out
			fmt.Println("worker: stopping —", ctx.Err())
			return
		case <-time.After(100 * time.Millisecond):
			fmt.Println("worker: tick", i)
		}
	}
}

func main() {
	ctx, cancel := context.WithTimeout(
		context.Background(), 350*time.Millisecond)
	defer cancel() // always, even on success

	tick(ctx) // returns when the context fires
	fmt.Println("main: clean exit")
}
terminal
$ go run ./07-context
worker: tick 1
worker: tick 2
worker: tick 3
worker: stopping — context deadline exceeded
main: clean exit

Notice: 3 ticks fit inside 350ms, then ctx.Done() fires. Swap WithTimeout for WithCancel and call cancel() yourself.

08 errgroup x/sync

Fan out three calls; one fails. The group hands you exactly the first error.

💡 Everyday version: a team lead hands three devs a task each and waits at standup. If anyone hit a blocker, the lead hears the FIRST blocker and can call the whole thing off. It's the WaitGroup whiteboard from program 02, but the workers can also report failures — which is why it's the default for calling several services at once.

// needs: go get golang.org/x/sync/errgroup
package main

import (
	"errors"
	"fmt"

	"golang.org/x/sync/errgroup"
)

func fetch(url string) error {
	if url == "https://bad.example" {
		return errors.New("boom: " + url)
	}
	fmt.Println("fetched", url)
	return nil
}

func main() {
	var g errgroup.Group
	g.SetLimit(2) // at most 2 at a time

	urls := []string{
		"https://go.dev",
		"https://bad.example",
		"https://github.com",
	}
	for _, u := range urls {
		g.Go(func() error {
			return fetch(u)
		})
	}

	// Wait returns the FIRST error (or nil)
	if err := g.Wait(); err != nil {
		fmt.Println("g.Wait() →", err)
	}
}
terminal
$ go run ./08-errgroup
fetched https://github.com
fetched https://go.dev
g.Wait() → boom: https://bad.example

Notice: the good fetches still ran; Wait surfaced the one failure. This is programs 02+03 with error handling built in — your default for real fan-out.

09 JSON both ways encoding

Struct to JSON and back, with tags doing the naming.

💡 Everyday version: your struct is the organized shelf in your warehouse; JSON is the cardboard box you ship. Marshal = packing the box with a labelled manifest (tags decide the labels); Unmarshal = unpacking a box someone sent you onto your shelf. Every request body you'll ever read is this one round trip.

package main

import (
	"encoding/json"
	"fmt"
)

type Note struct {
	ID   int      `json:"id"`
	Text string   `json:"text"`
	Tags []string `json:"tags,omitempty"`
}

func main() {
	// struct → JSON (marshal)
	n := Note{ID: 1, Text: "hello",
		Tags: []string{"go", "demo"}}
	out, _ := json.MarshalIndent(n, "", "  ")
	fmt.Println(string(out))

	// JSON → struct (unmarshal)
	raw := `{"id": 2, "text": "from the wire"}`
	var m Note
	if err := json.Unmarshal([]byte(raw), &m); err != nil {
		panic(err)
	}
	fmt.Printf("%+v\n", m)
}
terminal
$ go run ./09-json
{
  "id": 1,
  "text": "hello",
  "tags": [
    "go",
    "demo"
  ]
}
{ID:2 Text:from the wire Tags:[]}

Notice: the input had no tags, so Tags came back as a nil slice — and omitempty would hide it on the way out.

10 HTTP server + client one file

A JSON API and the code that calls it — the server runs in a goroutine (lesson 01 pays off).

💡 Everyday version: both halves of your day job in 40 lines — the service that answers, and the client that asks (also you, whenever your service calls another service). The counter n is in-memory state: it lives exactly as long as the process, which is why a restart resets anything you didn't put in a database.

package main

import (
	"encoding/json"
	"fmt"
	"net/http"
	"time"
)

type Pong struct {
	Msg string `json:"msg"`
	N   int    `json:"n"`
}

func main() {
	mux := http.NewServeMux()
	n := 0
	mux.HandleFunc("GET /ping",
		func(w http.ResponseWriter, r *http.Request) {
			n++
			json.NewEncoder(w).Encode(Pong{"pong", n})
		})

	go http.ListenAndServe(":8090", mux) // serve…
	time.Sleep(100 * time.Millisecond)   // …boot up

	for range 3 { // …and call ourselves
		res, err := http.Get(
			"http://localhost:8090/ping")
		if err != nil {
			panic(err)
		}
		var p Pong
		json.NewDecoder(res.Body).Decode(&p)
		res.Body.Close()
		fmt.Printf("%d ← %+v\n", res.StatusCode, p)
	}
}
terminal
$ go run ./10-httpapi
200 ← {Msg:pong N:1}
200 ← {Msg:pong N:2}
200 ← {Msg:pong N:3}

Notice: the counter increments across requests — server state lives as long as the process. (Spot the race if two clients hit it at once? Lesson 03 says n++ needs a mutex.)

11 MySQL connect + CRUD database/sql

Open, ping, create, insert, read back — the whole database handshake in 40 lines.

💡 Everyday version: sql.Open is saving a phone number to your contacts — no call made yet. db.Ping is actually dialling to check someone picks up. And the ? placeholder is a printed form with a fixed answer box: whatever the user writes stays INSIDE the box and can never rewrite the form — that's why injection can't happen.

// needs: go get github.com/go-sql-driver/mysql
// and a server:
//   docker run -d --name mysql-demo \
//     -e MYSQL_ROOT_PASSWORD=secret \
//     -e MYSQL_DATABASE=notes \
//     -p 3306:3306 mysql:8
package main

import (
	"database/sql"
	"fmt"

	_ "github.com/go-sql-driver/mysql"
)

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()

	if err := db.Ping(); err != nil { // real check
		panic("unreachable: " + err.Error())
	}
	fmt.Println("connected ✓")

	db.Exec(`CREATE TABLE IF NOT EXISTS pets (
		id   INT AUTO_INCREMENT PRIMARY KEY,
		name VARCHAR(50) NOT NULL)`)

	res, err := db.Exec(
		"INSERT INTO pets (name) VALUES (?)",
		"Gopher")
	if err != nil {
		panic(err)
	}
	id, _ := res.LastInsertId()
	fmt.Println("inserted pet with id", id)

	var name string
	db.QueryRow("SELECT name FROM pets WHERE id = ?",
		id).Scan(&name)
	fmt.Println("read back:", name)
}
terminal
$ go run ./11-mysql
connected ✓
inserted pet with id 1
read back: Gopher

Notice: sql.Open doesn't connect — it just validates the string. db.Ping() is the moment credentials actually get tested. The ? placeholders are your SQL-injection armor.

12 graceful shutdown production

The deploy-day pattern — this program sends itself a SIGTERM so you can watch the drain.

💡 Everyday version: closing a restaurant properly — lock the front door (stop accepting requests), let seated guests finish eating (drain in-flight work), then lights out. Kubernetes is the landlord: it rings the closing bell (SIGTERM) and cuts the power ~30s later. Skip this pattern and every deploy kicks users out mid-bite.

package main

import (
	"context"
	"fmt"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	srv := &http.Server{Addr: ":8091"}

	ctx, stop := signal.NotifyContext(
		context.Background(),
		os.Interrupt, syscall.SIGTERM)
	defer stop()

	go srv.ListenAndServe()
	fmt.Println("serving on :8091")

	go func() { // pretend k8s kills us in 300ms
		time.Sleep(300 * time.Millisecond)
		syscall.Kill(os.Getpid(), syscall.SIGTERM)
	}()

	<-ctx.Done() // block until the signal
	fmt.Println("SIGTERM received, draining…")

	shCtx, cancel := context.WithTimeout(
		context.Background(), 5*time.Second)
	defer cancel()
	if err := srv.Shutdown(shCtx); err != nil {
		panic(err)
	}
	fmt.Println("in-flight requests done, clean exit ✓")
}
terminal
$ go run ./12-shutdown
serving on :8091
SIGTERM received, draining…
in-flight requests done, clean exit ✓

Notice: everything from this lab is in here — a goroutine (01), a signal context (07), and a select-like block on ctx.Done(). Delete the fake-SIGTERM goroutine and it waits for a real Ctrl-C.
