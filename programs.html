01 goroutines start here

Three users signed up — three welcome emails go out at the same time.

💡 Everyday version: go sendEmail(…) is hiring a helper: they start dialling SMTP immediately while you carry on. But main is the shop owner — when the owner locks up, every helper goes home mid-call. (Your HTTP server does this hiring for you on every request.)

// goroutines: three signup emails, sent at the same time.
package main

import (
	"fmt"
	"time"
)

func sendEmail(to string) {
	fmt.Println("connecting to SMTP for", to)
	time.Sleep(150 * time.Millisecond) // SMTP round-trip
	fmt.Println("✉️  delivered to", to)
}

func main() {
	// three users just signed up
	go sendEmail("asha@example.com")
	go sendEmail("ravi@example.com")

	sendEmail("meera@example.com") // main sends one too —
	// which keeps the program alive long enough

	// when main returns, ALL goroutines die.
	// make main's send a `go` too and the program
	// exits instantly: zero emails delivered.
}
📖 Line by line — what & why
func sendEmail(to string) {

What: an ordinary function that takes a recipient. Why: goroutines run plain functions — nothing special is needed to make a function “concurrency-ready”.

fmt.Println("connecting to SMTP for", to)

What: logs BEFORE the slow part starts. Why: so the output can prove all three connections opened together, before any email finished.

time.Sleep(150 * time.Millisecond)

What: pauses this goroutine for 150ms, standing in for the SMTP network round-trip. Why: a real email send takes roughly this long; Sleep keeps the demo runnable without a mail server. Only THIS goroutine sleeps — the others keep going.

go sendEmail("asha@example.com")

What: starts sendEmail in a NEW goroutine and moves to the next line immediately — the email is not sent yet. Why: so signup #2 never waits behind signup #1's email. go costs a few KB; firing thousands is normal.

sendEmail("meera@example.com")

What: main sends the third email itself — no go — so main is busy for ~150ms. Why: this is what keeps the program alive. When main returns, ALL goroutines are killed instantly; main being busy for 150ms gives the other two time to finish. Delete this call's work and zero emails go out.

terminal
$ go run ./01-goroutines
connecting to SMTP for meera@example.com
connecting to SMTP for ravi@example.com
connecting to SMTP for asha@example.com
✉️  delivered to asha@example.com
✉️  delivered to ravi@example.com
✉️  delivered to meera@example.com

Notice: all three SMTP connections opened together and finished ~150ms later — sequentially this would take 450ms. Make main’s send a go too and the program exits with zero emails delivered.

02 wg.Wait() sync

GET /dashboard fans out to three services — and must respond only when ALL of them have answered.

💡 Everyday version: a WaitGroup is a tally on a whiteboard: wg.Add(1) = “one more call in flight”, wg.Done() = a call crossing itself off, wg.Wait() = you standing at the board, refusing to send the 200 until the tally reads zero.

  1. wg.Add(1) before go — write it on the board before the call leaves, or you might check the board while it still says 0 and respond with half a dashboard.
  2. defer wg.Done() first line inside — the call crosses itself off however it ends, even on an error.
  3. wg.Wait() blocks until the count is back to 0 — then, and only then, the response goes out.
// wg.Wait: GET /dashboard answers only after ALL services do.
package main

import (
	"fmt"
	"sync"
	"time"
)

func call(service string, ms int) {
	time.Sleep(time.Duration(ms) * time.Millisecond)
	fmt.Printf("%s answered in %dms\n", service, ms)
}

func main() {
	var wg sync.WaitGroup

	// the dashboard needs three services
	services := map[string]int{
		"profile-service":       100,
		"orders-service":        250,
		"notifications-service": 180,
	}
	fmt.Println("GET /dashboard → calling 3 services…")
	for name, latency := range services {
		wg.Add(1) // one more call in flight
		go func() {
			defer wg.Done() // crossed off, always
			call(name, latency)
		}()
	}

	wg.Wait() // block here until all three answer
	fmt.Println("200 OK — response assembled ✓")
}
📖 Line by line — what & why
var wg sync.WaitGroup

What: creates the counter. Its zero value is ready to use — it starts at 0. Why: this counter will track “how many service calls are still in flight”. No initialization ceremony needed.

services := map[string]int{ "profile-service": 100, … }

What: the three services the dashboard needs, with their (simulated) response times. Why: stand-ins for real HTTP clients. The latencies differ on purpose — so the output can prove Wait waits for the slowest.

wg.Add(1)

What: counter goes 0→1→2→3: “one more call is in flight”. Why: it MUST happen before go, in the parent. If the goroutine did its own Add, main could reach Wait while the counter still reads 0 — and respond with half a dashboard.

go func() {

What: launches the service call in its own goroutine. Why: this is the fan-out: all three calls now overlap instead of queueing. 100+250+180ms of waiting becomes 250ms.

defer wg.Done()

What: schedules counter−1 for the moment this goroutine exits — any exit. Why: defer fires even if the call errors or panics. Without it, one failed call = a counter stuck above 0 = wg.Wait() hangs forever = your handler never responds.

call(name, latency)

What: the actual work. Why: in production this line is http.Get(...) or a gRPC call. Everything around it stays identical.

wg.Wait()

What: main STOPS at this line. It sleeps (zero CPU) until the counter is back to 0. Why: this is the “don't respond yet”. Each Done() ticks the counter down; the last one (orders-service at 250ms) releases main.

fmt.Println("200 OK — response assembled ✓")

What: runs only after every service has reported. Why: past Wait, it is GUARANTEED all three answered — safe to assemble the response from their data.

terminal
$ go run ./02-waitgroup
GET /dashboard → calling 3 services…
profile-service answered in 100ms
notifications-service answered in 180ms
orders-service answered in 250ms
200 OK — response assembled ✓

Notice: the 200 OK left at 250ms — the slowest service, not the 530ms sum. That difference is the whole reason to fan out.

03 mutex vs the wallet race the big one

A cashback promo: 100 concurrent clients, 1,000 hits each, ₹1 per hit. Watch ₹44,546 vanish.

💡 Everyday version: two requests read balance ₹500 at the same moment, both compute ₹501, both write ₹501 — one rupee gone. At scale that’s the 😱 line below. The mutex is the single bathroom key at a petrol station: Lock() takes the key, everyone queues, Unlock() hands it back. One goroutine touches the balance at a time.

// mutex: 100 concurrent clients earn ₹1 cashback per hit.
package main

import (
	"fmt"
	"sync"
)

func main() {
	const clients, hits = 100, 1000
	var wg sync.WaitGroup

	// BROKEN: every request credits the wallet, no lock
	balance := 0
	for range clients {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range hits {
				balance++ // RACE: read, add, write
			}
		}()
	}
	wg.Wait()

	// FIXED: same traffic, one mutex
	var mu sync.Mutex
	safe := 0
	for range clients {
		wg.Add(1)
		go func() {
			defer wg.Done()
			for range hits {
				mu.Lock()
				safe++
				mu.Unlock()
			}
		}()
	}
	wg.Wait()

	fmt.Println("should be: ₹", clients*hits)
	fmt.Println("no mutex:  ₹", balance, "— money vanished 😱")
	fmt.Println("mutex:     ₹", safe, "✓")
}
📖 Line by line — what & why
const clients, hits = 100, 1000

What: 100 concurrent clients, each earning cashback 1,000 times. Why: enough volume that the race loses money on every run. Small numbers can get lucky and hide the bug — that's what makes races evil.

balance := 0

What: one plain int, shared by all 100 goroutines. Why: sharing is the point — and sharing without protection is the bug we're about to watch.

balance++

What: looks like one step; is actually three: READ balance, ADD 1, WRITE it back. Why: two clients both read ₹500, both compute ₹501, both write ₹501 — two credits happened, balance moved by one. Repeat thousands of times: ₹44,546 gone.

var mu sync.Mutex

What: the lock. Zero value ready, like WaitGroup. Why: think: ONE bathroom key for the whole petrol station. It protects whatever you consistently lock around — here, the balance.

mu.Lock()

What: take the key — or stand in line until whoever has it gives it back. Why: from this line to Unlock, no other goroutine can be in this section. read-add-write becomes one uninterruptible turn.

safe++

What: the same three steps as before — Why: — but now nobody can interleave with them. Every credit lands.

mu.Unlock()

What: hand the key back immediately. Why: keep the locked section as short as possible: everything inside it runs one-goroutine-at-a-time, so a fat locked section becomes your bottleneck.

wg.Wait() (before printing)

What: wait until every client is done before reading the totals. Why: reading balance while writers are still running would itself be a race — you'd print a number mid-flight.

terminal
$ go run ./03-mutex
should be: ₹ 100000
no mutex:  ₹ 55454 — money vanished 😱
mutex:     ₹ 100000 ✓
# balance++ is really: read, add 1, write —
# two clients interleave those steps and one
# write overwrites the other. Real money, gone.

Extra credit: go run -race ./03-mutex — Go names the exact racing line. This bug passes code review constantly; -race in CI is what actually catches it.

04 channels pipes

An upload endpoint hands finished files to a virus scanner — hand to hand.

💡 Everyday version: the channel is the hatch between two desks. Unbuffered means hand-to-hand: the upload goroutine stands there holding invoice_jan.pdf until the scanner takes it. That’s why the output alternates instead of uploading everything first.

// channels: uploaded files handed to the virus scanner.
package main

import "fmt"

func main() {
	uploads := make(chan string) // the hand-off point

	go func() { // goroutine A: the upload receiver
		files := []string{
			"invoice_jan.pdf",
			"invoice_feb.pdf",
			"salary_slip.pdf",
		}
		for _, f := range files {
			fmt.Println("upload complete:", f)
			uploads <- f // hand to the scanner
		}
		close(uploads) // no more uploads today
	}()

	for f := range uploads { // main: the scanner
		fmt.Println("  virus-scanned:", f)
	}
	fmt.Println("all uploads processed ✓")
}
📖 Line by line — what & why
uploads := make(chan string)

What: creates a typed pipe. Capacity 0 — it holds nothing. Why: capacity 0 means every send is a live hand-off: the sender physically waits for a receiver. Nothing queues up invisibly.

go func() { … }()

What: the upload receiver runs in its own goroutine. Why: required with an unbuffered channel: sender and receiver must BOTH be running to meet. Doing both in one goroutine would deadlock on the first send.

uploads <- f

What: sends the filename into the channel — and BLOCKS until the scanner takes it. Why: this blocking is free backpressure: if the scanner is slow, uploads slow down automatically. Memory can't fill up with unscanned files.

close(uploads)

What: marks the channel “no more values coming”. Why: always the SENDER closes — only it knows when it's done. Closing is what lets the receiving loop end instead of waiting forever.

for f := range uploads {

What: receives values one by one until the channel is closed and drained. Why: the idiomatic consumer: no counting, no done-flags. Sender closes, loop ends, program moves on.

terminal
$ go run ./04-channels
upload complete: invoice_jan.pdf
upload complete: invoice_feb.pdf
  virus-scanned: invoice_jan.pdf
  virus-scanned: invoice_feb.pdf
upload complete: salary_slip.pdf
  virus-scanned: salary_slip.pdf
all uploads processed ✓

Notice: the strict upload → scan alternation. Make it make(chan string, 3) and all three uploads complete before the first scan — the hatch became a tray.

05 select + timeout wait smart

cart:42 lives in the cache AND the database. Serve whichever answers first — inside a 100ms budget.

💡 Everyday version: you ordered from two food apps; you’ll eat whichever arrives first, and if nothing shows in 30 minutes you’re making noodles. select watches all three cases at once; time.After is the noodles. The user gets an answer in 100ms no matter what the database is doing.

// select: cache vs DB — serve the fastest, respect the budget.
package main

import (
	"fmt"
	"time"
)

func cacheGet(key string) <-chan string {
	ch := make(chan string, 1)
	go func() {
		time.Sleep(5 * time.Millisecond) // redis is quick
		ch <- key + " → 3 items (cache)"
	}()
	return ch
}

func dbQuery(key string) <-chan string {
	ch := make(chan string, 1)
	go func() {
		time.Sleep(3 * time.Second) // db is having a day
		ch <- key + " → 3 items (db)"
	}()
	return ch
}

func main() {
	cache := cacheGet("cart:42")
	db := dbQuery("cart:42")

	select {
	case v := <-cache:
		fmt.Println("hit:", v)
	case v := <-db:
		fmt.Println("hit:", v)
	case <-time.After(100 * time.Millisecond):
		fmt.Println("both slow → 504, don't hang")
	}
	fmt.Println("responded within the 100ms budget ✓")
}
📖 Line by line — what & why
func cacheGet(key string) <-chan string {

What: returns a receive-only channel; the answer will appear on it later. Why: the “return a channel immediately, deliver later” shape is what makes ANY slow thing usable inside select.

ch := make(chan string, 1)

What: buffer of exactly 1. Why: subtle but critical: the LOSER (the 3s DB query) can still drop its answer in the buffer and exit cleanly. With capacity 0 it would block forever on a send nobody receives — a goroutine leak.

time.Sleep(5 * time.Millisecond) / (3 * time.Second)

What: the simulated latencies: redis quick, DB struggling today. Why: swap these for a real redis GET and SQL query — the select below doesn't change at all.

select {

What: waits on ALL cases at once; runs whichever becomes ready FIRST; ignores the rest. Why: one keyword replaces polling loops and “who finished?” bookkeeping.

case v := <-cache: / case v := <-db:

What: each case tries to receive from one source. Why: identical treatment — select doesn't prefer any case; readiness decides. (Two ready at once → random pick.)

case <-time.After(100 * time.Millisecond):

What: time.After returns a channel that fires once, 100ms from now. Why: this converts “wait for an answer” into “wait AT MOST 100ms”. The user gets a fast 504 instead of a hanging spinner.

terminal
$ go run ./05-select
hit: cart:42 → 3 items (cache)
responded within the 100ms budget ✓

Notice: the DB was still ~3 seconds away when we responded. Change the cache latency to 200ms and re-run — you’ll see the 504 branch fire instead of a hang.

06 worker pool throughput

A newsletter to 9 users — but you only hold 3 SMTP connections.

💡 Everyday version: the 3 connections are baristas; the address list is the order queue. Whoever’s free grabs the next email. Same shape for image resizing, PDF generation, webhook delivery — any pile of jobs where “one goroutine each” would melt something downstream.

// worker pool: 9 newsletters, only 3 SMTP connections.
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	start := time.Now()
	queue := make(chan string)
	var wg sync.WaitGroup

	for conn := 1; conn <= 3; conn++ { // 3 connections
		wg.Add(1)
		go func() {
			defer wg.Done()
			for email := range queue { // until closed
				time.Sleep(100 * time.Millisecond) // send
				fmt.Printf("conn %d sent → %s\n",
					conn, email)
			}
		}()
	}

	users := []string{
		"asha", "ravi", "meera", "arjun", "divya",
		"karan", "nisha", "rahul", "sneha",
	}
	for _, u := range users {
		queue <- u + "@example.com"
	}
	close(queue) // newsletter fully queued
	wg.Wait()

	fmt.Printf("9 emails on 3 connections = %v\n",
		time.Since(start).Round(10*time.Millisecond))
}
📖 Line by line — what & why
queue := make(chan string)

What: one shared channel: the order line every worker pulls from. Why: the channel IS the work distribution — no scheduler, no “which worker is free?” logic anywhere.

for conn := 1; conn <= 3; conn++ { wg.Add(1); go func() {

What: hire exactly 3 workers, count them into a WaitGroup. Why: the pool size (3) caps how hard you hit the SMTP server — and it's the ONE number to tune when throughput changes.

for email := range queue {

What: each worker takes the next email, sends it, comes back for more — until the queue closes. Why: a free worker automatically grabs the next job because they all receive from the same channel. Fairness for free.

queue <- u + "@example.com"

What: feeds one address into the line. Blocks if all 3 workers are mid-send. Why: backpressure again: the feeder can never outrun the pool by more than one item.

close(queue)

What: “newsletter fully queued — nothing more is coming.” Why: this ends every worker's range loop. Forget it and all 3 workers wait forever → wg.Wait() deadlocks → Go prints all goroutines are asleep.

wg.Wait()

What: block until all 3 workers have exited (queue drained). Why: only now is “all emails sent” true — safe to print the elapsed time, or in a handler, to respond.

terminal
$ go run ./06-workerpool
conn 3 sent → asha@example.com
conn 1 sent → ravi@example.com
conn 2 sent → meera@example.com
conn 2 sent → divya@example.com
conn 3 sent → arjun@example.com
conn 1 sent → karan@example.com
conn 2 sent → rahul@example.com
conn 3 sent → nisha@example.com
conn 1 sent → sneha@example.com
9 emails on 3 connections = 300ms

Notice: 900ms of sending done in 300ms, and no email waited on a specific connection — just the next free one. Try 1 connection (900ms) or 9 (100ms).

07 context please stop

A heavy report endpoint with a 350ms budget — the context aborts batch 4 mid-flight.

💡 Everyday version: the customer walked out, so the kitchen stops cooking their order. The context is the order ticket every function holds a copy of; when the 350ms timer rings (or the client disconnects), ctx.Done() fires and the report stops burning CPU for a response nobody will read.

// context: a report that stops when the request times out.
package main

import (
	"context"
	"fmt"
	"time"
)

func generateReport(ctx context.Context) {
	for batch := 1; ; batch++ {
		select {
		case <-ctx.Done(): // out of time / client gone
			fmt.Println("aborting report:", ctx.Err())
			return
		case <-time.After(100 * time.Millisecond):
			fmt.Printf("processed batch %d (10k rows)\n",
				batch)
		}
	}
}

func main() {
	// this endpoint has a 350ms budget
	ctx, cancel := context.WithTimeout(
		context.Background(), 350*time.Millisecond)
	defer cancel()

	generateReport(ctx)
	fmt.Println("handler: told client 504, moved on")
}
📖 Line by line — what & why
ctx, cancel := context.WithTimeout(context.Background(), 350*time.Millisecond)

What: makes a context whose Done() channel closes automatically after 350ms, plus a cancel function. Why: Background() is the empty root — fine in main. In a real handler you'd wrap r.Context() instead, so a disconnecting client ALSO stops the report.

defer cancel()

What: releases the context's timer when main exits — even if we finished early. Why: every WithTimeout/WithCancel leaks a timer+goroutine until cancelled. go vet flags a missing cancel; this defer is non-negotiable.

case <-ctx.Done():

What: this case fires the instant the context is cancelled or times out. Why: THE key idea: Go cannot kill a goroutine from outside. The worker must check for the stop signal itself — this select in the loop is that check.

fmt.Println("aborting report:", ctx.Err())

What: ctx.Err() says WHY it stopped. Why: DeadlineExceeded → respond 504; Canceled → the client hung up, respond to nobody. Different reasons, different handling.

case <-time.After(100 * time.Millisecond):

What: one batch of work (100ms of row-crunching). Why: the loop shape matters: work in SMALL units, check Done between units. One giant uninterruptible batch would ignore the budget entirely.

terminal
$ go run ./07-context
processed batch 1 (10k rows)
processed batch 2 (10k rows)
processed batch 3 (10k rows)
aborting report: context deadline exceeded
handler: told client 504, moved on

Notice: exactly 3 batches fit in the budget, then ctx.Err() says why it stopped. Swap WithTimeout for WithCancel and call cancel() from another goroutine — same shutdown, your trigger.

08 errgroup x/sync

POST /checkout fans out to inventory, payment and shipping. The card declines.

💡 Everyday version: a team lead sends three devs off and waits at standup. Everyone reports; the lead hears the FIRST blocker. It’s the WaitGroup whiteboard from program 02, but workers can also file complaints — which is why it’s the default for real service fan-out.

// errgroup: POST /checkout fans out; payment declines.
// needs: go get golang.org/x/sync/errgroup
package main

import (
	"errors"
	"fmt"

	"golang.org/x/sync/errgroup"
)

func callService(name string) error {
	if name == "payment-service" {
		return errors.New(
			"payment-service: card declined")
	}
	fmt.Println(name, "→ OK")
	return nil
}

func main() {
	var g errgroup.Group

	for _, svc := range []string{
		"inventory-service",
		"payment-service",
		"shipping-service",
	} {
		g.Go(func() error {
			return callService(svc)
		})
	}

	// Wait = wg.Wait() + the first error
	if err := g.Wait(); err != nil {
		fmt.Println("checkout failed:", err)
		fmt.Println("→ respond 402, release the stock")
	}
}
📖 Line by line — what & why
var g errgroup.Group

What: the group — zero value ready. Think: WaitGroup with an error slot. Why: from golang.org/x/sync — the one “extra” import that's on every backend team's go.mod anyway.

g.Go(func() error { return callService(svc) })

What: starts the call in a goroutine AND registers it with the group. The function returns an error. Why: this one line replaces program 02's Add(1) + go + defer Done() — and adds the error plumbing WaitGroup can't do.

if url == "payment-service" { return errors.New(…) }

What: payment declines; the other two succeed. Why: modelled deliberately: partial success is the NORMAL failure mode of fan-out, and your code must handle it.

if err := g.Wait(); err != nil {

What: blocks like wg.Wait(), then hands you the FIRST non-nil error (or nil if all succeeded). Why: one line decides the whole checkout's fate. No collecting errors from three channels by hand.

fmt.Println("→ respond 402, release the stock")

What: the business consequence. Why: inventory-service already reserved stock and said OK — a failed checkout MUST undo that. Fan-out failures are always “handle the survivors” problems.

terminal
$ go run ./08-errgroup
shipping-service → OK
inventory-service → OK
checkout failed: payment-service: card declined
→ respond 402, release the stock

Notice: inventory and shipping still returned OK — that’s exactly why the failure path must release the reserved stock. g.Wait() handed back precisely the payment error, nothing vaguer.

09 JSON both ways encoding

The two JSON moments of every endpoint: decode the request body, encode the response.

💡 Everyday version: your struct is the organized shelf; JSON is the cardboard box on the truck. Unmarshal = unpacking the client’s box onto your shelf (unknown fields are ignored, missing ones stay zero); Marshal = packing your response with the labels the tags chose.

// json: the request and response bodies of POST /orders.
package main

import (
	"encoding/json"
	"fmt"
)

type Order struct {
	ID     int      `json:"id"`
	UserID int      `json:"user_id"`
	Items  []string `json:"items"`
	Coupon string   `json:"coupon,omitempty"`
}

func main() {
	// what the client POSTs
	body := `{"user_id": 7, "items": ["boat-airdopes"]}`
	var in Order
	if err := json.Unmarshal([]byte(body), &in); err != nil {
		panic(err) // in a handler: respond 400
	}
	fmt.Printf("decoded request: %+v\n", in)

	// what your handler sends back
	in.ID = 1042 // the DB assigned it
	out, _ := json.MarshalIndent(in, "", "  ")
	fmt.Println(string(out))
}
📖 Line by line — what & why
type Order struct { ID int `json:"id"` … }

What: the struct IS the contract: field names, types, and their wire spelling (the backtick tags). Why: Capitalized fields are required — the json package can't see lowercase ones. The tag maps Go's UserID to the API's user_id.

Coupon string `json:"coupon,omitempty"`

What: omitempty: if the value is empty (""), leave the key out of the output entirely. Why: keeps responses clean — clients see coupon only when there is one, instead of a forest of empty strings.

body := `{"user_id": 7, "items": ["boat-airdopes"]}`

What: exactly what arrives in an HTTP request body. Why: note there's no id — the client doesn't get to choose IDs. Missing fields simply stay at their zero value after decoding.

json.Unmarshal([]byte(body), &in)

What: parses the JSON INTO the struct — &in (a pointer) so it can fill YOUR variable. Why: the error return is your 400 Bad Request: malformed JSON or wrong types land there. Unknown fields are silently ignored — old clients don't break new servers.

in.ID = 1042

What: the server assigns the ID after “saving”. Why: the same struct now becomes the response — a common and convenient pattern.

json.MarshalIndent(in, "", " ")

What: struct → pretty-printed JSON bytes. Why: in real handlers use json.NewEncoder(w).Encode(in) — Marshal here just makes the output readable on this page. Same rules either way.

terminal
$ go run ./09-json
decoded request: {ID:0 UserID:7 Items:[boat-airdopes] Coupon:}
{
  "id": 1042,
  "user_id": 7,
  "items": [
    "boat-airdopes"
  ]
}

Notice: ID was 0 after decoding — clients don’t get to pick IDs; the DB does. And the empty Coupon disappeared from the response thanks to omitempty.

10 HTTP server + client one file

GET /orders/{id} — the microservice AND the service that calls it, in one file.

💡 Everyday version: both halves of the day job: the handler that answers (decode path → fetch → encode JSON) and the client that asks — which is also you, every time your service calls another one. The server runs in a goroutine; program 01 pays off.

// http: GET /orders/{id} — the service AND its caller.
package main

import (
	"encoding/json"
	"fmt"
	"net/http"
	"time"
)

type Order struct {
	ID     string `json:"id"`
	Status string `json:"status"`
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /orders/{id}",
		func(w http.ResponseWriter, r *http.Request) {
			id := r.PathValue("id")
			// (a real handler queries the DB here)
			json.NewEncoder(w).Encode(
				Order{ID: id, Status: "shipped"})
		})

	go http.ListenAndServe(":8090", mux) // the service
	time.Sleep(100 * time.Millisecond)   // let it boot

	// another service asks about order 1042
	res, err := http.Get(
		"http://localhost:8090/orders/1042")
	if err != nil {
		panic(err)
	}
	defer res.Body.Close()

	var o Order
	json.NewDecoder(res.Body).Decode(&o)
	fmt.Println("status code:", res.StatusCode)
	fmt.Printf("order: %+v\n", o)
}
📖 Line by line — what & why
mux := http.NewServeMux()

What: the router: it matches incoming request paths to handler functions. Why: since Go 1.22 it understands methods and path variables — most APIs need nothing fancier.

mux.HandleFunc("GET /orders/{id}", func(w, r) {…})

What: registers a handler for GET requests matching /orders/. Why: the method in the pattern means a POST to this path is auto-rejected with 405. {id} captures the path segment.

id := r.PathValue("id")

What: reads whatever matched {id} — here “1042”. Why: no regex, no string splitting. r carries everything about the request: path, headers, body.

json.NewEncoder(w).Encode(Order{…})

What: writes the struct as JSON straight into the response. Why: w is the response; encoding directly into it streams the bytes — no intermediate string needed.

go http.ListenAndServe(":8090", mux)

What: starts the server in a goroutine. Why: ListenAndServe blocks forever — run it in the foreground here and the client code below would never execute.

time.Sleep(100 * time.Millisecond)

What: crude “wait for the server to boot”. Why: fine for a demo. In tests, use httptest.NewServer — it's ready the moment it returns, no sleeping.

res, err := http.Get("http://localhost:8090/orders/1042")

What: the client half: a real HTTP request over a real socket. Why: err here means “couldn't even connect”. A 404 or 500 is NOT an err — check res.StatusCode for that.

defer res.Body.Close()

What: always close the response body. Why: unclosed bodies leak connections; the pool can't reuse them. This line follows every successful http.Get like a shadow.

json.NewDecoder(res.Body).Decode(&o)

What: the mirror of the server's Encode: response JSON → struct. Why: the two services agree on shape through the Order struct — change it in one place, both sides follow.

terminal
$ go run ./10-httpapi
status code: 200
order: {ID:1042 Status:shipped}

Notice: r.PathValue("id") pulled 1042 straight from the URL, and the caller decoded the same struct back out. This request/response loop is 90% of microservice life.

11 MySQL: the signup flow database/sql

What POST /signup actually does to the database: connect, insert the email, read it back.

💡 Everyday version: sql.Open is saving a phone number — no call made. db.Ping is dialling to check someone answers. The ? placeholder is a printed form with a fixed answer box: whatever the user types stays INSIDE the box and can never rewrite the form — that’s why injection can’t happen.

// mysql: what POST /signup does to the database.
// needs: go get github.com/go-sql-driver/mysql
// and a server:
//   docker run -d --name mysql-demo \
//     -e MYSQL_ROOT_PASSWORD=secret \
//     -e MYSQL_DATABASE=notes -p 3306:3306 mysql:8
package main

import (
	"database/sql"
	"fmt"

	_ "github.com/go-sql-driver/mysql"
)

func main() {
	db, err := sql.Open("mysql",
		"root:secret@tcp(localhost:3306)/notes")
	if err != nil {
		panic(err)
	}
	defer db.Close()

	if err := db.Ping(); err != nil {
		panic("db unreachable: " + err.Error())
	}
	fmt.Println("connected ✓")

	db.Exec("DROP TABLE IF EXISTS users") // rerunnable demo
	db.Exec(`CREATE TABLE users (
		id    INT AUTO_INCREMENT PRIMARY KEY,
		email VARCHAR(120) NOT NULL UNIQUE)`)

	// the signup form's email goes in via ? — never
	// by pasting it into the SQL string
	res, err := db.Exec(
		"INSERT INTO users (email) VALUES (?)",
		"asha@example.com")
	if err != nil {
		panic(err) // e.g. duplicate email → 409
	}
	id, _ := res.LastInsertId()
	fmt.Println("new user id:", id)

	var email string
	db.QueryRow(
		"SELECT email FROM users WHERE id = ?",
		id).Scan(&email)
	fmt.Println("stored:", email)
}
📖 Line by line — what & why
db, err := sql.Open("mysql", "root:secret@tcp(localhost:3306)/notes")

What: parses the connection string and prepares a connection POOL. No network yet. Why: “mysql” names the driver registered by the blank import. A wrong password does NOT fail here — Open is lazy.

defer db.Close()

What: returns every pooled connection when main exits. Why: in a server you'd open once at startup and close at shutdown — *sql.DB is shared and thread-safe, never per-request.

db.Ping()

What: the FIRST real network round-trip: connect, authenticate. Why: this is where a bad password or unreachable host actually explodes. Ping at boot so it fails loudly at deploy, not on your first user.

db.Exec(`CREATE TABLE users ( … email … UNIQUE)`)

What: runs DDL. The UNIQUE constraint makes duplicate emails impossible at the DATABASE level. Why: a rule enforced by the DB can't be bypassed by a buggy handler, a second service, or someone in a SQL console. (The DROP above it just makes this demo rerunnable.)

db.Exec("INSERT INTO users (email) VALUES (?)", "asha@example.com")

What: the statement and the value travel SEPARATELY; the driver combines them safely. Why: this is the SQL-injection defence. Never fmt.Sprintf a query — '; DROP TABLE users;-- in a form field stays a harmless string inside the ? box.

if err != nil { panic(err) } // e.g. duplicate email → 409

What: the INSERT can fail — most interestingly when UNIQUE rejects a duplicate. Why: in a real handler this err becomes 409 Conflict: email already registered. The constraint does the checking; you just translate the error.

id, _ := res.LastInsertId()

What: fetches the AUTO_INCREMENT id MySQL just assigned. Why: you need it for the response body ({"id": 1, …}) and for follow-up queries.

db.QueryRow("SELECT … WHERE id = ?", id).Scan(&email)

What: runs a one-row query and copies the columns into your variables (via pointers). Why: QueryRow for exactly-one-row; Scan fills &email. If no row matched, Scan returns sql.ErrNoRows — that's your 404.

terminal
$ go run ./11-mysql
connected ✓
new user id: 1
stored: asha@example.com

Notice: UNIQUE on email means a duplicate signup makes the INSERT error — your handler turns that into a 409; the database enforces a business rule for free. (DROP TABLE just keeps the demo rerunnable.)

12 graceful shutdown production

Deploy day for the orders API — SIGTERM arrives, it stops taking orders and finishes the open ones.

💡 Everyday version: closing a restaurant properly: lock the front door (stop accepting), let seated guests finish (drain in-flight requests), then lights out. Kubernetes is the landlord — it rings the bell (SIGTERM) and cuts power ~30s later. Skip this and every deploy hangs up on users mid-request.

// graceful shutdown: deploy day for your orders API.
// (sends ITSELF a SIGTERM so you can watch the drain)
package main

import (
	"context"
	"fmt"
	"net/http"
	"os"
	"os/signal"
	"syscall"
	"time"
)

func main() {
	srv := &http.Server{Addr: ":8091"}

	ctx, stop := signal.NotifyContext(
		context.Background(),
		os.Interrupt, syscall.SIGTERM)
	defer stop()

	go srv.ListenAndServe()
	fmt.Println("orders API serving on :8091")

	go func() { // pretend k8s deploys in 300ms
		time.Sleep(300 * time.Millisecond)
		syscall.Kill(os.Getpid(), syscall.SIGTERM)
	}()

	<-ctx.Done() // block until the signal
	fmt.Println("SIGTERM: stop taking orders, finish the open ones…")

	shCtx, cancel := context.WithTimeout(
		context.Background(), 5*time.Second)
	defer cancel()
	if err := srv.Shutdown(shCtx); err != nil {
		panic(err)
	}
	fmt.Println("all in-flight requests served — clean exit ✓")
}
📖 Line by line — what & why
srv := &http.Server{Addr: ":8091"}

What: builds the server as a value we keep a reference to. Why: the usual one-liner http.ListenAndServe gives you no handle — and you need the handle to call .Shutdown() on it later.

ctx, stop := signal.NotifyContext(…, os.Interrupt, syscall.SIGTERM)

What: a context that cancels itself when Ctrl-C (Interrupt) or SIGTERM arrives. Why: SIGTERM is what Kubernetes/Docker send before killing a container. This one line converts “OS signal” into the same ctx.Done() mechanism as program 07.

go srv.ListenAndServe()

What: serve traffic in a goroutine. Why: it blocks forever, and main has other work: standing guard for the signal.

go func() { time.Sleep(300ms); syscall.Kill(os.Getpid(), syscall.SIGTERM) }()

What: the demo trick: after 300ms, the program sends ITSELF a SIGTERM. Why: simulates deploy day so you can watch the sequence. Delete this goroutine and the program waits for your real Ctrl-C.

<-ctx.Done()

What: main parks here — receiving from a channel blocks — until the signal lands. Why: the cheapest possible “wait”: no loop, no polling, zero CPU.

shCtx, cancel := context.WithTimeout(…, 5*time.Second)

What: a deadline for the drain itself. Why: k8s force-kills ~30s after SIGTERM. Your drain must finish before that — 5s says “finish open requests, but don't wait forever on a stuck one”.

srv.Shutdown(shCtx)

What: stops accepting NEW connections, waits for in-flight requests to complete, then returns. Why: this is the graceful part. The alternative — just exiting — resets every open connection: users see errors on every deploy.

terminal
$ go run ./12-shutdown
orders API serving on :8091
SIGTERM: stop taking orders, finish the open ones…
all in-flight requests served — clean exit ✓

Notice: the pieces are all from this lab — a goroutine (01), a signal context (07), blocking on ctx.Done(). Delete the fake-SIGTERM goroutine and it waits for your real Ctrl-C.
