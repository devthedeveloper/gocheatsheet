Ship it — one file, no runtime

Go's superpower: go build produces a single self-contained binary. No interpreter, no node_modules, nothing to install on the server. Because our SQLite driver is pure Go, you can even build for a different operating system from your laptop:

terminal
# build a binary for this machine:
$ go build -o linkboard .
$ ./linkboard
2026/07/07 17:52:00 linkboard listening on http://localhost:4000

# ...or cross-compile for a Linux server / a Raspberry Pi / a Mac:
$ GOOS=linux  GOARCH=amd64 go build -o linkboard-linux .
$ GOOS=linux  GOARCH=arm64 go build -o linkboard-pi    .
$ GOOS=darwin GOARCH=arm64 go build -o linkboard-mac   .

Copy that one file (and, when it runs, its linkboard.db and index.html) to any little server, run it behind an HTTPS proxy, and your board is on the internet. That's the whole deployment story for a Go app this size.

You built all of this

Four small files: main.go (wiring), store.go (the database), handlers.go (the web logic), index.html (the page). About 180 lines of Go. A real, persistent, votable, browsable link board — and you can explain every line.

Two quick wins before you go

Want to cement it? Try these on linkboard first — each is a five-minute change and each is a preview of a Reddit-book idea:

  • Add a domain to each link (parse it from the URL, show it in the list) — practice touching the struct, the SQL, and the frontend together.
  • Cap voting spam cheaply: store a vote count per link per IP in a second tiny table. You'll feel why real accounts are the clean answer — which is exactly Chapter 6.
