One File, Whole API

a working link-sharing API in ~60 lines — before your coffee cools

In this chapter you'll learn ✍️

  • the handler shape every Go web program is built from
  • routing with Go's built-in ServeMux — no framework
  • reading JSON in and writing JSON out
  • enough to have a real API you can curl

The plan for linkboard

We're building linkboard: post a link, list the links. That's the whole app today. No database yet — we'll keep the links in a plain Go slice in memory. It's the fastest possible path to a running API you fully understand, and every later chapter grows this exact file.

Two endpoints to start:

GET /links  — list every link  ·  POST /links  — add one

The whole thing, one file

Make a folder, drop this in main.go, and that's the entire program. We'll read it piece by piece right after:

main.go 
// linkboard — a tiny anonymous link board, chapter 1: everything in one file,
// with the links kept in memory. No database yet (that's chapter 2).
package main

import (
	"encoding/json"
	"log"
	"net/http"
	"sync"
)

// Link is one submission on the board. The `json:"..."` tags decide the key
// names in the JSON we send back.
type Link struct {
	ID    int64  `json:"id"`
	Title string `json:"title"`
	URL   string `json:"url"`
	Votes int64  `json:"votes"`
}

// Our "database" for now: a slice in memory. A mutex guards it because the
// server runs each request in its own goroutine, so two POSTs can arrive at
// the same instant. (Goroutines & mutexes: see the cheatsheet.)
var (
	mu     sync.Mutex
	links  []Link
	nextID int64 = 1
)

// listLinksHandler — GET /links
func listLinksHandler(w http.ResponseWriter, r *http.Request) {
	mu.Lock()
	defer mu.Unlock()

	w.Header().Set("Content-Type", "application/json")
	json.NewEncoder(w).Encode(links)
}

// createLinkHandler — POST /links   Body: {"title": "...", "url": "..."}
func createLinkHandler(w http.ResponseWriter, r *http.Request) {
	var input struct {
		Title string `json:"title"`
		URL   string `json:"url"`
	}
	if err := json.NewDecoder(r.Body).Decode(&input); err != nil {
		http.Error(w, "body must be valid JSON", http.StatusBadRequest)
		return
	}

	mu.Lock()
	link := Link{ID: nextID, Title: input.Title, URL: input.URL}
	nextID++
	links = append(links, link)
	mu.Unlock()

	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(http.StatusCreated) // 201
	json.NewEncoder(w).Encode(link)
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("GET /links", listLinksHandler)
	mux.HandleFunc("POST /links", createLinkHandler)

	log.Println("linkboard listening on http://localhost:4000")
	log.Fatal(http.ListenAndServe(":4000", mux))
}

that's a real, concurrent web server. you built an API! ⚡

Reading it, three ideas at a time

1. The handler — the shape of all Go web code

Look at listLinksHandler and createLinkHandler. Both have the same signature, and every web handler you ever write in Go looks like this:

the shape 
func name(w http.ResponseWriter, r *http.Request) {
	// r = the request the client sent   (read from it)
	// w = your reply                     (write to it)
}
  • r is the incoming request — its body, URL, headers. You read from it.
  • w is how you answer — set a header, a status code, write bytes.

2. Struct tags turn Go into JSON

The Link struct has little backtick tags: `json:"title"`. They rename the fields when Go converts to JSON — Go's Title becomes the key "title". json.NewEncoder(w).Encode(link) writes the struct out as JSON; json.NewDecoder(r.Body).Decode(&input) reads JSON back into a struct. That's the whole JSON story for now.

3. The router knows methods

mux.HandleFunc("POST /links", ...) — since Go 1.22, the built-in router understands the method + path in one string. GET /links and POST /links are two different routes to two different functions. No third-party router needed — this is the standard library.

Why the mutex?

Go runs every request in its own lightweight thread (a goroutine), so two people could POST at the same instant and both touch the links slice together — a data race. The three lines of sync.Mutex (mu.Lock() … mu.Unlock()) make sure only one goroutine edits the slice at a time. Curious how goroutines work? That's a whole page on the cheatsheet — but you don't need the details to use this pattern. (Good news: in the next chapter the database takes over this job and the mutex disappears.)

Why did the empty list say null?

An empty Go slice that was never filled is nil, and nil becomes JSON null, not []. Harmless here, but worth knowing — the moment we switch to a database next chapter, the list comes back as a tidy [] instead.

Pin to your brain 📌

  • A handler is func(w http.ResponseWriter, r *http.Request) — read r, write w.
  • Routes are "METHOD /path" on an http.NewServeMux() — std lib, no framework.
  • Struct tags map Go fields to JSON keys; Encode writes JSON, Decode reads it.
  • Set the status with w.WriteHeader(...) — 201 Created for a new thing.
