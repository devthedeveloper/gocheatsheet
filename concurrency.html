1 Goroutines

the unit of concurrency — cheap, but not free of consequences

A goroutine is a function run concurrently, scheduled by the Go runtime onto OS threads — light enough to spawn millions.

How it works

go f() puts f on a run queue and returns immediately. The runtime multiplexes many goroutines (G) onto a few OS threads (M) through logical processors (P) — the G-M-P scheduler. GOMAXPROCS sets how many run in parallel (defaults to CPU count). Stacks start at ~2 KB and grow on demand, which is why millions fit where OS threads (1–2 MB each) would not.

Crucially: when main returns the process exits and every goroutine dies instantly, no cleanup. So you always need a way to wait (a WaitGroup or a channel).

func main() {
	go greet("A")          // starts, returns instantly
	go greet("B")

	// main must stick around, or A and B never run:
	time.Sleep(time.Second) // crude — real code uses sync.WaitGroup

	// there is NO way to kill a goroutine from outside.
	// it exits only by returning — so you design the exit in
	// (a closed channel, or ctx.Done()).
}

⚠ Gotchas they probe

  • main returning kills all goroutines — unsynchronized "fire and forget" loses work.
  • You cannot stop a goroutine externally; it must return on its own. Signal it (ctx / closed channel) and have it check.
  • A goroutine blocked forever on a channel is a leak — it costs memory and never gets collected.
  • Goroutines touching shared memory still need synchronization — concurrency ≠ safety.

Interview Q&A

Goroutine vs OS thread — what's the difference?
A goroutine is a user-space construct: ~2 KB growable stack, scheduled by the Go runtime, context-switched without a kernel call. An OS thread is ~1–2 MB, scheduled by the kernel. Go multiplexes many goroutines onto few threads (the G-M-P model), so you can run hundreds of thousands cheaply.
Follow-up — how many can you run? Millions; the limit is memory (stacks), not thread count.
How do you stop a running goroutine?
You don't — not from outside. A goroutine ends only by returning. You give it a cancellation signal — ctx.Done() or a channel you close() — and the goroutine selects on it and returns when it fires. Anything else is a leak.
What happens if main returns while goroutines are still working?
They're killed immediately — no deferred functions run, no cleanup. That's why every spawned goroutine needs something (WaitGroup, channel, errgroup) that makes main wait for it.

2 Channels

pass data and synchronize in one move

A typed conduit between goroutines that transfers a value and synchronizes the two ends. "Don't communicate by sharing memory; share memory by communicating."

How it works

Unbuffered (make(chan T)) is a synchronous hand-off: a send blocks until a receiver is ready, and vice-versa. Buffered (make(chan T, n)) holds up to n items; send blocks only when full, receive only when empty. close(ch) announces "no more values"; a range loop drains then ends, and a receive after draining yields the zero value with ok == false.

ch := make(chan int, 2) // buffered, capacity 2
ch <- 1                 // doesn't block (room)
ch <- 2                 // doesn't block (now full)
close(ch)               // sender says: no more

for v := range ch {     // drains 1, 2, then loop ends
	fmt.Println(v)
}
v, ok := <-ch           // 0, false  (drained + closed)

⚠ Gotchas they probe

  • Send on a closed channel panics; closing twice panics. Rule: only the sender closes, exactly once.
  • A receiver must never close — it can't know if more is coming, and risks a send-on-closed panic elsewhere.
  • Unbuffered send with no receiver = deadlock (if it's the last goroutine) or a leak.
  • You only need to close if receivers range or otherwise wait on it.

Interview Q&A

Buffered vs unbuffered — when do you use each?
Unbuffered when you need a synchronization guarantee — the send doesn't complete until someone receives (a hand-off). Buffered when you want to decouple producer and consumer speeds, or cap the number of in-flight items (e.g. one slot per worker).
Follow-up — does buffering change correctness? It can hide backpressure. An over-large buffer lets a fast producer pile up memory instead of being throttled by a slow consumer.
Who is responsible for closing a channel, and why?
The sender, exactly once. Only the sender knows when there are no more values. If a receiver closed it, another sender could then panic with "send on closed channel." With multiple senders, coordinate a close via a separate "done" signal or a sync.WaitGroup + a single closer goroutine.
What does receiving from a closed channel return?
First it drains any buffered values, then it returns the element's zero value with ok == false — forever, without blocking. The v, ok := <-ch form is how you detect closure.
What happens with a nil channel?
Send and receive both block forever; closing one panics. This is occasionally useful: setting a channel variable to nil disables its case in a select.

3 select

wait on many channels; first ready wins

Blocks until one of several channel operations can proceed, then runs that one. If several are ready, it picks uniformly at random.

How it works

Each case is a channel send or receive. select waits until at least one can proceed. A default case makes it non-blocking (run default if nothing's ready). time.After gives you a timeout; ctx.Done() gives you cancellation. A nil channel case is silently skipped — a clean way to turn a case off mid-loop.

select {
case v := <-results:
	use(v)
case <-ctx.Done():         // cancelled / deadline hit
	return ctx.Err()
case <-time.After(2 * time.Second):
	return errTimeout        // don't wait forever
}

⚠ Gotchas they probe

  • Several cases ready → random choice. Never depend on ordering.
  • default makes select non-blocking — wrap it in a tight loop and you've built a CPU-burning spin.
  • time.After inside a for loop allocates a fresh timer every iteration — use a single time.Ticker instead.
  • select{} with no cases blocks forever (instant deadlock if it's alone).

Interview Q&A

How do you put a timeout on a channel operation?
Wrap the receive/send in a select with a case <-time.After(d) (relative) or case <-ctx.Done() (propagated deadline). The first to fire wins; if it's the timeout, you bail instead of blocking indefinitely.
Two cases are ready at the same time — which runs?
One is chosen uniformly at random. This is deliberate — it prevents starvation where an always-ready case would monopolize the select.
What's the nil-channel trick in select?
A receive/send on a nil channel blocks forever, so that case is effectively disabled. Setting a channel variable to nil (e.g. after it's drained) removes its case from consideration without restructuring the select — cleaner than juggling booleans.

4 The sync toolbox

shared-memory coordination for when channels are the wrong tool

Locks and helpers for guarding shared state directly: Mutex, RWMutex, WaitGroup, Once, and lock-free atomic.

How it works

Mutex — one goroutine in the critical section at a time (Lock/Unlock). RWMutex — many readers or one writer, for read-heavy data. WaitGroup — wait for N goroutines: Add before go, defer Done, Wait. Once — run something exactly once ever (lazy init). atomic — lock-free operations on a single integer / pointer / bool.

type Counter struct {
	mu sync.Mutex
	n  int
}
func (c *Counter) Inc() {   // pointer receiver — see gotcha
	c.mu.Lock()
	defer c.mu.Unlock()
	c.n++                   // one goroutine at a time
}

var once sync.Once
once.Do(loadConfig)         // runs exactly once, ever

var hits atomic.Int64
hits.Add(1)                 // lock-free; simpler than a mutex for one number

⚠ Gotchas they probe

  • Copying a Mutex (a value receiver on a struct that holds one, or passing it by value) locks a copy — protecting nothing. Use pointer receivers; go vet catches it.
  • Don't hold a lock across I/O or other blocking calls — it serializes everyone.
  • WaitGroup.Add must run before the go, not inside the goroutine (or Wait can return early).
  • RWMutex only wins when reads vastly outnumber writes; otherwise plain Mutex is simpler and often faster.

Interview Q&A

Mutex or channel — how do you choose?
Use a mutex to guard shared state that stays in place (a map, a counter, a cache). Use a channel to transfer ownership of data or to signal between goroutines. The proverb is "share memory by communicating," but a mutex around a map is frequently the simpler, faster answer — pick the one that makes the code obvious.
What does a value receiver on a mutex-holding type cause?
Each call copies the struct — including the mutex — so the method locks a fresh copy that nobody else sees. Zero protection; the race is still there. Always use pointer receivers on types with a mutex. go vet's copylocks check flags this.
atomic vs mutex?
atomic for a single word — a counter, a flag, a pointer swap — where it's lock-free and faster. A Mutex when you must update several fields together as one consistent step. Don't build compound invariants out of multiple atomics; that's how subtle races sneak in.
When is RWMutex actually worth it?
Only when reads massively outnumber writes, because it carries more bookkeeping overhead than a plain Mutex. Under a balanced or write-heavy load, a Mutex usually wins. Measure before reaching for it.

5 context

cancellation as a tree — cut a branch, everything below it stops

The standard way to carry cancellation, deadlines, and request-scoped values across API and goroutine boundaries.

How it works

Contexts form a tree: cancelling a parent cancels every child. ctx.Done() is a channel that closes on cancel or timeout; ctx.Err() tells you which (Canceled or DeadlineExceeded). You pass ctx as the first argument down the call chain — a client disconnect at the top propagates all the way to the database query at the bottom.

ctx, cancel := context.WithTimeout(r.Context(), 5*time.Second)
defer cancel()                     // ALWAYS — even on success

rows, err := db.QueryContext(ctx, "SELECT ...")  // aborts if ctx fires

// CPU-bound loop? You must check it yourself:
for _, item := range huge {
	select {
	case <-ctx.Done():
		return ctx.Err()
	default:
	}
	process(item)
}

⚠ Gotchas they probe

  • Always defer cancel(), even when the operation succeeds — otherwise you leak the timer and its goroutine. go vet flags a missing cancel.
  • A context only helps if you check it (select on Done()) or pass it to ctx-aware calls (QueryContext, http.NewRequestWithContext).
  • Don't stash dependencies in ctx.Value — only request-scoped data (trace id, user id). Use a private key type to avoid collisions.
  • Never store a context in a struct field; pass it explicitly.

Interview Q&A

How does cancelling one request stop all the work it started?
Every derived context shares the cancellation tree. When you cancel the top (or its deadline passes), every child's Done() channel closes. Each layer either selects on Done() or passes ctx into blocking calls, so the signal reaches the database query, the outbound HTTP call, the worker loop — all of it unwinds.
WithCancel vs WithTimeout vs WithDeadline?
WithCancel — you cancel manually. WithTimeout — cancels after a relative duration. WithDeadline — cancels at an absolute time. All three return a cancel function you must call (defer it) to release resources.
Where does the context go, and where does it come from?
It's always the first parameter: func Do(ctx context.Context, ...). Never a struct field. At the top you start from r.Context() in a handler, or context.Background() in main/tests; context.TODO() is a placeholder when you haven't wired it yet.

6 The patterns

the four shapes that cover most real concurrent code

Worker pool, fan-out/fan-in, pipeline, and errgroup — assemble goroutines + channels into reusable structures.

Worker pool — bounded parallelism

N workers pull from one jobs channel; a closer goroutine Waits then closes the results channel so the collector's range ends.

func pool(jobs []int, workers int) []int {
	in, out := make(chan int), make(chan int)
	var wg sync.WaitGroup
	for w := 0; w < workers; w++ {   // fixed crew
		wg.Add(1)
		go func() {
			defer wg.Done()
			for j := range in { out <- process(j) }
		}()
	}
	go func() { for _, j := range jobs { in <- j }; close(in) }()
	go func() { wg.Wait(); close(out) }()  // close AFTER all workers done
	var res []int
	for r := range out { res = append(res, r) }
	return res
}

Fan-in — merge many channels into one

func merge(cs ...<-chan int) <-chan int {
	out := make(chan int)
	var wg sync.WaitGroup
	for _, c := range cs {
		wg.Add(1)
		go func(c <-chan int) { defer wg.Done(); for v := range c { out <- v } }(c)
	}
	go func() { wg.Wait(); close(out) }()
	return out
}

Semaphore — cap concurrency without a pool

sem := make(chan struct{}, 10) // 10 tokens = max 10 at once
for _, t := range tasks {
	sem <- struct{}{}          // acquire (blocks when full)
	go func(t Task) {
		defer func() { <-sem }() // release
		run(t)
	}(t)
}

⚠ Gotchas they probe

  • Forget to close the results channel and the collector's range hangs forever → deadlock.
  • Close results too early (before workers finish) and you get a send-on-closed panic.
  • Unbounded "one goroutine per item" can exhaust memory or hammer a downstream — bound it with a pool or semaphore.

Interview Q&A

How do you limit concurrency to N?
Three idiomatic options: a worker pool (N goroutines ranging a jobs channel), a semaphore (a buffered channel of N tokens you acquire/release), or errgroup.Group with SetLimit(N). All cap how many run at once; choose by whether you also need result collection or error propagation.
How do you collect results from many goroutines safely?
Send them into a results channel, and have a separate goroutine do wg.Wait(); close(out). Then range the results channel in the collector — it ends cleanly when every worker is done. The key is that the closer is separate from the producers.
errgroup vs WaitGroup?
errgroup (golang.org/x/sync) is a WaitGroup that also propagates the first error, cancels a shared context when any goroutine fails, and (with SetLimit) bounds concurrency. It's the default for fan-out where any call can fail.

7 Races & deadlocks

the two failure modes of concurrent code

A race is unsynchronized shared access (undefined behavior); a deadlock is goroutines waiting on each other forever.

How it works

A data race: two goroutines touch the same memory, at least one writes, with no happens-before ordering between them → the result is undefined. A deadlock: every goroutine is blocked. Go detects the total case (all goroutines asleep) and aborts with fatal error: all goroutines are asleep - deadlock!; a partial deadlock just hangs. Detect races with go test -race / go run -race, which instruments memory accesses.

// DATA RACE — n++ is read-modify-write; interleavings lose updates
n := 0
for i := 0; i < 1000; i++ {
	go func() { n++ }()   // -race flags this; final n < 1000
}

// DEADLOCK — unbuffered send, nobody receiving
ch := make(chan int)
ch <- 1                   // fatal error: all goroutines asleep

⚠ Gotchas they probe

  • -race only catches races on paths your test actually exercises — weak tests + -race = false confidence.
  • Classic deadlock: acquiring two mutexes in different orders in different goroutines. Fix: a single global lock order.
  • Classic leak-not-deadlock: a goroutine blocked on a channel nobody will read — the program keeps running but bleeds goroutines.

Interview Q&A

What exactly is a data race?
Two goroutines access the same variable concurrently, at least one is a write, and there's no synchronization (no happens-before relationship) ordering them. The Go memory model calls the result undefined — not "the wrong number," but genuinely unpredictable, including torn reads.
How do you detect and fix races?
Detect with the race detector: go test -race ./... (run it in CI). It instruments memory accesses and reports the two conflicting stacks. Fix by adding synchronization — a Mutex, an atomic type, or by handing the data through a channel so only one goroutine owns it at a time.
How do you prevent deadlocks with multiple locks?
Always acquire locks in a consistent global order across the whole program. Deadlock needs a cycle in the "who waits for whom" graph; a fixed ordering makes a cycle impossible. Also: keep critical sections short and never block (I/O) while holding a lock.
What triggers "fatal error: all goroutines are asleep"?
The runtime detects that every goroutine is blocked — nobody can make progress — so it aborts rather than hang. Common cause: an unbuffered channel send/receive with no counterpart, or a WaitGroup.Wait() whose count never reaches zero because a Done() was missed.

C4The hangCore

What happens when you run this, and why?

func main() {
	ch := make(chan int)
	ch <- 1
	fmt.Println(<-ch)
}
Reveal answer

Deadlock. The channel is unbuffered, so ch <- 1 blocks until a receiver is ready — but the only goroutine is stuck on that very send, so the receive on the next line never runs. The runtime detects all goroutines are blocked and aborts.

Fix: either buffer it (make(chan int, 1)), or do the send in a separate goroutine (go func(){ ch <- 1 }()) so a receiver can run.

$ go run ./c4 fatal error: all goroutines are asleep - deadlock!
C5The disappearing incrementsCore

This should print 1000. It doesn't, and it's not even deterministic. Diagnose and fix.

var wg sync.WaitGroup
counter := 0
for i := 0; i < 1000; i++ {
	wg.Add(1)
	go func() { defer wg.Done(); counter++ }()
}
wg.Wait()
fmt.Println(counter)
Reveal answer

Data race. counter++ is read-modify-write; two goroutines read the same value, both add one, both write it back — one increment lost. Final value is < 1000 and varies run to run.

Fix: a sync.Mutex around counter++, or make it an atomic.Int64 and call counter.Add(1). Confirm with go run -race.

C6The slow leakCore

Called under load, this quietly eats memory. Where's the leak, and how do you close it?

func leaky() int {
	ch := make(chan int)
	go func() { ch <- expensive() }() // sends result
	select {
	case v := <-ch:
		return v
	case <-time.After(10 * time.Millisecond):
		return -1                     // timed out
	}
}
Reveal answer

Goroutine leak. When the timeout wins, leaky() returns — but the inner goroutine is now stuck forever on ch <- expensive(), because the channel is unbuffered and nobody will ever receive. Every timed-out call strands one goroutine.

Fix: make the channel buffered (make(chan int, 1)) so the sender can deposit its result and exit even after the receiver has moved on. (A ctx-aware version passes cancellation to expensive too.)

# the buggy version, called 100 times: $ go run ./c6 leaked goroutines still alive: 100
C7The premature WaitCore

Sometimes this waits for all three, sometimes it returns instantly having waited for none. Why?

var wg sync.WaitGroup
for i := 0; i < 3; i++ {
	go func() {
		wg.Add(1)          // <-- here
		defer wg.Done()
		doWork()
	}()
}
wg.Wait()
Reveal answer

Add is inside the goroutine. wg.Wait() can run before any goroutine has scheduled and called Add — so the counter is still 0 and Wait returns immediately, not waiting for the work. It's also a race on the WaitGroup counter.

Fix: call wg.Add(1) in the loop, before go. Rule: Add happens in the parent, Done in the child.

C8Bounded worker poolCore

Implement workerPool(jobs []int, workers int) []int that runs process(j)=j*j across exactly workers goroutines and returns all results. (Order doesn't matter — sort before returning if you like.)

Reveal a verified solution
func workerPool(jobs []int, workers int) []int {
	in, out := make(chan int), make(chan int)
	var wg sync.WaitGroup
	for w := 0; w < workers; w++ {
		wg.Add(1)
		go func() { defer wg.Done(); for j := range in { out <- j*j } }()
	}
	go func() { for _, j := range jobs { in <- j }; close(in) }()
	go func() { wg.Wait(); close(out) }()   // the crucial part
	var res []int
	for r := range out { res = append(res, r) }
	sort.Ints(res)
	return res
}

Key move: a separate goroutine does wg.Wait(); close(out) so the collector's range out terminates. Closing out inline would race the workers.

$ go run ./c8 # workerPool([1,2,3,4,5], 3) [1 4 9 16 25]
C9Run with timeoutCore

Implement runWithTimeout(ctx, fn func() int) (int, error): return fn's result, or ctx.Err() if the context fires first. Bonus: don't leak the goroutine when it times out.

Reveal a verified solution
func runWithTimeout(ctx context.Context, fn func() int) (int, error) {
	done := make(chan int, 1)          // buffered! sender never blocks → no leak
	go func() { done <- fn() }()
	select {
	case v := <-done:
		return v, nil
	case <-ctx.Done():
		return 0, ctx.Err()
	}
}

The buffered channel is the anti-leak trick (same lesson as C6): on timeout, fn's goroutine still deposits its result into the 1-slot buffer and exits cleanly instead of blocking forever.

$ go run ./c9 # fn sleeps 200ms, ctx times out at 50ms: 0 true # errors.Is(err, context.DeadlineExceeded) # fn returns instantly, no timeout: 7 <nil>
C10Fan-in mergeChallenge

Implement merge(cs ...<-chan int) <-chan int that combines several input channels into one, closing the output only after every input is drained.

Reveal a verified solution
func merge(cs ...<-chan int) <-chan int {
	out := make(chan int)
	var wg sync.WaitGroup
	for _, c := range cs {
		wg.Add(1)
		go func(c <-chan int) {        // pass c in (pre-1.22 habit, still clear)
			defer wg.Done()
			for v := range c { out <- v }
		}(c)
	}
	go func() { wg.Wait(); close(out) }()
	return out
}

One forwarding goroutine per input; a WaitGroup tracks them; a closer goroutine shuts out when all inputs are exhausted. Same "separate closer" structure as the worker pool.

$ go run ./c10 # merge(gen(1,2,3), gen(4,5,6)), then sorted [1 2 3 4 5 6]
C11Bounded parallel map, order preservedChallenge

Implement parallelMap(in []int, limit int, f func(int) int) []int: apply f to every element with at most limit running at once, returning results in input order. (This one has a neat trick that avoids any lock.)

Reveal a verified solution
func parallelMap(in []int, limit int, f func(int) int) []int {
	out := make([]int, len(in))
	sem := make(chan struct{}, limit)   // token bucket = concurrency cap
	var wg sync.WaitGroup
	for i, v := range in {
		wg.Add(1)
		sem <- struct{}{}               // acquire (blocks when `limit` busy)
		go func(i, v int) {
			defer wg.Done()
			defer func() { <-sem }()    // release
			out[i] = f(v)               // distinct index per goroutine → no race
		}(i, v)
	}
	wg.Wait()
	return out
}

The trick: each goroutine writes to its own index of a pre-sized slice. Different indices are different memory, so there's no data race and no mutex needed — and order is preserved for free. go run -race confirms it's clean.

$ go run -race ./c11 # parallelMap([1..6], 2, *10) [10 20 30 40 50 60]
